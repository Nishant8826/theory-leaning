# 🚑 Error Handling, Logging & Monitoring

## Simple Explanation
*   **Error Handling:** Imagine you are a **Firefighter**. When a fire starts, you don't just let it burn; you have a standard way to put it out (like using water or foam). **Error Handling (try-catch)** is the "Fire Extinguisher" for your code.
*   **Logging:** Imagine a **Pilot's Flight Box**. It records everything that happens in the plane. If something goes wrong, investigators look at the "Log" to see why.
*   **Monitoring:** Imagine a **Patient's Heart Monitor** in a hospital. If the heart stops, it beeps loudly! **Monitoring** is exactly that for your app.

---

## What
*   **Error Handling:** The way your code handles "Bad Things" (like a database crashing or a user typing a wrong password) without crashing the whole app.
*   **Logging:** Recording "What Happened" in your app (Events, User Actions, Errors) for later review.
*   **Monitoring:** Real-time dashboards and alerts that tell you when your server is slow or crashing.

---

## Why
1.  **To Stay "Up":** Good error handling means a small bug in "Comments" doesn't crash the "Login" system.
2.  **To Debug Faster:** Instead of guessing, you look at the "Logs" and see the exact line of code that failed.
3.  **To Know Before the User:** Monitoring tells **you** the app is slow *before* the customer starts complaining on Twitter.

---

## How it works

### Error Handling (The Try-Catch Block)
```javascript
try {
    const result = await database.get(user_id); // Try this...
} catch (error) {
    console.log("Database failed! Sending a safe error message."); // Catch the fire!
    res.status(500).json({ error: "Something went wrong!" });
}
```

### Logging (Standard Levels)
*   **INFO:** "Server started successfully." (Normal things).
*   **WARN:** "User tried 5 wrong passwords." (Suspicious but not broken).
*   **ERROR:** "Database Connection Failed!" (Something broke).

---

## Real-World Example
*   **E-commerce (Payment Error):** If the payment fails, the app uses **Error Handling** to say "Please check your card details," instead of showing a blank screen.
*   **Social Media (Logging):** The app "Logs" every time a new user signs up so the marketing team knows if the app is growing.
*   **Banking (Monitoring):** If the "Transfer" feature is slow, the **Monitor** sends an SMS to the developer: *"System is slow! Fix it now!"*

## Interview Questions
**1. What is a "Try-Catch" block?**
*   **A:** It’s a mechanism to "try" a block of code and "catch" any errors that occur, preventing the entire application from crashing.

**2. Difference between "Logging" and "Monitoring"?**
*   **A:** **Logging** is for searching the "Past" (what happened). **Monitoring** is for checking the "Present" (is it working now?).

**3. What are the common "Log Levels"?**
*   **A:** DEBUG, INFO, WARN, ERROR, and FATAL.

**4. What is a "Centralized Logging System"?**
*   **A:** Since a big app has 100 servers, you can't check each one. You send all logs to one place (like **ELK Stack** or **Splunk**).

**5. How to handle errors in an API?**
*   **A:** By using standard HTTP status codes (400 for user errors, 500 for server errors) and sending a clear, safe message to the frontend.
