# 🧱 S.O.L.I.D Principles

## Simple Explanation
Imagine you are building a **Lego Castle**. 
*   If all the bricks are glued together, you can't change the tower without breaking the whole castle. (Bad Design).
*   If the bricks are independent and fit perfectly, you can swap a Blue tower for a Red one easily. (SOLID Design).

SOLID is a set of **5 Rules** that ensure your code is like a Lego set—easy to snap together, easy to change, and hard to break.

---

## 🛠️ What is S.O.L.I.D?
It is an acronym for 5 design principles:
1.  **S - Single Responsibility:** One class, one job.
2.  **O - Open/Closed:** Open for extension, closed for modification.
3.  **L - Liskov Substitution:** Children must be able to stand in for their parents.
4.  **I - Interface Segregation:** Don't force a class to do things it doesn't need.
5.  **D - Dependency Inversion:** Depend on "Ideas," not "Tools."

---

## 🚀 Deep Dive: The 5 Principles

### 1. **S**ingle Responsibility (SRP)
*   **Rule:** A class should have only ONE reason to change.
*   **Example:** A `User` class should only handle User data. It should NOT handle "Sending Emails" or "Saving to Database." Create an `EmailService` and a `UserRepository` instead.

### 2. **O**pen/Closed (OCP)
*   **Rule:** You should be able to add new features without changing the old code.
*   **Example:** If you want to add a new "Discount Type," don't add an `if-else` in the `PriceCalculator`. Instead, create a new `Discount` class that the calculator can use.

### 3. **L**iskov Substitution (LSP)
*   **Rule:** If a function works with a `Bird`, it must also work with a `Penguin`. 
*   **Common Mistake:** Making an `Ostrich` class inherit from `Bird` and then throwing an error in the `fly()` method. If an Ostrich can't fly, it shouldn't inherit from a "FlyingBird" class!

### 4. **I**nterface Segregation (ISP)
*   **Rule:** A class shouldn't have to implement methods it doesn't use.
*   **Example:** A "Simple Printer" shouldn't have to implement a `fax()` method just because it inherits from a "Smart Device" interface. Break into `Printer` and `FaxMachine` interfaces.

### 5. **D**ependency Inversion (DIP)
*   **Rule:** High-level logic shouldn't care about low-level details.
*   **Example:** Your `Store` shouldn't depend directly on `PayPal`. It should depend on a `PaymentProcessor` "Idea." This way, you can swap PayPal for Stripe in 5 minutes without changing the Store code.

---

## 🏢 Real-World Example: A Universal Remote
A **Universal Remote** follows SOLID. 
*   It works with any TV (LSP). 
*   It only has buttons you need (ISP). 
*   You can add a new TV brand without opening the remote's hardware (OCP).

---

## 💻 Code Example: Dependency Inversion (DIP) in Node.js
```javascript
// ❌ BAD: Store is "Tied" to PayPal
class Store {
    constructor() { this.payment = new PayPal(); } // Hard-coded
}

// ✅ GOOD: Store depends on an "Idea" (PaymentProcessor)
class Store {
    constructor(paymentProcessor) {
        this.paymentProcessor = paymentProcessor;
    }
    
    purchase(amount) {
        this.paymentProcessor.pay(amount);
    }
}

// Now we can use ANY processor!
const myStore = new Store(new StripeProcessor());
myStore.purchase(100);
```

## 🎤 Interview Questions
**Q1: What does SOLID stand for?**
*   **A:** Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, and Dependency Inversion.

**Q2: What is the "Open/Closed" principle?**
*   **A:** It means you should be able to add new functionality (Open for extension) without modifying the existing, tested code (Closed for modification).

**Q3: Why is SRP important?**
*   **A:** It makes code easier to understand and test. If a class only does one thing, there are fewer chances for a bug when you change it.
