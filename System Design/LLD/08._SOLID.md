# 💎 SOLID Principles (The Clean Code Bible)

## Simple Explanation
Imagine you are building a **Modular Kitchen**. 
*   The **Sink** only handles water. (Single Responsibility).
*   You can add a new **Dishwasher** without breaking the existing pipes. (Open/Closed).
*   Any **Tap** you buy from the store should fit into your sink perfectly. (Liskov Substitution).
*   Your **Stove** shouldn't have a button for "Water" if it only burns gas. (Interface Segregation).
*   The **Kitchen** shouldn't depend on one specific brand of "Fridge"—it should work with any fridge. (Dependency Inversion).

**SOLID** is a set of 5 rules that make your code clean, easy to change, and hard to break.

---

## 1. **S**ingle Responsibility Principle (SRP)
### What
A class should have **only one job**.

### Why
If your `User` class is also sending emails and saving to the database, changing the email format might accidentally break the user login!

### Real-World Example
A **Chef** only cooks. A **Waiter** only serves. If the Chef repeats the order to the customer, they might get confused and burn the food!

---

## 2. **O**pen/Closed Principle (OCP)
### What
Code should be **Open for extension** (you can add new features) but **Closed for modification** (you don't change existing code).

### Why
Every time you touch "old code," you risk adding new bugs.

### Real-World Example
A **Smartphone**. You can add new "Apps" (Extension) to it, but you don't need to open the phone's case and change the wires (Modification).

---

## 3. **L**iskov Substitution Principle (LSP)
### What
A **Child class** should be able to stand in for its **Parent** without breaking anything.

### Why
If a "Square" inherits from "Rectangle" but acts differently, your "Area Calculator" might give the wrong result and crash.

### Real-World Example
If you replace a **1.5V AA Battery** with a **1.5V Rechargeable Battery**, your remote should still work perfectly. If it doesn't, you broke LSP!

---

## 4. **I**nterface Segregation Principle (ISP)
### What
Don't force a class to have methods it **doesn't use**. Split big interfaces into small, specific ones.

### Why
A "Simple Printer" shouldn't be forced to have a `scan()` and `fax()` function if it can't do them. It just makes the code messy.

### Real-World Example
A **Restaurant Menu**. You should have a "Pizza Menu" and a "Drink Menu" separately. You shouldn't force a user to look at the "Wine List" if they only want water.

---

## 5. **D**ependency Inversion Principle (DIP)
### What
Your high-level code shouldn't depend on "Specific Tools." It should depend on "General Ideas" (Abstractions).

### Why
If your app only works with **MySQL**, and tomorrow you want to switch to **MongoDB**, you will have to rewrite everything. If you depend on a "Generic Database Interface," switching is easy!

### Real-World Example
An **Electric Socket**. Your lamp doesn't care if the electricity comes from "Solar" or "Coal." It just needs a "Socket" (The Abstraction) to work.

## Practice Tasks
- **Task 1 (Basic):** You have a `Report` class that calculates data and also prints it to a PDF. Which SOLID principle is being broken?
- **Task 2 (Thinking):** Why is the **Open/Closed Principle** good for teamwork? (Hint: Can 2 people work on the same file?)
- **Task 3 (Design):** You are building a "Music App." You have a `BasicUser` and a `PremiumUser`. How would you use **Liskov Substitution** to handle their `playSong()` feature?

## Interview Questions
**1. What does SOLID stand for?**
*   **A:** Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, and Dependency Inversion.

**2. What is SRP?**
*   **A:** It means a class should only have one reason to change (one single responsibility).

**3. What is the benefit of OCP?**
*   **A:** It allows you to add new features without risking bugs in your existing, tested code.

**4. Can you explain the Liskov Substitution Principle simply?**
*   **A:** It means objects of a superclass should be replaceable with objects of its subclasses without affecting the correctness of the program.

**5. Why is Dependency Inversion important for large apps?**
*   **A:** It makes the app "Plug-and-Play." You can swap out a database, a payment gateway, or an email service without changing your core business logic.
