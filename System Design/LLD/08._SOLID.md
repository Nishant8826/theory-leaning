# S.O.L.I.D Principles

S.O.L.I.D is an **acronym for five core object-oriented design
principles** that help developers write **clean, maintainable, scalable,
and testable code**.

S.O.L.I.D stands for: - 
- **S** -- Single Responsibility Principle 
- **O** -- Open/Closed Principle  
- **L** -- Liskov Substitution Principle  
- **I** -- Interface Segregation Principle 
- **D** -- Dependency Inversion Principle

### Why S.O.L.I.D Matters

-   Cleaner code
-   Easier maintenance
-   Better scalability
-   Easier testing

------------------------------------------------------------------------

# **S** : Single Responsibility Principle (SRP)

> The Single Responsibility Principle (SRP) states that a class, function, or module should have one, and only one, reason to change

### Why SRP Matters
- `Reduced Coupling`: Changes to one feature (e.g., how you log data) won't break unrelated features (e.g., how you calculate prices).
- `Better Testability`: It is significantly easier to write unit tests for a function that only does one thing.
- `Reusability`: Small, focused modules can be reused across different parts of an application.
``` js
// Responsibility 1: Only handles user data
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }
}

// Responsibility 2: Only handles data persistence
class UserRepository {
  save(user) {
    console.log(`Saving ${user.name} to the database...`);
  }
}

// Responsibility 3: Only handles email communication
class EmailService {
  sendWelcome(user) {
    console.log(`Sending welcome email to ${user.email}...`);
  }
}

// Usage
const newUser = new User("Jane Doe", "jane@example.com");
const repo = new UserRepository();
const mailer = new EmailService();

repo.save(newUser);
mailer.sendWelcome(newUser);
```

# **O** : Open/Closed Principle (OCP)

> The Open/Closed Principle (OCP) states that software entities (classes, modules, functions, etc.) should be open for extension but closed for modification. In practice, this means you should be able to add new functionality without changing existing, tested code. Adhering to this principle reduces the risk of introducing bugs into stable parts of your application and makes it more scalable.

❌ The Violating Example (Bad)

This AreaCalculator class uses a switch statement to handle different shapes. Every time you add a new shape (like a Triangle), you must modify the existing calculate method, which risks breaking the code for existing shapes. 
``` js
class AreaCalculator {
  calculate(shape) {
    switch (shape.type) {
      case 'square':
        return shape.size * shape.size;
      case 'circle':
        return Math.PI * shape.radius ** 2;
      // Adding a new shape requires modifying this switch statement
      default:
        throw new Error("Unknown shape");
    }
  }
}
```
✅ Adhering to OCP (Good)
Instead of a central switch, let each shape define its own area logic. The AreaCalculator now only needs to call a common method. To add a new shape, you simply extend the system by creating a new class—the original calculator remains untouched.

```js
// Base class or interface-like structure
class Shape {
  getArea() {
    throw new Error("Method 'getArea()' must be implemented.");
  }
}

class Square extends Shape {
  constructor(size) {
    super();
    this.size = size;
  }
  getArea() { return this.size * this.size; }
}

class Circle extends Shape {
  constructor(radius) {
    super();
    this.radius = radius;
  }
  getArea() { return Math.PI * this.radius ** 2; }
}

// This class is now "Closed for Modification"
class AreaCalculator {
  calculate(shape) {
    return shape.getArea();
  }
}

// Adding a Triangle is now an "Extension" without touching existing code
class Triangle extends Shape {
  constructor(base, height) {
    super();
    this.base = base;
    this.height = height;
  }
  getArea() { return (this.base * this.height) / 2; }
}
```
### Key Benefits of OCP

- `Stability`: You don't risk breaking existing, production-ready features when adding new ones.
- `Easier Testing`: Since old code hasn't changed, you only need to write unit tests for the new extensions.
- `Reduced Complexity`: Avoids long, unmanageable if-else or switch blocks as the project grows.

# **L** : Liskov Substitution Principle (LSP)

> Subclasses must be replaceable for their base class.

``` js
class Bird {}
```

------------------------------------------------------------------------

# **I** : Interface Segregation Principle (ISP)

> No client should depend on unused methods.

``` js
class Printer {
  print() {}
}
```

------------------------------------------------------------------------

# **D** : Dependency Inversion Principle (DIP)

> Depend on abstractions, not implementations.

``` js
class Database {}
```
