## 2. Encapsulation

**Encapsulation** means **keeping data safe inside a class** and allowing access to it **only through controlled methods**.

In simple words:  
👉 **Data + Methods = One unit (class)**  
👉 **Direct access is blocked, safe access is allowed**

---

## Core Idea (Very Simple)
Think of encapsulation like a **capsule**:
- Inside: data (variables)
- Outside: buttons (methods) to interact safely

You **cannot touch the data directly**, you must go through proper methods.

---

## Core Mechanisms

### 1. Data Hiding
Internal data is hidden using **private variables**.

### 2. Getter and Setter Methods
- **Getter** → read data
- **Setter** → update data (with validation)

---

## JavaScript Example (Industry-Level Simple)

### Example: User Profile System
```js
class User {
  #age; // private variable

  constructor(age) {
    this.setAge(age);
  }

  getAge() {
    return this.#age;
  }

  setAge(age) {
    if (age < 0) {
      throw new Error("Age cannot be negative");
    }
    this.#age = age;
  }
}

const user = new User(25);
user.getAge();   // ✅ allowed
user.#age = 10; // ❌ error (cannot access private field)
```

✔ **Data is protected**  
✔ **Validation is enforced**  
✔ **External code stays safe**

## Access Modifiers (JavaScript Context)

JavaScript mainly supports:

- Private → #variable
- Public → normal variables & methods

Note: protected exists conceptually but not strictly enforced in JavaScript.

```js
class Account {
  #balance = 0; // private

  deposit(amount) {
    this.#balance += amount;
  }

  getBalance() {
    return this.#balance;
  }
}
```

## Key Benefits (Simple)
- `Data Safety`: No one can change data directly
- `Easy Maintenance`: Internal logic can change without breaking code
- `Clean Code`: Everything related stays in one place

## Real-World Analogy
- Capsule Medicine: You take it, don’t open it
- ATM PIN: You enter it, can’t see the logic
- Encapsulation: Use data safely, don’t touch internals
