# 🌐 API Design

## Simple Explanation
Suppose you are in a **Library**. There is a counter where you can:
1.  **Ask** for a book (GET).
2.  **Give** a new book (POST).
3.  **Correct** a mistake in a book's name (PATCH/PUT).
4.  **Remove** an old book (DELETE).

The **API** is that counter. Without it, you can't talk to the library's data.

---

## 🛠️ What is API Design?
It is the process of creating a clean, easy-to-use, and secure way for the computer and the server to "talk" to each other.

## ❓ Why is it needed?
1.  **To be understood:** If your API names are `get_a`, `get_b`, nobody will understand them.
2.  **To be secure:** You must ensure only the "owner" of the book can delete it.
3.  **To be fast:** A well-designed API only sends the data the user actually needs.

## 🏗️ How does it work? (The Flow)
User Request → API Endpoint (`/users/1`) → (Check Auth) → Service Logic → Database → JSON Response.

---

## 🚀 Deep Dive (Important Interviews Topic)

### 1. REST (The Common Way)
*   Representational State Transfer. It’s a set of rules for making APIs.
*   **Main Methods:**
    *   `GET`: Fetch data (e.g., Load a profile).
    *   `POST`: Create data (e.g., New post).
    *   `PUT`: Replace data (e.g., Update all profile info).
    *   `PATCH`: Change part of the data (e.g., Just Change Bio).
    *   `DELETE`: Remove data.

### 2. Status Codes (The "Quick Answers")
The server gives a 3-digit code to tell you "What happened?":
*   `200 OK`: Success!
*   `201 Created`: New item is made.
*   `400 Bad Request`: You sent something wrong (form mistake).
*   `401 Unauthorized`: You need to login.
*   `403 Forbidden`: You are logged in, but not allowed here.
*   `404 Not Found`: It doesn't exist.
*   `500 Internal Server Error`: Something broke on the server!

## 🏢 Real-World Example: Instagram Feed API
When you refresh your feed, your phone calls an API like:
`GET https://instagram.com/api/v1/feed?page=1`
The server responds with a JSON "package" that has the 20 newest photos for you.

## 💻 Code Example: A Simple Express.js API
```javascript
const express = require('express');
const app = express();
app.use(express.json());

// GET - Ask for a user
app.get('/api/users/:id', (req, res) => {
    const userId = req.params.id;
    res.json({ id: userId, name: "Nishant", bio: "Design expert!" });
});

// POST - Create a new user
app.post('/api/users', (req, res) => {
    const newUser = req.body;
    console.log("Saving user to DB:", newUser);
    res.status(201).json({ message: "Created!", data: newUser });
});

app.listen(3000);
```

## 🎤 Interview Questions
**Q1: What is the difference between PUT and PATCH?**
*   **A:** PUT replaces the **entire** object. PATCH only updates the **parts** you send (e.g., just the bio).

**Q2: What is an API Endpoint?**
*   **A:** It’s the URL you use to talk to the server (e.g., `/api/login`).

**Q3: What is "Statelessness" in REST?**
*   **A:** Each request from the phone to the server must have **all the info** (like who you are) because the server "forgets" who you are as soon as the last request finishes. (That’s why we use JWT tokens in every request!).
