# 📋 API Design & Class Design

## Simple Explanation
*   **API Design:** Imagine a **Restaurant Menu**. If the menu is messy and uses strange words, you won't know what to order. A good API is like a **clean menu**—it’s easy to read and gives you exactly what you expect.
*   **Class Design:** Imagine a **Lego Block**. It has a specific shape and "Pips" on top so it can connect to other blocks. A good class is **designed** to be small, clear, and easy to connect.

---

## What
*   **API Design:** The set of rules and "Endpoints" (URLs) your Backend uses to talk to the Frontend.
*   **Class Design:** The blueprints of your individual classes (Properties, Methods).

---

## Why
1.  **Clear Communication:** A good API means the Frontend team doesn't have to ask the Backend team 100 questions.
2.  **Scalability:** Well-designed classes are easy to reuse later.
3.  **To Prevent Bugs:** If your API is clear, you won't accidentally send wrong data (like a String where a Number was expected).

---

## Best Practices for API Design
1.  **Use Nouns, Not Verbs:**
    *   ❌ `POST /createOrder`
    *   ✅ `POST /orders`
2.  **Use HTTP Methods Properly:**
    *   `GET` (Read), `POST` (Create), `PUT` (Update), `DELETE` (Remove).
3.  **Version Your API:**
    *   ✅ `/v1/users`, `/v2/users`. (This prevents breaking your app for users who haven't updated yet).
4.  **Send Clear Errors:**
    *   Don't just say `500 Server Error`. Say `400 Bad Request: "Email already exists"`.

---

## Best Practices for Class Design
1.  **Be Small (SRP):** One class should only have one job.
2.  **Hide Data (Encapsulation):** Only expose what is necessary.
3.  **Composition over Inheritance:** Instead of always "Inheriting" everything, try to "Include" (Compose) other classes as building blocks.

---

## Real-World Example
*   **Instagram API:** `GET /users/nishant_dev/posts` is easy to understand. It tells you exactly whose posts you are getting.
*   **E-commerce Class:** A `Product` class should only store "What the product is." It shouldn't have code for "Shipping" or "Taxes." We should have separate `Shipping` and `Tax` classes for that.

## Interview Questions
**1. What is a RESTful API?**
*   **A:** It is an API design style that uses standard HTTP methods (GET, POST, etc.) and focuses on "Resources" (like Users or Orders) instead of actions.

**2. Difference between PUT and PATCH?**
*   **A:** **PUT** updates the **entire** object. **PATCH** only updates **part** of the object (like only changing the email).

**3. What is an API Endpoint?**
*   **A:** It is a specific URL (`/api/v1/login`) that performs a specific action in your backend.

**4. Why is Versioning important for APIs?**
*   **A:** It allows you to make big changes to your Backend without breaking the "Old Version" of your Mobile App that some users are still using.

**5. What is "High Cohesion" in class design?**
*   **A:** It means all the methods inside a class are closely related and help the class do its **one** job.
