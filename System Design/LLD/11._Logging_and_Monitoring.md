# 📋 Logging & Monitoring

## Simple Explanation
Imagine you are a **Pilot** flying a massive airplane. 
*   **Logging:** A "Black Box" in the plane that records every button you press and every instruction you hear. (The History).
*   **Monitoring:** A "Dashboard" in front of you that shows your "Fuel," "Speed," and "Altitude" in real-time. (The Now).

Without these, the plane can crash, and you won't even know why!

---

## 🛠️ What is Logging?
It is the act of **keeping a text record** of what is happening inside the computer—what errors occurred, who logged in, and which requests were processed.

## 🛠️ What is Monitoring?
It is the practice of **watching the "Vital Signs"** of your server—how much RAM is used, how fast the requests are, and if the hard drive is full.

---

## ❓ Why are they needed?
1.  **To find bugs:** If User 101 says, "I can't see my feed," you look at the **Logs** to see exactly what happened to User 101 at 10:00 PM.
2.  **To prevent crashes:** If your **Monitoring** shows "Hard Drive is 99% Full," you can fix it before the app dies.
3.  **To see speed:** Monitoring shows you "Load time for home page is 10 seconds!" (Then you know you need caching).

---

## 🏗️ How does it work? (The Flow)
User Error → App creates Log Entry → Saved in **Log Tool (Elastic/Winston)** → Developer searches Logs → Fixed.

---

## 🚀 Deep Dive (Important Interviews Topic)

### 1. Log Levels (Which one to read?)
*   **DEBUG:** Very detailed, tiny things (only for developers).
*   **INFO:** Normal things (e.g., "User logged in").
*   **WARN:** Something strange happened but the app still works (e.g., "Login took 5 seconds").
*   **ERROR:** Something broke (e.g., "Database is down").
*   **FATAL:** The app has crashed completely!

### 2. Metrics (The Dashboard)
Things we watch in **Monitoring**:
*   **Latency:** How slow is the app? (Goal: less than 100ms).
*   **Throughput:** How many users are clicking right now? (Goal: 10,000 per minute).
*   **Error Rate:** What % of clicks lead to a 500 error? (Goal: 0.01%).

---

## 🏢 Real-World Example: An E-commerce Crash
A big sale starts. Suddenly, the site is slow.
*   **Monitoring** shows: "CPU is at 100%!"
*   **Logging** shows: "Error: Database connection timeout in Order Service."
*   **Solution:** The developer sees the logs and adds 2 more database copies instantly!

## 💻 Code Example: Simple Logging with Winston (Node.js)
```javascript
const winston = require('winston');

// A simple Winston logger
const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [new winston.transports.File({ filename: 'app.log' })]
});

// Using the logger
const loginUser = (user) => {
    logger.info(`User ${user.id} logged in from India.`);
    if (!user.email) logger.error("Missing email in user profile!");
};
```

## 🎤 Interview Questions
**Q1: What is the difference between Logging and Monitoring?**
*   **A:** Logging is about **Events** (The History). Monitoring is about **Metrics** (The dashboard of "Now").

**Q2: What is an "Alert"?**
*   **A:** It’s a rule in a monitoring tool that says: *"If the Error Rate goes above 5%, send a Slack message/Email to the developers instantly!"*

**Q3: Why shouldn't we log sensitive data like Passwords or Credit Cards?**
*   **A:** For **Security**. If a hacker gets access to your logs, they will have everyone's secrets. Always "Mask" or remove sensitive data before logging.
