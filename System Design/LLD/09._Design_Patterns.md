# 🎨 Design Patterns

## Simple Explanation
Imagine you are building a **House**. 
*   Instead of reinventing "The Door" or "The Window" every time, you use a **standard 3-bedroom, 2-bath house plan**. 
*   These "Plans" are battle-tested and work for everyone. 

Design Patterns are **repeatable solutions** to common software problems. They are "Shortcuts" to writing solid code that other developers can understand instantly.

---

## 🛠️ What are Design Patterns?
A Design Pattern is a **Template** for solving a problem that occurs frequently in software development. They are divided into three main families:
1.  **Creational:** How we "make" objects (The Factory).
2.  **Structural:** How we "connect" objects (The Adapter).
3.  **Behavioral:** How objects "talk" to each other (The Observer).

---

## 🚀 Deep Dive: The Top 3 Patterns (Crucial for Interviews)

### 1. Singleton (Creational)
*   **Rule:** Only **ONE** copy of a class exists in the entire app.
*   **Use Case:** A "Database Connection" or "App Theme." You don't want 50 different connections to one database.
*   **Example:** A single "Manager" at the gate of a building.

### 2. Observer (Behavioral)
*   **Rule:** One object (The **Subject**) notifies many others (The **Observers**) when something changes.
*   **Use Case:** YouTube Notifications. When a YouTuber uploads a video (The Subject), all his subscribers (The Observers) get an alert.

### 3. Factory (Creational)
*   **Rule:** Instead of calling `new Car()`, you call `CarFactory.create()`.
*   **Use Case:** A logic that decides whether to make a "SUV" or a "SportsCar" based on user input.

---

## 🏢 Real-World Example: Instagram Features
*   **Singleton:** The main "Auth" service that knows if you are logged in.
*   **Observer:** When you "Like" a post, the `LikeService` notifies the `NotificationService` to send a ping to the owner.
*   **Factory:** When you "Upload" an image, the `MediaFactory` creates an `Image` object or a `Video` object based on the file type.

## 💻 Code Example: Singleton in Node.js
```javascript
// Singleton for a Database Connection
class DatabaseConnection {
    constructor() {
        if (!DatabaseConnection.instance) {
            this.connectionString = 'mysql://host:3306/my_db';
            DatabaseConnection.instance = this;
        }
        return DatabaseConnection.instance;
    }
}

// Both are now looking at the exact same "Instance"!
const db1 = new DatabaseConnection();
const db2 = new DatabaseConnection();
console.log(db1 === db2); // true
```

## 🎤 Interview Questions
**Q1: What are the three categories of Design Patterns?**
*   **A:** Creational, Structural, and Behavioral.

**Q2: What is the Singleton Pattern?**
*   **A:** It ensures a class has only one instance and provides a global point of access to it (e.g., Database Connection).

**Q3: When should we use the Factory Pattern?**
*   **A:** When you have many similar objects and you want to **hide the "creation logic"** so the main code doesn't have to worry about how they are made.
