# 🏛️ The 4 Pillars of OOPS

## Simple Explanation
Suppose you are in a **Car**. 
*   **Encapsulation:** All the complex engine wires are "Hidden" from you. You only see a clean Dashboard. (Security).
*   **Abstraction:** You only need to know "Steer" and "Brake"—you don't need to know how the brakes actually work. (Simplicity).
*   **Inheritance:** A "Sports Car" is still a "Car." It just has a faster engine. It "Inherits" all the basics like wheels and pedals. (Reuse).
*   **Polymorphism:** You can "Press" a button to Honk. If it’s a Normal Car, it says "Beep." If it’s an Ambulance, it says "Wooo-Wooo." One name, Many sounds! (Flexibility).

---

## 🛠️ What are the 4 Pillars?
They are the four core ideas that make OOPS a powerful way to organize code:
1.  **Encapsulation (Security):** Keeping data safe inside the "Capsule" (the object).
2.  **Abstraction (Simplicity):** Showing only what’s necessary and hiding the mess.
3.  **Inheritance (Reuse):** Using an old class to build a new one.
4.  **Polymorphism (Flexibility):** Same action, different behaviors.

---

## ❓ Why are they needed?
1.  **To build big apps:** Without these, your code will become a giant plate of "Spaghetti" (hard to follow).
2.  **To prevent bugs:** Encapsulation ensures no one accidentally changes "Bank Balance" without permission.
3.  **To save time:** Inheritance lets you reuse code from yesterday to build features for today.

---

## 🏢 Real-World Example: An ATM Machine
1.  **Encapsulation:** You can't see the cash-counting sensors. (They are private).
2.  **Abstraction:** You only press the "Withdraw" button. (You don't care how it counts).
3.  **Inheritance:** A "Savings ATM" and a "Salary ATM" both inherit from a "Basic ATM."
4.  **Polymorphism:** The "Check Balance" button shows $1,000 for User A and $5,000 for User B. (Same action, different result).

---

## 💻 Code Example: The 4 Pillars in Action
```javascript
// A simple example for a Library app
class User { 
    #pin = 1234; // Encapsulation: The '#' makes it "Private" (Hidden)

    login(inputPin) {
        // Abstraction: User just calls login(), not this logic
        return inputPin === this.#pin;
    }
}

class Admin extends User { // Inheritance: Admin is a type of User
    deleteBook() {
        console.log("Admin can delete books!");
    }
}
```

## 🎤 Interview Questions
**Q1: What are the 4 main pillars of OOPS?**
*   **A:** Encapsulation, Abstraction, Inheritance, and Polymorphism. (Remember them as **A-PIE**).

**Q2: What is the benefit of Abstraction?**
*   **A:** It reduces complexity. Developers only need to know "what" a function does, not the "how."

**Q3: Is Inheritance always good?**
*   **A:** No. If you use it too much, your code becomes "tightly coupled" (hard to change). Sometimes "Composition" is better than Inheritance.
