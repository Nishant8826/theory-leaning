# Object-Oriented Programming (OOPS)

## 1. History of Programming Before OOPS

### a) Early Programming (1940s–1950s)
- Programming was done in **machine language** (binary 0s and 1s).
- Extremely difficult to write, read, debug, and maintain.
- Programs were hardware-dependent.

### b) Assembly Language (1950s)
- Used symbolic instructions (ADD, MOV, etc.).
- Easier than machine language but still:
  - Hardware-dependent
  - Complex for large programs

### c) Procedural Programming (1960s–1970s)
Examples:
- C
- FORTRAN
- COBOL
- Pascal

**Key features:**
- Programs were divided into **functions/procedures**.
- Focus was on **steps (algorithms)** rather than data.

**Problems with Procedural Programming:**
- Data was **global** and unprotected.
- No clear way to model **real-world entities**.
- Difficult to manage **large and complex software**.
- Code reuse was limited.
- Maintenance became hard as program size increased.


## 2. Why OOPS Came Into Picture

As software systems grew larger (operating systems, databases, enterprise software), procedural programming failed to handle complexity effectively.

### Major Problems That Needed a Solution
1. **Poor data security**
2. **High complexity**
3. **Low reusability**
4. **Difficult maintenance**
5. **No real-world representation**

To solve these issues, **Object-Oriented Programming (OOPS)** was introduced.


## 3. Birth of Object-Oriented Programming

- Concept first introduced in **1960s**
- First true OOP language: **Simula (1967)** – developed for simulations
- Later improvements:
  - **Smalltalk (1970s)** – pure object-oriented language
  - **C++ (1980s)** – combined OOP with C
  - **Java (1995)** – platform-independent OOP
  - **Python, C#, Ruby** – modern OOP languages


## 4. What Is OOPS?

**Object-Oriented Programming System (OOPS)** is a programming paradigm based on the concept of **objects**, which contain:
- **Data (attributes)**
- **Methods (functions)**

Instead of focusing on functions, OOPS focuses on **objects and their interactions**.


## 5. Why OOPS Is Better

### a) Real-World Modeling
- Programs are designed using real-world entities like:
  - Student
  - Car
  - BankAccount

### b) Data Security
- **Encapsulation** hides data and protects it from unauthorized access.

### c) Code Reusability
- **Inheritance** allows reuse of existing code.

### d) Flexibility and Scalability
- **Polymorphism** allows one interface with multiple implementations.

### e) Easy Maintenance
- Changes in one class do not affect the entire program.


## 6. Core Principles of OOPS

1. **Class** – Blueprint of an object
2. **Object** – Instance of a class
3. **Encapsulation** – Binding data and methods together
4. **Abstraction** – Hiding implementation details
5. **Inheritance** – Acquiring properties of another class
6. **Polymorphism** – One name, many forms


## 7. Conclusion

OOPS came into picture to:
- Handle **large and complex software**
- Improve **security and maintainability**
- Promote **code reuse**
- Model **real-world problems efficiently**

Today, OOPS is the foundation of most modern software systems.
