# 🌀 Consistent Hashing

## Simple Explanation
Suppose you have 3 friends sharing the work of "Sorting Mail." Each friend handles mail for certain letters of the alphabet (A-H, I-P, Q-Z).

But what if a new friend joins? You have to move **all** the mail around to give the new friend some work. That's a mess!

**Consistent Hashing** is a clever way to organize this "Circle" so that if a friend joins or leaves, you only have to move a **tiny bit** of mail.

---

## 🛠️ What is Consistent Hashing?
It’s a way to distribute data across multiple "nodes" (servers) so that adding or removing a node causes the **least amount of movement** of data.

In regular hashing, adding one server might mean moving 90% of your data. In Consistent Hashing, you only move about **1/N** (e.g., if you have 10 servers, you only move 10% of your data).

## ❓ Why is it needed?
1.  **To scale easily:** Adding a new server shouldn't take hours of moving data.
2.  **To handle failures:** If a server crashes, we only lose its data (which we redistribute to only one neighbor).
3.  **For Caching:** It ensures that a specific key (like a user's ID) always maps to the same server to fetch their high-speed "Cache" data.

## 🏗️ How does it work? (The Circle)
1.  Imagine a **Hash Ring** (a 360-degree circle).
2.  Both the **Servers** and the **Data Keys** are mapped to different spots on this circle.
3.  Each data key "looks clockwise" along the circle to find the next server. That server becomes the "owner" of that data.

---

## 🏢 Real-World Example: Amazon DynamoDB
Amazon uses Consistent Hashing to store shopping cart data across thousands of servers. If one server in a data center catches fire, only a fraction of the users are affected, and their data is instantly moved to the "Next" server on the ring.

---

## 💻 Code Example: A Mock Hash Ring Concept
(Note: Real rings use high-level math, but this demonstrates the "finding the owner" logic).
```javascript
const servers = [100, 250, 600]; // Their "positions" on the 0-1000 ring
const getDataOwner = (keyHash) => {
    // Find the first server whose position > keyHash
    for (const pos of servers) {
        if (pos >= keyHash) return pos;
    }
    return servers[0]; // If it's at the end, circle back to the start
};

console.log(`Owner for key at 150: Server at ${getDataOwner(150)}`);
console.log(`Owner for key at 800: Server at ${getDataOwner(800)}`);
```

## 🎤 Interview Questions
**Q1: What is the main problem with regular (Modulo) hashing?**
*   **A:** If you change the number of servers (N), the position of every single key changes (`index = hash % N`). This means moving almost all the data.

**Q2: What are "Virtual Nodes"?**
*   **A:** To make the distribution more even, we can place the same server at **multiple spots** on the ring. This prevents one server from being unlucky and getting too much data.

**Q3: Which systems use Consistent Hashing?**
*   **A:** Cassandra, DynamoDB, and many Load Balancers use it to distribute requests efficiently.
