# 🏗️ System Design Basics

## Simple Explanation
Imagine you are opening a small tea shop. At first, only 5 people come every day. You can handle them alone with one stove and one pan. But suddenly, your tea goes viral! Now 1,000 people are standing outside. 

If you don't change anything, your shop will "crash" because you can't work that fast. System Design is the art of planning how to add more "stoves" (CPU), more "pans" (Memory), and more "staff" (Servers) so that your shop never stops serving tea, no matter how many people show up.

---

## ☁️ What is System Design?
It is the process of defining the architecture, components, and data of a system to meet specific requirements. In simple words, it’s a **Blueprint** for building an app that can handle millions of users without breaking.

## ❓ Why is it needed?
1.  **To handle growth:** So the app doesn't slow down when users increase.
2.  **To prevent crashes:** So if one server fails, the app stays alive.
3.  **To save money:** By using resources efficiently.
4.  **To keep data safe:** Ensuring information is never lost.

## 🛠️ How does it work?
We focus on four main pillars, often called "System Properties":

### 1. Scalability (Adding Power)
How easily can we make the system bigger?
*   **Vertical Scaling:** "Buying a bigger stove." Adding more RAM/CPU to one single machine. (Simple but has a limit).
*   **Horizontal Scaling:** "Buying 10 more stoves." Adding more machines work together. (Unlimited growth, used by Google/Meta).

### 2. Reliability (Never Losing Data)
Can the system keep working even if something breaks?
*   **How:** We use `Redundancy` (keeping copies). If one hard drive burns, we use the copy.

### 3. Availability (Always Online)
Is the website "Up" when I click the link?
*   **High Availability:** We aim for "Five Nines" (99.999% uptime), meaning the site is only down for a few minutes a year.

### 4. Consistency (Same Data for Everyone)
Do all users see the same thing? If I change my profile picture, does my friend see the new one or the old one?

---

## 🏢 Real-World Example: The "Flash Sale"
Imagine an e-commerce site like Amazon during a Big Sale.
*   **Scalability:** They add 5,000 extra servers just for that day.
*   **Availability:** They ensure the "Buy Now" button works every second.
*   **Reliability:** Even if a server crashes mid-payment, they ensure your money and order are safe.

## 💻 Code Example: Horizontal Scaling Concept
In Node.js, we often use the `cluster` module to use all the "Cores" of a CPU. This is like having 4 cooks in one kitchen instead of 1.

```javascript
const cluster = require('cluster');
const http = require('http');
const numCPUs = require('os').cpus().length;

if (cluster.isMaster) {
  console.log(`Master ${process.pid} is running`);

  // Create "Workers" (One for each CPU Core)
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }
} else {
  // Each worker can handle its own traffic
  http.createServer((req, res) => {
    res.writeHead(200);
    res.end('Hello! This request is handled by a Worker.');
  }).listen(8000);
}
```

## 🎤 Interview Questions
**Q1: What is the difference between Scalability and Availability?**
*   **A:** Scalability is about handling *more* load. Availability is about staying *online* even during a failure.

**Q2: Why is Horizontal Scaling better than Vertical Scaling?**
*   **A:** Horizontal scaling is cheaper, allows for unlimited growth, and has no "Single Point of Failure." If one machine dies, the others take over.

**Q3: What is Redundancy?**
*   **A:** It is the practice of keeping extra copies of data or servers so that if one fails, the "redundant" ones can continue the work.
