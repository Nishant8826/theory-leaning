# 🛡️ Rate Limiting

## Simple Explanation
Suppose you have a free **Drinking Water Stand** in your city. Everyone is welcome to take one bottle of water. 

But suddenly, someone shows up with 1,000 empty bottles and starts filling them all, taking all the water. Now, nobody else can get a single bottle. To stop this, you hire a guard who says: **"Only one bottle per person every 10 minutes."** This guard is your **Rate Limiter**.

---

## 🛠️ What is Rate Limiting?
It is a "Traffic Guard" that limits how many times a user or IP address can call an API in a certain amount of time.

## ❓ Why is it needed?
1.  **To stop hackers/bots:** A bot can click "Sign Up" 1,000 times a second to crash your server. A Rate Limiter stops them.
2.  **To save money:** Each API call (like "Search") costs you money in CPU and Database power.
3.  **To ensure fairness:** So one single user doesn't take all the resources from the other 999,999 users.

## 🏗️ How does it work? (The Flow)
User Click → Rate Limiter (Check: Did they click 5 times already?) → If Yes, **Block (429 Too Many Requests)**. If No, **Allow**.

---

## 🚀 Deep Dive (Important Interviews Topic)

### 1. Common Algorithms
*   **Token Bucket (The most popular):** A user has a "Bucket" of 5 tokens. Every click uses 1 token. Every minute, 1 new token is added. If the bucket is empty, they are blocked. (Simple and flexible).
*   **Fixed Window:** You allow 100 requests every minute. If you reach 100 at second 59, you are blocked until second 60. (Easy but not perfectly smooth).
*   **Sliding Window:** A smarter way that keeps track of the "Last 60 seconds" accurately at all times.

### 🏢 Real-World Example: Twitter (X) or GitHub
Try to refresh your Twitter feed 1,000 times in 1 minute. Eventually, it will show an error: *"You have reached your limit. Please try again later."* They use Rate Limiting to ensure that millions of people can all see their feeds without the servers catching fire.

## 💻 Code Example: A Mock Token Bucket in Node.js
```javascript
let tokens = 3; // Starting with 3 tokens

const requestAPI = (userId) => {
    if (tokens > 0) {
        tokens--;
        console.log(`Access Granted for ${userId}! Remaining tokens: ${tokens}`);
        return 200; // Success
    } else {
        console.log(`Access Denied for ${userId}! (Too many requests)`);
        return 429; // Error Code for Rate Limit
    }
};

// Refill tokens every 5 seconds
setInterval(() => {
    if (tokens < 3) tokens++;
    console.log("Refilling tokens...");
}, 5000);

requestAPI(101);
```

## 🎤 Interview Questions
**Q1: What is the HTTP Status Code for Rate Limiting?**
*   **A:** 429—Too Many Requests.

**Q2: What is "Throttling"?**
*   **A:** It’s a softer way of Rate Limiting. Instead of blocking the user completely, we just "Slow down" their requests (like making them wait 2 extra seconds for each answer).

**Q3: Where should we place the Rate Limiter?**
*   **A:** Usually at the **Edge**—like in the API Gateway or Load Balancer. This way, we block malicious traffic before it even enters our expensive servers.
