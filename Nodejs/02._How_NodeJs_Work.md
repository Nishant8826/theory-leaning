# How Node.js Works

## Simple Explanation
Node.js is **single-threaded** — it does one thing at a time on the main thread. But it can handle **thousands of requests at the same time** without freezing, thanks to its **Event Loop** and **non-blocking I/O**.

Think of it like a **coffee shop with one barista**:
- The barista (Event Loop) takes your order (request).
- While the coffee machine (I/O operation) brews, the barista takes the next person's order.
- When the coffee is ready, the barista serves it (callback fires).

---

## Real-World Example
When your API receives 1000 users logging in at the same time:
- Node.js doesn't create 1000 threads.
- Instead, it delegates I/O (DB queries) to libuv's thread pool.
- The Event Loop picks up results and responds.

---

## Step-by-Step: How a Request is Handled

```
1. Client sends a request
        ↓
2. Request goes into the Event Queue
        ↓
3. Event Loop picks it up (FIFO order)
        ↓
4. Is it non-blocking? (e.g., simple math, string processing)
   YES → Execute immediately → Send response
        ↓
   NO (blocking I/O: file read, DB query, API call)
        ↓
5. Delegate to libuv Thread Pool
        ↓
6. Event Loop moves on to next request (doesn't wait!)
        ↓
7. When thread pool finishes → registers a "completion" callback
        ↓
8. Event Loop picks up the callback → sends response to client
```

## Code Example

### Non-blocking (async) vs Blocking (sync)

```js
const fs = require("fs");

// ✅ Non-blocking — doesn't freeze the server
fs.readFile("data.txt", "utf8", (err, data) => {
  if (err) throw err;
  console.log("File read done:", data);
});
console.log("This runs BEFORE file is read!"); // prints first

// ❌ Blocking — freezes everything until done
const data = fs.readFileSync("data.txt", "utf8");
console.log("File data:", data);
console.log("This waits..."); // prints AFTER file is fully read
```

## The Event Loop in Detail

The Event Loop is a continuous cycle that checks queues:

```
           ┌─────────────────────────┐
           │       Call Stack        │ ← Currently executing code
           └──────────┬──────────────┘
                      │ (stack empty?)
           ┌──────────▼──────────────┐
           │    Microtask Queue      │ ← Promises, process.nextTick
           │    (Higher Priority)    │
           └──────────┬──────────────┘
                      │ (microtasks done?)
           ┌──────────▼──────────────┐
           │   Macrotask Queue       │ ← setTimeout, setInterval, I/O callbacks
           └─────────────────────────┘
```

### Priority Order:
1. **Synchronous code** (runs first, on call stack)
2. **`process.nextTick()`** (runs before any Promises)
3. **Promises / microtasks** (run before next I/O event)
4. **I/O callbacks** (file, network, DB)
5. **`setTimeout` / `setInterval`**

```js
console.log("1 - sync");

process.nextTick(() => console.log("2 - nextTick"));

Promise.resolve().then(() => console.log("3 - Promise"));

setTimeout(() => console.log("4 - setTimeout"), 0);

console.log("5 - sync");

// Output: 1 → 5 → 2 → 3 → 4
```

## libuv & Thread Pool

**libuv** is a C++ library underneath Node.js that:
- Manages the Event Loop
- Provides a **Thread Pool** (default: 4 threads)
- Handles OS-level async I/O

Operations handled by the thread pool:
- File system operations (`fs.readFile`)
- DNS lookups
- Crypto operations (`bcrypt.hash`)
- Compression (zlib)

> ⚠️ Network I/O (HTTP requests, TCP) is handled by the OS itself — NOT the thread pool.

## Interview Questions

- **Q: How does Node.js handle concurrent requests with one thread?**
  A: Via the Event Loop and non-blocking I/O. I/O tasks are delegated to libuv's thread pool. The main thread processes callbacks when I/O completes — never blocking.

- **Q: What is the Event Loop in Node.js?**
  A: A mechanism that continuously monitors the call stack and queues (microtask, I/O, timers). It processes tasks in priority order, enabling async behavior.

- **Q: What is libuv?**
  A: A cross-platform C++ library that powers Node.js's event loop, thread pool, and async I/O operations.

- **Q: What is the difference between blocking and non-blocking I/O?**
  A: Blocking I/O waits for the operation to complete before moving on (freezes the thread). Non-blocking I/O immediately returns and triggers a callback when done.

- **Q: What is `process.nextTick()`?**
  A: A way to schedule a callback to run at the end of the current operation, before any I/O events or timers — higher priority than Promises.

- **Q: What is the thread pool in Node.js?**
  A: A pool of worker threads (default 4) managed by libuv, used for blocking operations like file I/O and crypto so the event loop isn't blocked.

- **Q: Is Node.js good for CPU-intensive tasks?**
  A: No. CPU-heavy tasks (e.g., video encoding, heavy data processing) block the event loop. Use Worker Threads or separate services for those.
