# REST API Design

**REST** (Representational State Transfer) is a set of rules for designing APIs that are consistent, scalable, and easy to use.

A **REST API** is an interface that allows different apps to communicate with each other over HTTP using standard conventions.

---

## Real-World Example
When you use a mobile banking app:
- `GET /accounts/123/balance` → returns your balance
- `POST /transactions` → creates a new transfer
- `GET /transactions?month=march` → lists march transactions

All these are REST API endpoints following consistent conventions.

---

## REST Principles

### 1. Use Nouns for Resources (not verbs)
```
❌ Bad:  GET /getUsers
         POST /createUser
         DELETE /deleteUser/1

✅ Good: GET /users
         POST /users
         DELETE /users/1
```

### 2. Use HTTP Methods Correctly

| HTTP Method | Action | Example |
|------------|--------|---------|
| `GET` | Read / Retrieve | `GET /products` |
| `POST` | Create | `POST /products` |
| `PUT` | Replace (full update) | `PUT /products/1` |
| `PATCH` | Partial Update | `PATCH /products/1` |
| `DELETE` | Delete | `DELETE /products/1` |

### 3. Use Correct HTTP Status Codes
```
200 OK           — Request succeeded
201 Created      — Resource created successfully
204 No Content   — Success but no body (e.g., DELETE)
400 Bad Request  — Client sent invalid data
401 Unauthorized — Not logged in
403 Forbidden    — Logged in but not allowed
404 Not Found    — Resource doesn't exist
409 Conflict     — Duplicate (e.g., email already exists)
422 Unprocessable — Validation failed
500 Internal Server Error — Something crashed on the server
```

### 4. Nest Resources for Relationships
```
GET /users/5/posts          → all posts by user 5
GET /users/5/posts/12       → specific post by user 5
POST /users/5/posts         → create a post for user 5
```

### 5. Filter, Sort, Paginate with Query Strings
```
GET /products?category=electronics
GET /products?minPrice=100&maxPrice=500
GET /users?sort=createdAt&order=desc
GET /posts?page=2&limit=10
```

## Code Example — Full REST API

```js
const express = require("express");
const app = express();
app.use(express.json());

let products = [
  { id: 1, name: "Laptop", price: 50000, category: "electronics" },
  { id: 2, name: "Phone", price: 20000, category: "electronics" },
  { id: 3, name: "Shirt", price: 500, category: "clothing" },
];

// GET all products (with optional query filter)
app.get("/products", (req, res) => {
  const { category, minPrice, maxPrice } = req.query;

  let result = products;
  if (category) result = result.filter((p) => p.category === category);
  if (minPrice) result = result.filter((p) => p.price >= Number(minPrice));
  if (maxPrice) result = result.filter((p) => p.price <= Number(maxPrice));

  res.json({ count: result.length, products: result });
});

// GET single product
app.get("/products/:id", (req, res) => {
  const product = products.find((p) => p.id === parseInt(req.params.id));
  if (!product) return res.status(404).json({ error: "Product not found" });
  res.json(product);
});

// POST create product
app.post("/products", (req, res) => {
  const { name, price, category } = req.body;

  if (!name || !price) {
    return res.status(400).json({ error: "name and price are required" });
  }

  const newProduct = { id: products.length + 1, name, price, category };
  products.push(newProduct);
  res.status(201).json(newProduct);
});

// PUT full update
app.put("/products/:id", (req, res) => {
  const index = products.findIndex((p) => p.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: "Not found" });

  products[index] = { id: parseInt(req.params.id), ...req.body };
  res.json(products[index]);
});

// PATCH partial update
app.patch("/products/:id", (req, res) => {
  const product = products.find((p) => p.id === parseInt(req.params.id));
  if (!product) return res.status(404).json({ error: "Not found" });

  Object.assign(product, req.body); // only update provided fields
  res.json(product);
});

// DELETE
app.delete("/products/:id", (req, res) => {
  const index = products.findIndex((p) => p.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: "Not found" });

  products.splice(index, 1);
  res.status(204).send(); // 204 No Content — no body needed
});

app.listen(3000, () => console.log("REST API on port 3000"));
```

## API Versioning
```js
// Prevent breaking existing clients when you update the API
app.use("/api/v1/users", userRoutesV1);
app.use("/api/v2/users", userRoutesV2);
```

---

## Good REST Response Structure
```json
// Success
{
  "success": true,
  "data": { "id": 1, "name": "Nishant" },
  "message": "User fetched successfully"
}

// Error
{
  "success": false,
  "error": "User not found",
  "statusCode": 404
}
```

## Interview Questions

- **Q: What is REST?**
  A: An architectural style for designing APIs based on HTTP. Key principles: use nouns for resources, correct HTTP methods, stateless communication, consistent response structure.

- **Q: What is the difference between PUT and PATCH?**
  A: PUT replaces the entire resource with the new data. PATCH only updates the provided fields, leaving the rest unchanged.

- **Q: What HTTP status code should you return when creating a resource?**
  A: `201 Created` — signals the resource was successfully created and may include the new resource in the response body.

- **Q: What does `204 No Content` mean?**
  A: Success, but there is no response body to return. Often used for DELETE operations.

- **Q: What is the difference between 401 and 403?**
  A: `401 Unauthorized` means unauthenticated (not logged in). `403 Forbidden` means authenticated but lacking permission.

- **Q: What is API versioning and why is it important?**
  A: Versioning (`/api/v1/`) allows backward-compatible changes — old clients keep working while new clients use updated endpoints.

- **Q: Should you use verbs in REST API URLs?**
  A: No. Use nouns for resources (`/users`, `/products`). The HTTP method (GET, POST, DELETE) communicates the action.
