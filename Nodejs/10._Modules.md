# Modules (CommonJS vs ES Modules)

**Modules** let you split your Node.js code into separate files and share code between them using `require`/`exports` or `import`/`export`.

Node.js supports two module systems:
- **CommonJS (CJS)** = the original Node.js way (`require`)
- **ES Modules (ESM)** = the modern JavaScript standard (`import/export`)

---

## Real-World Example
A Node.js backend app typically has:
- `db.js` — database connection module
- `auth.js` — authentication helpers
- `userRoutes.js` — route definitions
- `index.js` — assembles everything together

---

## CommonJS (CJS) — The Traditional Way

CommonJS is Node.js's built-in module system. Used in most older Node.js projects.

```js
// math.js — export
function add(a, b) { return a + b; }
function multiply(a, b) { return a * b; }
const PI = 3.14159;

module.exports = { add, multiply, PI };

// OR export one thing as default:
// module.exports = add;
```

```js
// app.js — import
const { add, multiply, PI } = require("./math");

console.log(add(2, 3));       // 5
console.log(multiply(4, 5));  // 20
console.log(PI);              // 3.14159
```

### Importing built-in modules:
```js
const fs = require("fs");
const path = require("path");
const http = require("http");
```

### Importing npm packages:
```js
const express = require("express");
const mongoose = require("mongoose");
```

---

## ES Modules (ESM) — The Modern Way

ES Modules use `import`/`export` syntax — the standard in modern JavaScript.

To use ESM in Node.js: add `"type": "module"` in `package.json`, or use `.mjs` file extension.

```js
// math.mjs — named exports
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }
export const PI = 3.14159;

// Default export (one per file)
export default function subtract(a, b) { return a - b; }
```

```js
// app.mjs — imports
import subtract, { add, multiply, PI } from "./math.mjs";

console.log(add(2, 3));         // 5
console.log(subtract(10, 4));   // 6
```

### Dynamic import (lazy loading)
```js
async function loadModule() {
  const { add } = await import("./math.mjs");
  console.log(add(2, 3));
}
```

## CJS vs ESM Comparison

| Feature | CommonJS | ES Modules |
|---------|---------|------------|
| Syntax | `require()`, `module.exports` | `import`, `export` |
| Loading | Synchronous | Asynchronous |
| File extension | `.js` | `.mjs` or `.js` (with `"type": "module"`) |
| `__dirname` available | ✅ Yes | ❌ No (use `import.meta.url`) |
| Default in Node | ✅ Yes | 🟡 Opt-in |
| Tree shaking | ❌ No | ✅ Yes |

### `__dirname` equivalent in ESM
```js
import { fileURLToPath } from "url";
import { dirname } from "path";

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
console.log(__dirname); // works like CommonJS __dirname
```

## Interview Questions

- **Q: What is CommonJS?**
  A: Node.js's built-in module system using `require()` to import and `module.exports` to export. Synchronous and the default in Node.js.

- **Q: What is the difference between `module.exports` and `exports`?**
  A: `module.exports` is the actual exported object. `exports` is just a reference to it. Reassigning `exports = {}` breaks the reference — always use `module.exports` for full object exports.

- **Q: How do you enable ES Modules in Node.js?**
  A: Add `"type": "module"` in `package.json`, or use `.mjs` file extension.

- **Q: What is the key difference between CJS and ESM loading?**
  A: CJS loads modules synchronously (blocking). ESM loads asynchronously, enabling better performance and tree shaking.

- **Q: What is tree shaking?**
  A: A build optimization that removes unused exported code. ESM supports it; CJS does not.

- **Q: How do you get `__dirname` in ES Modules?**
  A: Use `fileURLToPath(import.meta.url)` and `dirname()` from the `path` module — `__dirname` is not available natively in ESM.
