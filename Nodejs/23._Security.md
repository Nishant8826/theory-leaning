# Rate Limiting & Security Basics

**Rate limiting** = Restricting how many requests a user/IP can make in a given time window.

**Security** = Protecting your API from common attacks and vulnerabilities.

This prevents abuse, brute-force attacks, and server overload.

---

## Real-World Example
An attacker tries to brute-force your `/login` endpoint — sending 10,000 password attempts per minute. With rate limiting, after 5 failed attempts in a window, they're blocked for 15 minutes.

## Rate Limiting with `express-rate-limit`

### Basic Rate Limit (all routes)
```js
const rateLimit = require("express-rate-limit");

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100,                  // max 100 requests per 15 min per IP
  standardHeaders: true,     // send rate limit info in headers
  legacyHeaders: false,
  message: {
    error: "Too many requests. Please wait 15 minutes and try again.",
  },
});

app.use("/api", limiter); // apply to all /api routes
```

### Stricter Limit for Login (Brute-force protection)
```js
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 5,                    // only 5 attempts per 15 min
  message: {
    error: "Too many login attempts. Please try again after 15 minutes.",
  },
});

app.post("/api/auth/login", loginLimiter, loginController);
```

## Security Headers with `helmet`

**Helmet** sets various HTTP headers to protect against common web attacks.

```js
const helmet = require("helmet");

app.use(helmet()); // applies all recommended security headers

// What helmet sets:
// - X-XSS-Protection
// - X-Content-Type-Options: nosniff
// - Strict-Transport-Security (HSTS)
// - X-Frame-Options: DENY (prevent clickjacking)
// - Content-Security-Policy
// - And many more...
```

## CORS — Cross-Origin Resource Sharing

Controls which domains can access your API.

```js
const cors = require("cors");

// Allow all origins (good for public APIs)
app.use(cors());

// Allow specific origin only (recommended for production)
app.use(cors({
  origin: "https://yourfrontend.com",
  methods: ["GET", "POST", "PUT", "DELETE"],
  allowedHeaders: ["Content-Type", "Authorization"],
  credentials: true, // allow cookies to be sent
}));

// Dynamic origin
app.use(cors({
  origin: (origin, callback) => {
    const allowedOrigins = ["https://app.com", "https://admin.app.com"];
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error("Not allowed by CORS"));
    }
  },
}));
```

## Input Sanitization — Prevent NoSQL Injection

```bash
npm install express-mongo-sanitize xss-clean
```

```js
const mongoSanitize = require("express-mongo-sanitize");
const xss = require("xss-clean");

// Remove $ and . from user input (prevents MongoDB injection)
app.use(mongoSanitize());

// Sanitize user input to prevent XSS (script injection)
app.use(xss());
```

## Other Security Best Practices

### Limit Request Body Size
```js
app.use(express.json({ limit: "10kb" })); // reject large payloads
```

### Prevent Parameter Pollution
```bash
npm install hpp
```
```js
const hpp = require("hpp");
app.use(hpp()); // prevents: ?sort=asc&sort=desc (pollution)
```

### Security Checklist
```
✅ Use HTTPS (not HTTP) in production
✅ Use helmet() for security headers
✅ Rate limit sensitive endpoints (login, register, forgot-password)
✅ Validate and sanitize all user input
✅ Hash passwords with bcrypt (never store plain text)
✅ Use short JWT expiry (15m access + 7d refresh)
✅ Keep dependencies updated (npm audit fix)
✅ Use environment variables — never hardcode secrets
✅ Log and monitor suspicious activity
✅ Use parameterized queries or ORMs to prevent SQL injection
```

## Complete Security Middleware Setup
```js
const express = require("express");
const helmet = require("helmet");
const cors = require("cors");
const rateLimit = require("express-rate-limit");
const mongoSanitize = require("express-mongo-sanitize");
const xss = require("xss-clean");
const hpp = require("hpp");

const app = express();

// Security headers
app.use(helmet());

// CORS
app.use(cors({ origin: process.env.FRONTEND_URL, credentials: true }));

// Rate limiting
app.use("/api", rateLimit({ windowMs: 15 * 60 * 1000, max: 100 }));

// Body parsing with size limit
app.use(express.json({ limit: "10kb" }));

// Sanitization
app.use(mongoSanitize()); // NoSQL injection prevention
app.use(xss());           // XSS prevention
app.use(hpp());           // Parameter pollution prevention
```

## Interview Questions

- **Q: What is rate limiting and why is it important?**
  A: Limiting how many requests an IP/user can make in a time window. Prevents brute-force attacks, DDoS, and API abuse.

- **Q: What is `helmet` used for?**
  A: A middleware that sets secure HTTP headers to protect against common attacks like XSS, clickjacking, MIME sniffing, and more.

- **Q: What is CORS and when do you need it?**
  A: Cross-Origin Resource Sharing — a browser mechanism to control which domains can access your API. Needed when frontend and backend are on different domains.

- **Q: What is NoSQL injection?**
  A: An attack where malicious payload (like `{ "$gt": "" }`) is injected into MongoDB queries to bypass filters. Prevented by sanitizing user input.

- **Q: What is XSS and how do you prevent it in Node.js?**
  A: Cross-Site Scripting — injecting malicious scripts into web pages. Prevented by using `xss-clean` middleware to sanitize user input and setting CSP headers.

- **Q: How do you store secret keys securely in a Node.js app?**
  A: Use environment variables (`.env` + `dotenv`) — never hardcode secrets in source code. In production, use platform-managed secrets (AWS Secrets Manager, etc.).
