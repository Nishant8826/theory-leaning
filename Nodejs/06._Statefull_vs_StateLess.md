# Stateful vs Stateless

**State** = Data about a user's session (who they are, what they're doing).

- **Stateful** = The server **remembers** each user between requests.
- **Stateless** = The server **forgets** everything after each request — the client must send its identity with every request.

---

## Real-World Example

**Stateful (Session-based auth):**
- You log in → server creates a session → browser receives a `sessionId` cookie.
- Every request sends the cookie → server looks up the session in memory/DB.

**Stateless (JWT-based auth):**
- You log in → server creates a JWT token.
- Every request sends the token in headers → server verifies the token (no lookup needed).

---

## Stateful System

The server stores session data (e.g., in memory or a session store like Redis).

### How it works:
```
1. User logs in
2. Server creates a session: { sessionId: "abc123", userId: 42 }
3. Server stores session in memory (or Redis)
4. Sends cookie: sessionId=abc123 to browser
5. Next request → browser sends cookie → server finds session → knows who you are
```

### Code Example (Express + express-session)
```js
const session = require("express-session");

app.use(session({
  secret: "mySecret",
  resave: false,
  saveUninitialized: false,
  cookie: { secure: false, maxAge: 1000 * 60 * 60 }, // 1 hour
}));

app.post("/login", (req, res) => {
  // Authenticate user...
  req.session.userId = 42;        // store in session
  req.session.username = "Nishant";
  res.json({ message: "Logged in!" });
});

app.get("/profile", (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: "Not logged in" });
  }
  res.json({ user: req.session.username });
});
```

### Pros & Cons of Stateful
| Pros | Cons |
|------|------|
| ✅ Easier to invalidate sessions (just delete from server) | ❌ Server must store session data |
| ✅ More control over user sessions | ❌ Hard to scale horizontally |
| ✅ Immediate logout support | ❌ If server crashes, sessions are lost |

---

## Stateless System

The server doesn't store anything — all user data is in the **token sent by the client**.

### How it works:
```
1. User logs in
2. Server creates a JWT token: { userId: 42, role: "admin" }
3. Signs it with a secret key → compact string
4. Sends token to client
5. Client stores token (localStorage or cookie)
6. Every request → client sends: Authorization: Bearer <token>
7. Server verifies token signature — no DB lookup needed
```

### Code Example (Express + JWT)
```js
const jwt = require("jsonwebtoken");
const JWT_SECRET = process.env.JWT_SECRET;

app.post("/login", (req, res) => {
  // Authenticate user (check DB)...
  const token = jwt.sign(
    { userId: 42, role: "admin" },
    JWT_SECRET,
    { expiresIn: "1h" }
  );
  res.json({ token });
});

app.get("/profile", (req, res) => {
  const authHeader = req.headers.authorization;
  if (!authHeader) return res.status(401).json({ error: "No token" });

  const token = authHeader.split(" ")[1]; // "Bearer <token>"
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    res.json({ userId: decoded.userId, role: decoded.role });
  } catch {
    res.status(401).json({ error: "Invalid token" });
  }
});
```

### Pros & Cons of Stateless
| Pros | Cons |
|------|------|
| ✅ Server doesn't store any state | ❌ Can't easily invalidate a token before expiry |
| ✅ Highly scalable | ❌ Token payload visible (don't store sensitive data) |
| ✅ Works across multiple servers | ❌ Larger request size (token in every request) |

---

## Comparison Table

| Feature | Stateful (Session) | Stateless (JWT) |
|---------|-----------------|----------------|
| Server stores session | ✅ Yes (in memory/Redis/DB) | ❌ No |
| Scalability | 🔴 Harder (sticky sessions needed) | 🟢 Easy (any server can verify) |
| Logout support | ✅ Instant (delete session) | 🟡 Needs workaround (blacklist) |
| Performance | 🟡 DB/memory lookup needed | 🟢 Verify with crypto only |
| Best for | Traditional web apps | REST APIs, microservices, mobile apps |

## Interview Questions

- **Q: What is the difference between stateful and stateless authentication?**
  A: Stateful stores session data on the server (session-cookie). Stateless sends all user data in a self-contained token (JWT) — server stores nothing.

- **Q: What is a session?**
  A: A server-side record tied to a client via a cookie (sessionId). It stores user data (userId, role) for the duration of the session.

- **Q: What is a JWT?**
  A: JSON Web Token — a signed, base64-encoded string containing a payload (userId, role, expiry). The server verifies its cryptographic signature without storing anything.

- **Q: Why is stateless preferred in modern APIs?**
  A: It's horizontally scalable (any server can verify tokens), no shared storage needed, and works well with microservices and mobile clients.

- **Q: What is the biggest security concern with JWT?**
  A: Once issued, JWTs can't be revoked before expiry unless you maintain a blacklist. Always use short expiry times and refresh tokens.

- **Q: What is a refresh token?**
  A: A longer-lived token used to get a new access token when it expires — without requiring the user to log in again.
