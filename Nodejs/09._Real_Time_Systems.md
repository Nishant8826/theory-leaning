# Real-Time Systems

**Real-time systems** deliver data to clients **instantly** when it becomes available — with minimal delay (latency). Users don't have to refresh the page; updates appear automatically.

---

## Real-World Example
- 💬 **Chat apps** (WhatsApp, Slack) — messages appear instantly
- 📈 **Stock market dashboards** — prices update live
- 🎮 **Multiplayer games** — player positions sync in real time
- 🔔 **Push notifications** — you're notified without refreshing
- 📝 **Google Docs** — multiple users can type and see changes live

---

## Three Approaches to Real-Time

There are three main methods to build real-time systems, each with different trade-offs:

---

## 1. WebSockets

WebSockets provide a **full-duplex**, **persistent** connection between client and server.

### How WebSockets Work
- The connection starts as a normal **HTTP/HTTPS request**.
- The client sends a request to the server asking to **upgrade** the protocol from HTTP to WebSocket (`ws://` or `wss://`).
- If the server accepts, the connection is established.
- After this, both client and server can **send messages at any time**.
- The connection stays open until either side explicitly closes it.

### Key Characteristics
- **Protocol:** `ws://` (insecure) or `wss://` (secure, over HTTPS)
- **Communication type:** Full duplex (bi-directional)
- **Connection:** Persistent
- **Latency:** Very low

### Advantages
- Real-time, instant communication
- Efficient for frequent message exchange
- Reduced overhead vs. repeated HTTP requests
- Ideal for interactive apps

### Disadvantages
- Requires persistent connections (more server memory)
- Harder to scale horizontally without extra infrastructure
- If server crashes, active connections are lost

### Common Use Cases
- Chat applications
- Online multiplayer games
- Live trading platforms
- Collaborative tools (Google Docs-like apps)

### Scaling WebSockets
- Use **load balancers with sticky sessions**
- Use **message brokers** (Redis Pub/Sub, Kafka, RabbitMQ)
- Implement **reconnection logic** on the client

### Code Example
```js
// Server-side (Node.js)
const WebSocket = require("ws");
const wss = new WebSocket.Server({ port: 8080 });

wss.on("connection", (ws) => {
  ws.on("message", (message) => {
    ws.send(`Received: ${message}`); // echo back
  });
});

// Client-side (Browser)
const socket = new WebSocket("wss://example.com/socket");
socket.onopen = () => socket.send("Hello Server");
socket.onmessage = (event) => console.log("Message:", event.data);
socket.onclose = () => console.log("Connection closed");
```

## 2. Polling

Polling is the simplest real-time technique — works entirely over HTTP.

### How Polling Works
- The client sends an HTTP request to the server at a **fixed interval** (e.g., every 5 seconds).
- If data is available, the server sends it back.
- If no data, the server sends an empty/default response.
- The connection closes after every response.

### Key Characteristics
- **Protocol:** HTTP/HTTPS
- **Communication type:** Client → Server only
- **Connection:** Short-lived (stateless)
- **Latency:** Depends on polling interval

### Advantages
- Very simple to implement
- Works with all servers, browsers, and proxies
- Easy to scale using standard HTTP load balancers

### Disadvantages
- High network overhead — wasted requests
- Not truly real-time (delay = polling interval)
- Increased server load

### Variants of Polling

#### Short Polling
Client sends requests at fixed intervals — most common and least efficient.

#### Long Polling
Client sends a request. Server **holds it open** until new data arrives or timeout occurs. Once data is sent, client immediately reconnects.

### Common Use Cases
- Low-frequency notifications
- Apps behind strict firewalls
- Legacy systems where WebSockets aren't supported

### Code Example
```js
// Client (short polling — every 5 seconds)
setInterval(async () => {
  const response = await fetch("/api/data");
  const data = await response.json();
  console.log("Latest data:", data);
}, 5000);

// Server
app.get("/api/data", (req, res) => {
  res.json({ message: "Latest data", time: new Date() });
});
```

## 3. SSE — Server-Sent Events

SSE allows the server to **push updates** to the client over a single HTTP connection — one direction only (server → client).

### How SSE Works
- Client sends one HTTP request to an SSE endpoint.
- Server keeps the connection open.
- Server **pushes events** whenever new data arrives.
- Built-in browser auto-reconnect support.

### Key Characteristics
- **Protocol:** HTTP/HTTPS
- **Communication type:** One-way (Server → Client only)
- **Connection:** Persistent
- **Data format:** Text-based (event stream)

### Important Limitation
The client **cannot send messages** over SSE — it must use a separate HTTP request for that.

### Advantages
- Simpler than WebSockets
- Automatic reconnection supported in browsers
- Works well with HTTP/2
- Lightweight for server-to-client streaming

### Disadvantages
- One-way communication only
- Not for bi-directional needs

### Common Use Cases
- Live news/price feeds
- Real-time notifications
- Live logs and dashboards
- AI response streaming (like ChatGPT)

### Code Example
```js
// Server (Express)
app.get("/events", (req, res) => {
  res.setHeader("Content-Type", "text/event-stream");
  res.setHeader("Cache-Control", "no-cache");
  res.setHeader("Connection", "keep-alive");

  const intervalId = setInterval(() => {
    res.write(`data: ${JSON.stringify({ time: new Date() })}\n\n`);
  }, 3000);

  req.on("close", () => {
    clearInterval(intervalId); // cleanup when client disconnects
    res.end();
  });
});

// Client
const eventSource = new EventSource("/events");
eventSource.onmessage = (event) => {
  console.log("New event:", JSON.parse(event.data));
};
```

## Comparison Table

| Feature | WebSockets | Polling | SSE |
|---------|-----------|---------|-----|
| Protocol | WS / WSS | HTTP | HTTP |
| Direction | Bi-directional | Client → Server | Server → Client |
| Connection | Persistent | Short-lived | Persistent |
| Latency | Very Low | High (interval-based) | Low |
| Complexity | High | Low | Medium |
| Scalability | Harder | Easy | Moderate |
| Use case | Chat, games, collaboration | Notifications, legacy | Feeds, logs, notifications |

---

## Choosing the Right Approach

- Use **WebSockets** when you need **bi-directional, real-time interaction** (chat, games).
- Use **Polling** when simplicity and reliability matter more than instant updates.
- Use **SSE** when you only need **server-to-client** streaming (dashboards, AI chat streams).

---

## Fault Tolerance

### WebSockets
- Implement reconnection logic on the client
- Use heartbeat/ping-pong messages to detect dead connections
- Store session data externally (Redis)

### Polling
- Naturally fault-tolerant — stateless requests
- Easy recovery after server restart

### SSE
- Built-in browser auto-reconnect
- Use `Last-Event-ID` header to resume from where you left off

## Interview Questions

- **Q: What is a real-time system?**
  A: A system that delivers updates to clients with minimal delay — data is pushed or pulled as soon as it's available, without user-initiated refresh.

- **Q: What are the three main approaches to build real-time apps?**
  A: WebSockets (full-duplex, persistent), Polling (client asks repeatedly), and SSE (server pushes one-way over HTTP).

- **Q: What is the main advantage of WebSockets over polling?**
  A: WebSockets maintain a persistent connection — no repeated requests. Server can push data instantly. Polling wastes resources with unnecessary requests.

- **Q: What is SSE and when would you choose it?**
  A: Server-Sent Events — server pushes updates to client over HTTP. Use it when you only need server-to-client streaming (notifications, score updates, AI responses).

- **Q: Why is WebSocket hard to scale?**
  A: Each WebSocket holds a stateful connection. Multiple servers don't share connections natively — you need sticky sessions or a message broker (Redis Pub/Sub).

- **Q: What is a message broker and why is it used with WebSockets?**
  A: A message broker (like Redis Pub/Sub or Kafka) allows multiple server instances to communicate and broadcast messages to all WebSocket clients regardless of which server they're connected to.

- **Q: What is the difference between short polling and long polling?**
  A: Short polling: client makes requests at fixed intervals (e.g., every 5s). Long polling: client sends a request, server holds it until data is ready, then client immediately reconnects.
