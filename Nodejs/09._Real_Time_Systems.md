# Real Time System
Real-time systems are systems that are able to deliver data to clients as soon as it is available, with minimal latency. They are widely used in chat applications, notifications, live dashboards, stock prices, multiplayer games, collaboration tools, and monitoring systems.

There are several methods to create real-time systems. The most commonly used approaches are **WebSockets**, **Polling (and its variants)**, and **Server-Sent Events (SSE)**.

---

## WebSockets
WebSockets provide a **full-duplex**, **persistent** connection between client and server.

### How WebSockets Work
- The connection starts as a normal **HTTP/HTTPS request**.
- The client sends a request to the server asking to **upgrade** the protocol from HTTP to WebSocket (`ws://` or `wss://`).
- If the server accepts the upgrade request, the connection is established.
- After this, both client and server can **send messages at any time**.
- The connection stays open until either the client or server explicitly closes it.

### Key Characteristics
- **Protocol**: `ws://` (insecure) or `wss://` (secure, over HTTPS)
- **Communication type**: Full duplex (bi-directional)
- **Connection**: Persistent
- **Latency**: Very low

### Advantages
- Real-time, instant communication.
- Efficient for frequent message exchange.
- Reduced overhead compared to repeated HTTP requests.
- Ideal for interactive applications.

### Disadvantages
- Requires persistent connections (more server memory).
- Harder to scale horizontally without extra infrastructure.
- If a server crashes, active connections are lost.
- Needs additional handling for reconnection and fault tolerance.

### Common Use Cases
- Chat applications
- Online multiplayer games
- Live trading platforms
- Collaborative tools (Google Docs-like apps)
- Real-time dashboards

### Scaling WebSockets
To scale WebSockets:
- Use **load balancers with sticky sessions**.
- Use **message brokers** (Redis, Kafka, RabbitMQ).
- Use **WebSocket gateways**.
- Implement **reconnection logic** on the client.

---

## Polling
Polling is the simplest real-time technique and works entirely over **HTTP/HTTPS**.

### How Polling Works
- The client sends an HTTP request to the server at a fixed interval (e.g., every 5 seconds).
- The server checks for new data.
- If data is available, the server sends it back.
- If no data is available, the server sends an empty or default response.
- The connection is closed after every response.

### Key Characteristics
- **Protocol**: HTTP/HTTPS
- **Communication type**: Client → Server only
- **Connection**: Short-lived (stateless)
- **Latency**: Depends on polling interval

### Advantages
- Very simple to implement.
- Works with all servers, browsers, and proxies.
- Easy to scale using traditional HTTP load balancers.
- No persistent connection required.

### Disadvantages
- High network overhead.
- Wasted requests when no data changes.
- Increased server load.
- Not truly real-time.

### Variants of Polling

#### Short Polling
- Client sends requests at fixed intervals.
- Most common and least efficient.

#### Long Polling
- Client sends a request.
- Server **holds the request open** until new data is available or a timeout occurs.
- Once the response is sent, the client immediately sends another request.
- Reduces unnecessary requests compared to short polling.

### Why Polling Is Still Used
- More reliable in unstable environments.
- No risk of losing stateful connections.
- Easier recovery if a server crashes.
- Better suited for very large-scale applications with infrequent updates.

### Common Use Cases
- Notifications with low update frequency
- Applications behind strict firewalls
- Legacy systems

---

## SSE (Server-Sent Events)
Server-Sent Events allow servers to push updates to the client over HTTP.

### How SSE Works
- The client sends a single HTTP request to the server.
- The server keeps the connection open.
- The server sends updates (events) whenever new data is available.
- The connection remains open until the server closes it or an error occurs.

### Key Characteristics
- **Protocol**: HTTP/HTTPS
- **Communication type**: One-way (Server → Client)
- **Connection**: Persistent
- **Data format**: Text-based (event stream)

### Important Limitation
- The client **cannot send messages** to the server over the same connection.
- If client needs to send data, it must use a separate HTTP request.

### Advantages
- Simpler than WebSockets.
- Automatic reconnection support in browsers.
- Works well with HTTP/2.
- Lightweight and efficient for server-to-client updates.

### Disadvantages
- One-way communication only.
- Not supported in very old browsers.
- Limited to text-based data (JSON is commonly used).

### Common Use Cases
- Live news feeds
- Stock price updates
- Notifications
- Live logs and monitoring

---

## Comparison Table

| Feature | WebSockets | Polling | SSE |
|------|-----------|---------|-----|
| Protocol | WS / WSS | HTTP / HTTPS | HTTP / HTTPS |
| Direction | Bi-directional | Client → Server | Server → Client |
| Connection | Persistent | Short-lived | Persistent |
| Latency | Very Low | High (interval-based) | Low |
| Complexity | High | Low | Medium |
| Scalability | Harder | Easy | Moderate |
| Browser Support | Excellent | Excellent | Good |

---

## Choosing the Right Approach

- Use **WebSockets** when you need **bi-directional, real-time interaction**.
- Use **Polling** when simplicity, reliability, and scalability matter more than instant updates.
- Use **SSE** when you only need **server-to-client streaming updates**.

---

## Summary
- WebSockets offer the most powerful real-time capabilities but require careful scaling.
- Polling is the easiest to implement but least efficient.
- SSE provides a balanced solution for server-driven updates.

Each method has its own trade-offs, and the best choice depends on the application requirements.

## Basic Code Examples

### WebSockets Example (JavaScript)
```js
// Client-side
const socket = new WebSocket('wss://example.com/socket');

socket.onopen = () => {
  socket.send('Hello Server');
};

socket.onmessage = (event) => {
  console.log('Message from server:', event.data);
};

socket.onclose = () => {
  console.log('Connection closed');
};
```

```js
// Server-side (Node.js with ws library)
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', ws => {
  ws.on('message', message => {
    ws.send(`Received: ${message}`);
  });
});
```

---

### Polling Example (JavaScript)
```js
setInterval(async () => {
  const response = await fetch('/api/data');
  const data = await response.json();
  console.log(data);
}, 5000);
```

```js
// Server-side (Express.js)
app.get('/api/data', (req, res) => {
  res.json({ message: 'Latest data' });
});
```

---

### SSE Example (JavaScript)
```js
// Client-side
const eventSource = new EventSource('/events');

eventSource.onmessage = (event) => {
  console.log('New event:', event.data);
};
```

```js
// Server-side (Express.js)
app.get('/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');

  setInterval(() => {
    res.write(`data: ${JSON.stringify({ time: new Date() })}\n\n`);
  }, 3000);
});
```

## Fault Tolerance and Reliability

### WebSockets
- Requires reconnection logic on client
- Use heartbeat/ping-pong messages
- Store session data externally (Redis, DB)

### Polling
- Naturally fault-tolerant
- Stateless requests
- Easy recovery after server restart

### SSE
- Built-in auto-reconnect
- Event IDs help resume streams

---

## Performance Considerations

- WebSockets are best for high-frequency, bi-directional data
- Polling suits low-frequency updates
- SSE is optimal for continuous server updates

---

## Summary
- WebSockets: powerful but complex
- Polling: simple and reliable
- SSE: efficient one-way streaming

Choosing correctly improves scalability, performance, and user experience.
