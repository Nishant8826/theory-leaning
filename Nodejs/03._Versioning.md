# Versioning (Semantic Versioning / SemVer)

## Simple Explanation
Every npm package has a version number in the format: **Major.Minor.Patch**

This standard is called **Semantic Versioning (SemVer)** and tells developers what kind of changes were made in each release.

Think of it like a software passport:
- **Major** = Big change — things might break (like moving to a new country).
- **Minor** = New feature — safe to update (like getting a new stamp).
- **Patch** = Bug fix — always update (like fixing a typo in your passport).

---

## Real-World Example
You're using `express@4.18.2`:
- `4` = Major (breaking API changes from v3)
- `18` = Minor (new features added to v4)
- `2` = Patch (bug fixes in v4.18)

---

## Version Parts Explained

### Major (`X.0.0`)
Incremented when there are **breaking changes** — the new version may NOT be compatible with the old one.
```
express 3.x → express 4.x: had major breaking changes in routing API
```

### Minor (`0.X.0`)
Incremented when **new features** are added that are **backward-compatible** (old code still works).
```
express 4.17.0 → 4.18.0: added new methods, old code still works
```

### Patch (`0.0.X`)
Incremented for **bug fixes** only — no new features, no breaking changes.
```
express 4.18.1 → 4.18.2: fixed a small security or logic bug
```

## Version Range Symbols in `package.json`

```json
{
  "dependencies": {
    "express": "^4.18.2",
    "mongoose": "~7.0.0",
    "lodash": "4.17.21"
  }
}
```

| Symbol | Meaning | Example |
|--------|---------|---------|
| `^` (caret) | Update Minor and Patch, lock Major | `^4.18.2` → allows `4.19.0`, `4.18.3`, NOT `5.0.0` |
| `~` (tilde) | Update Patch only, lock Major and Minor | `~4.18.2` → allows `4.18.3`, NOT `4.19.0` |
| No symbol | Exact version only | `4.18.2` → only installs exactly `4.18.2` |
| `*` | Any version | Dangerous — avoid in production |

## Code Example

### Sample `package.json`
```json
{
  "name": "my-backend",
  "version": "1.0.0",
  "description": "My Node.js backend API",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "dev": "nodemon index.js",
    "test": "jest"
  },
  "dependencies": {
    "express": "^4.18.2",
    "mongoose": "^7.0.0",
    "dotenv": "^16.0.0"
  },
  "devDependencies": {
    "nodemon": "^3.0.0",
    "jest": "^29.0.0"
  }
}
```

### Key npm Commands
```bash
npm install express               # install latest compatible version
npm install express@4.18.2        # install exact version
npm update                        # update all packages (respects ^ and ~)
npm outdated                      # list packages with newer versions
npm audit                         # check for security vulnerabilities
npm audit fix                     # auto-fix vulnerabilities
```

## `dependencies` vs `devDependencies`

| | `dependencies` | `devDependencies` |
|--|--------------|-----------------|
| What | Needed in production | Only needed during development |
| Examples | `express`, `mongoose`, `dotenv` | `nodemon`, `jest`, `eslint` |
| Install command | `npm install express` | `npm install nodemon --save-dev` |
| Production deploy | ✅ Included | ❌ Not included |

## `package-lock.json`

A file automatically generated by npm that **locks the exact versions** of all packages — including nested dependencies.

- Ensures everyone on the team installs the **exact same versions**.
- Always commit `package-lock.json` to version control.
- Never manually edit it.

## Interview Questions

- **Q: What is Semantic Versioning (SemVer)?**
  A: A versioning standard using `Major.Minor.Patch`. Major = breaking changes, Minor = new features (backward-compatible), Patch = bug fixes.

- **Q: What does `^` (caret) mean in `package.json`?**
  A: It allows updates to Minor and Patch versions but locks the Major. E.g., `^4.18.2` allows `4.19.0` but not `5.0.0`.

- **Q: What does `~` (tilde) mean in `package.json`?**
  A: It allows only Patch updates, locking Major and Minor. E.g., `~4.18.2` allows `4.18.3` but not `4.19.0`.

- **Q: What is `package-lock.json`?**
  A: A file that locks the exact resolved versions of all installed packages (including transitive dependencies) to ensure consistent installs across environments.

- **Q: What is the difference between `dependencies` and `devDependencies`?**
  A: `dependencies` are needed in production. `devDependencies` are only needed in development (e.g., testing tools, build tools, nodemon).

- **Q: What does `npm audit` do?**
  A: It checks your installed packages against a vulnerability database and reports any known security issues.
