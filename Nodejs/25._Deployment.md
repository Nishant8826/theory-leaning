# Deployment Basics

**Deployment** = Making your Node.js app accessible to the world by running it on a remote server (not just your laptop).

When your app is on `localhost:3000`, only you can access it. Deployment puts it on a real server with a public URL so anyone can use it.

---

## Real-World Example
You built a REST API for a todo app. To let mobile/web clients use it:
1. Push code to GitHub
2. Connect to a hosting platform (Railway, Render, Heroku)
3. Set environment variables
4. Deploy → get a public URL like `https://myapi.railway.app`

---

## Deployment Checklist (Before Deploying)

```
✅ Set NODE_ENV=production in environment variables
✅ All secrets in .env (never hardcoded)
✅ .gitignore includes node_modules/ and .env
✅ package.json has a "start" script
✅ Error handling for all routes
✅ Database is hosted (MongoDB Atlas, not localhost)
✅ CORS configured for production frontend URL
✅ Rate limiting enabled
✅ Security middleware (helmet) applied
```

## Step 1: Prepare Your App

### `package.json` scripts
```json
{
  "scripts": {
    "start": "node index.js",              // production
    "dev": "nodemon index.js",             // development
    "build": "echo No build step needed"
  },
  "engines": {
    "node": ">=18.0.0"                     // specify Node.js version
  }
}
```

### Environment-aware code
```js
const isProduction = process.env.NODE_ENV === "production";

if (!isProduction) {
  // Dev-only: detailed error messages
  app.use((err, req, res, next) => {
    res.status(err.statusCode || 500).json({
      error: err.message,
      stack: err.stack, // show stack in dev
    });
  });
} else {
  // Production: minimal error info
  app.use((err, req, res, next) => {
    res.status(err.statusCode || 500).json({
      error: err.isOperational ? err.message : "Internal Server Error",
    });
  });
}
```

## Step 2: Host Your Database (MongoDB Atlas)

1. Go to [cloud.mongodb.com](https://cloud.mongodb.com)
2. Create a free cluster
3. Add your IP to the IP Whitelist (or use `0.0.0.0/0` for all)
4. Create a database user
5. Get your connection string:
   ```
   mongodb+srv://user:pass@cluster0.xxxxx.mongodb.net/myapp
   ```
6. Add to your production environment variables as `MONGODB_URI`

## Step 3: Deploy to Hosting Platforms

### Option A: Railway (Recommended for beginners)
```bash
# Install Railway CLI
npm install -g @railway/cli

# Login
railway login

# Initialize project
railway init

# Deploy
railway up
```

Or connect GitHub repo in Railway dashboard → auto-deploy on push.

### Option B: Render
1. Go to [render.com](https://render.com) → New → Web Service
2. Connect your GitHub repo
3. Set:
   - Build command: `npm install`
   - Start command: `npm start`
4. Add environment variables
5. Deploy

### Option C: Heroku
```bash
# Install Heroku CLI, then:
heroku create my-app-name
heroku config:set MONGODB_URI="..." JWT_SECRET="..."
git push heroku main
```

### Option D: VPS (Ubuntu Server — Manual)
```bash
# On your VPS:
# Install Node.js
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# Clone your repo
git clone https://github.com/yourname/yourrepo.git
cd yourrepo
npm install

# Install PM2 (process manager)
npm install -g pm2
pm2 start index.js --name "my-api"
pm2 save          # persist across reboots
pm2 startup       # auto-start on server restart
```

## PM2 — Process Manager (Must Know)

PM2 keeps your Node.js app running even if it crashes or the server restarts.

```bash
npm install -g pm2

pm2 start index.js --name "my-api"   # start app
pm2 list                              # list all apps
pm2 logs my-api                       # view live logs
pm2 restart my-api                    # restart
pm2 stop my-api                       # stop
pm2 delete my-api                     # remove from PM2

# Cluster mode — use all CPU cores
pm2 start index.js -i max --name "my-api"
```

## Nginx as Reverse Proxy (Advanced)

On a VPS, use Nginx to forward requests to your Node.js app.

```nginx
# /etc/nginx/sites-available/my-api
server {
    listen 80;
    server_name api.myapp.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}
```

## Environment Variables in Production

Each platform has its own way to set them:
- **Railway / Render / Heroku** → Dashboard UI → Environment Variables section
- **AWS EC2 / VPS** → PM2 ecosystems file or systemd service environment
- **Docker** → `docker run -e KEY=VALUE` or `docker-compose.yml`

## CI/CD — Automatic Deployment

Set up GitHub Actions to auto-deploy when you push to `main`:

```yaml
# .github/workflows/deploy.yml
name: Deploy to Railway

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Deploy to Railway
        run: npx @railway/cli up
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
```

## Interview Questions

- **Q: What does `NODE_ENV=production` do?**
  A: Tells your app it's running in production mode. Libraries optimize behavior (e.g., Express disables dev warnings). Your code can branch for prod-specific behavior.

- **Q: What is PM2?**
  A: A production process manager for Node.js. It keeps apps running after crashes, restarts on reboot, manages logs, and supports clustering.

- **Q: What is the difference between Railway, Render, and a VPS?**
  A: Railway/Render = managed PaaS — just push code, they handle infra. VPS = you manage the server (Nginx, PM2, firewall) — more control, more work.

- **Q: What is Nginx and why use it with Node.js?**
  A: Nginx is a web server/reverse proxy. In front of Node.js, it handles SSL termination, load balancing, and static file serving — offloading work from Node.js.

- **Q: Why shouldn't you run Node.js as root on a VPS?**
  A: Security risk — if your app is compromised, root access gives attackers full server control. Run Node.js as a non-privileged user.

- **Q: What is a reverse proxy?**
  A: A server (like Nginx) that sits in front of your app, forwards client requests to it, and returns responses. Handles SSL, load balancing, and caching.

- **Q: What is CI/CD?**
  A: Continuous Integration/Continuous Deployment — automated pipelines that run tests and deploy your app whenever you push code (using tools like GitHub Actions).
