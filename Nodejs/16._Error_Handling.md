# Error Handling in Node.js

## Simple Explanation
**Error handling** = Catching things that go wrong and responding gracefully — instead of crashing the server or leaving requests hanging forever.

A server that crashes on every error is like a shop that closes every time a customer asks for something that's out of stock. Instead, tell them nicely: "Sorry, that item isn't available."

---

## Real-World Example
- User sends invalid JSON → send `400 Bad Request`
- User requests a non-existent product → send `404 Not Found`
- Database connection fails → send `500 Internal Server Error`
- User tries to access admin panel without permission → send `403 Forbidden`

---

## Types of Errors in Node.js

| Type | Example |
|------|---------|
| Operational Errors | 404 not found, validation failure, DB timeout |
| Programmer Errors | `null.property`, calling non-function, logic bugs |
| Async Errors | Unhandled promise rejections, uncaught exceptions |

## Code Examples

### Basic Express Error Handling
```js
const express = require("express");
const app = express();
app.use(express.json());

// A route that might throw an error
app.get("/user/:id", async (req, res, next) => {
  try {
    const user = await getUserFromDB(req.params.id);

    if (!user) {
      return res.status(404).json({ error: "User not found" });
    }

    res.json(user);
  } catch (err) {
    next(err); // pass error to the error-handling middleware
  }
});

// ✅ Error-handling middleware — ALWAYS 4 params (err, req, res, next)
// Must be placed AFTER all routes
app.use((err, req, res, next) => {
  console.error("Error:", err.stack);
  res.status(err.status || 500).json({
    success: false,
    error: err.message || "Internal Server Error",
  });
});

app.listen(3000);
```

### Custom Error Class
```js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.status = statusCode >= 400 && statusCode < 500 ? "fail" : "error";
    this.isOperational = true;
    Error.captureStackTrace(this, this.constructor);
  }
}

// Using custom error
app.get("/products/:id", async (req, res, next) => {
  try {
    const product = await Product.findById(req.params.id);
    if (!product) {
      throw new AppError("Product not found", 404); // custom error
    }
    res.json(product);
  } catch (err) {
    next(err);
  }
});

// Error handler reads the custom properties
app.use((err, req, res, next) => {
  if (err.isOperational) {
    return res.status(err.statusCode).json({
      success: false,
      error: err.message,
    });
  }
  // Unknown/programmer error — don't leak details
  console.error("UNHANDLED ERROR:", err);
  res.status(500).json({ success: false, error: "Something went wrong" });
});
```

### Async Error Wrapper (Avoid Repetitive try/catch)
```js
// Utility wrapper
const catchAsync = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next);
};

// Now routes are clean — no try/catch needed!
app.get("/users/:id", catchAsync(async (req, res) => {
  const user = await User.findById(req.params.id);
  if (!user) throw new AppError("User not found", 404);
  res.json(user);
}));
```

### Handling Unhandled Promise Rejections & Uncaught Exceptions
```js
// Catches unhandled promise rejections globally
process.on("unhandledRejection", (reason, promise) => {
  console.error("Unhandled Rejection:", reason);
  server.close(() => process.exit(1)); // graceful shutdown
});

// Catches synchronous errors not caught anywhere
process.on("uncaughtException", (err) => {
  console.error("Uncaught Exception:", err.message);
  process.exit(1); // mandatory restart
});
```

### Handling Mongoose/Database Errors
```js
app.use((err, req, res, next) => {
  // MongoDB duplicate key error (e.g., email already exists)
  if (err.code === 11000) {
    const field = Object.keys(err.keyValue)[0];
    return res.status(409).json({
      error: `${field} already exists`,
    });
  }

  // Mongoose validation error
  if (err.name === "ValidationError") {
    const messages = Object.values(err.errors).map((e) => e.message);
    return res.status(422).json({ errors: messages });
  }

  // JWT errors
  if (err.name === "JsonWebTokenError") {
    return res.status(401).json({ error: "Invalid token" });
  }

  if (err.name === "TokenExpiredError") {
    return res.status(401).json({ error: "Token expired, please log in again" });
  }

  res.status(500).json({ error: "Internal Server Error" });
});
```

## Interview Questions

- **Q: How do you handle errors in Express?**
  A: Use `try/catch` in async route handlers and call `next(err)` to pass to error-handling middleware — a function with `(err, req, res, next)` placed after all routes.

- **Q: What is the difference between operational and programmer errors?**
  A: Operational = expected failures (404, validation fail, DB timeout). Programmer = bugs (null reference, wrong types). Handle both differently — don't leak programmer error details to clients.

- **Q: What is a `catchAsync` wrapper?**
  A: A function that wraps async route handlers and passes rejected Promises to `next()` automatically — avoiding repetitive try/catch.

- **Q: What does `process.on("unhandledRejection")` do?**
  A: Catches Promise rejections that were never handled. Without it, Node.js (v15+) crashes the process automatically.

- **Q: What does `process.on("uncaughtException")` do?**
  A: Catches synchronous errors that bubble up to the process level. After handling, always restart the process — the state may be corrupted.

- **Q: Why should error-handling middleware be placed last in Express?**
  A: Express only calls it when `next(err)` is called. It must be after all routes so that any route can delegate errors to it.
