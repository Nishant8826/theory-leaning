# Async Patterns in Node.js

## Simple Explanation
Since most Node.js operations (reading files, querying databases, calling APIs) are **asynchronous**, you need a pattern to handle them properly.

Over time, three patterns evolved:
1. **Callbacks** — the original way (messy with nesting)
2. **Promises** — cleaner, chainable
3. **Async/Await** — looks like synchronous code, cleanest and most readable

---

## Real-World Example
Getting a user profile after login:
1. Verify the token
2. Find user in database
3. Fetch their posts
4. Send back the response

Each step depends on the previous one → async patterns control this flow.

---

## 1. Callbacks (Original Pattern)

A function passed as an argument to be called when async work completes.

```js
const fs = require("fs");

// Simple callback
fs.readFile("users.json", "utf8", function (err, data) {
  if (err) {
    console.error("Error:", err.message);
    return;
  }
  console.log("Data:", data);
});

// Callback hell — chaining 3 dependent async calls
getUser(userId, function (err, user) {
  if (err) return handleError(err);

  getPosts(user.id, function (err, posts) {
    if (err) return handleError(err);

    getComments(posts[0].id, function (err, comments) {
      if (err) return handleError(err);
      console.log(comments); // finally!
    });
  });
});
```

❌ Problems: nested, hard to read, hard to handle errors consistently.

---

## 2. Promises

An object representing a future value — can be chained with `.then()`.

```js
const fs = require("fs/promises");

// Reading a file with promises
fs.readFile("users.json", "utf8")
  .then((data) => {
    const users = JSON.parse(data);
    console.log(users);
  })
  .catch((err) => {
    console.error("Error:", err.message);
  });

// Chained async operations (clean!)
getUser(userId)
  .then((user) => getPosts(user.id))
  .then((posts) => getComments(posts[0].id))
  .then((comments) => console.log(comments))
  .catch((err) => console.error("Error:", err));

// Run multiple async tasks in parallel
Promise.all([getUser(1), getPosts(1), getStats()])
  .then(([user, posts, stats]) => {
    res.json({ user, posts, stats });
  })
  .catch(console.error);
```

---

## 3. Async/Await (Modern — Recommended)

Makes async code look synchronous. Built on top of Promises.

```js
const fs = require("fs/promises");

async function loadUsers() {
  try {
    const data = await fs.readFile("users.json", "utf8");
    const users = JSON.parse(data);
    console.log(users);
  } catch (err) {
    console.error("Failed:", err.message);
  }
}

loadUsers();

// Chaining async operations — clean and readable!
async function getUserData(userId) {
  try {
    const user = await getUser(userId);
    const posts = await getPosts(user.id);
    const comments = await getComments(posts[0].id);
    return { user, posts, comments };
  } catch (err) {
    throw new Error(`Failed to get user data: ${err.message}`);
  }
}
```

### Express route with async/await
```js
app.get("/users/:id", async (req, res) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) return res.status(404).json({ error: "User not found" });
    res.json(user);
  } catch (err) {
    res.status(500).json({ error: "Server error" });
  }
});
```

## Parallel vs Sequential

### Sequential (slower — one after another)
```js
const user = await getUser(id);         // wait...
const posts = await getPosts(user.id);  // then wait...
// Total time = time(user) + time(posts)
```

### Parallel (faster — simultaneously)
```js
const [user, posts] = await Promise.all([getUser(id), getPosts(id)]);
// Total time = max(time(user), time(posts))
```

## EventEmitter — Event-Driven Pattern

Node.js is built on events. Many built-in modules (streams, HTTP) use this pattern.

```js
const EventEmitter = require("events");
const emitter = new EventEmitter();

// Register a listener
emitter.on("user:login", (user) => {
  console.log(`${user.name} just logged in!`);
  // Send welcome email, update last login, etc.
});

// Emit the event
emitter.emit("user:login", { name: "Nishant", id: 42 });
```

## Comparison Table

| Feature | Callbacks | Promises | Async/Await |
|---------|---------|---------|------------|
| Readability | ❌ Nested | 🟡 Flat chain | ✅ Synchronous-like |
| Error handling | ❌ Per callback | 🟡 `.catch()` | ✅ `try/catch` |
| Debugging | ❌ Hard | 🟡 Medium | ✅ Easy |
| Parallel ops | 🟡 Complex | ✅ `Promise.all()` | ✅ `Promise.all()` |
| Used in | Older code, `fs` | Modern APIs | Recommended (all async work) |

## Interview Questions

- **Q: What is the difference between callbacks, Promises, and async/await?**
  A: Callbacks = functions called when async work is done (can nest messily). Promises = chainable objects for future values. Async/await = syntactic sugar over Promises, looks synchronous.

- **Q: What is callback hell?**
  A: Deeply nested callbacks where each step depends on the previous one — hard to read, debug, and maintain. Solved by Promises or async/await.

- **Q: What does `async` return?**
  A: An `async` function always returns a Promise, even if you return a plain value.

- **Q: How do you run multiple async operations at the same time?**
  A: Use `Promise.all([op1(), op2(), op3()])` — all operations start simultaneously, and it resolves when all finish.

- **Q: What happens if one Promise in `Promise.all()` rejects?**
  A: The entire `Promise.all()` rejects immediately with that error. Use `Promise.allSettled()` if you want all results regardless of failures.

- **Q: What is the EventEmitter in Node.js?**
  A: A built-in class for event-driven programming. Objects can emit named events and register listeners that react to those events.
