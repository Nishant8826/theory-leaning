# Environment Variables (dotenv)

## Simple Explanation
**Environment variables** are configuration values stored **outside your code** — in a special file or the system environment — so sensitive data (passwords, API keys, DB URLs) never gets hardcoded into your source code.

Think of it like a **locked safe inside your restaurant**:
- The safe (`.env` file) holds secret recipes (API keys, passwords).
- The kitchen staff (application code) access it when needed.
- Customers (public/GitHub) never see the contents.

---

## Real-World Example
Your app needs:
- A database connection URL (`mongodb://user:pass@host/db`)
- A JWT secret key
- An API port number
- A third-party API key

If you hardcode these, they end up in GitHub → security breach. Instead, keep them in `.env`.

---

## Setup

```bash
npm install dotenv
```

Create a `.env` file in your project root:
```env
# .env
PORT=3000
NODE_ENV=development
MONGODB_URI=mongodb://localhost:27017/myapp
JWT_SECRET=my-super-secret-key-123
BCRYPT_SALT_ROUNDS=10
EMAIL_HOST=smtp.gmail.com
EMAIL_USER=nishant@example.com
EMAIL_PASS=apppassword123
```

## Loading Environment Variables

```js
// Load at the very top of your entry file (index.js or app.js)
require("dotenv").config();

// Now access via process.env
const PORT = process.env.PORT || 3000;
const MONGO_URI = process.env.MONGODB_URI;
const JWT_SECRET = process.env.JWT_SECRET;

console.log("Running on port:", PORT);
console.log("Environment:", process.env.NODE_ENV);

// In server setup:
const app = require("./app");
app.listen(PORT, () => console.log(`Server on port ${PORT}`));
```

## Best Practices

### 1. Add `.env` to `.gitignore`
```gitignore
# .gitignore
node_modules/
.env
.env.local
.env.production
```

### 2. Create a `.env.example` (safe to commit)
```env
# .env.example — commit this to show teammates what variables are needed
PORT=
MONGODB_URI=
JWT_SECRET=
EMAIL_USER=
```

### 3. Validate Environment Variables at Startup
```js
const requiredEnvVars = ["MONGODB_URI", "JWT_SECRET", "PORT"];

requiredEnvVars.forEach((key) => {
  if (!process.env[key]) {
    console.error(`Missing required environment variable: ${key}`);
    process.exit(1); // crash early rather than fail later
  }
});
```

### 4. Config Module Pattern
```js
// config/index.js — centralized config
require("dotenv").config();

module.exports = {
  port: process.env.PORT || 3000,
  mongoUri: process.env.MONGODB_URI,
  jwtSecret: process.env.JWT_SECRET,
  jwtExpiry: process.env.JWT_EXPIRY || "7d",
  nodeEnv: process.env.NODE_ENV || "development",
  isProduction: process.env.NODE_ENV === "production",
  bcryptRounds: parseInt(process.env.BCRYPT_SALT_ROUNDS) || 10,
};
```

```js
// usage in other files
const config = require("./config");
app.listen(config.port);
mongoose.connect(config.mongoUri);
```

## Different `.env` Files per Environment

```
.env              → local development defaults
.env.test         → used during testing
.env.production   → loaded on production server (managed by hosting platform)
```

```js
// Load specific env file
require("dotenv").config({ path: ".env.test" });
```

## Common Variables to Store in `.env`

| Variable | Example |
|---------|---------|
| `PORT` | `3000` |
| `NODE_ENV` | `development` / `production` |
| `MONGODB_URI` | `mongodb+srv://...` |
| `JWT_SECRET` | `random-long-string` |
| `JWT_EXPIRY` | `7d` |
| `BCRYPT_SALT_ROUNDS` | `10` |
| `EMAIL_HOST`, `EMAIL_USER`, `EMAIL_PASS` | SMTP config |
| `REDIS_URL` | `redis://localhost:6379` |
| `AWS_ACCESS_KEY`, `AWS_SECRET_KEY` | Cloud storage |
| `STRIPE_SECRET_KEY` | Payment gateway |

## Interview Questions

- **Q: What is a `.env` file?**
  A: A plain text file that stores environment-specific configuration as key=value pairs. Loaded by the `dotenv` package into `process.env` at runtime.

- **Q: Why should you never commit `.env` to Git?**
  A: It contains sensitive data like API keys, passwords, and DB credentials. Exposing them in a public repo is a major security risk.

- **Q: What is `process.env` in Node.js?**
  A: A built-in object that holds all environment variables — from the OS, CI/CD pipeline, or loaded from a `.env` file via `dotenv`.

- **Q: What is a `.env.example` file?**
  A: A template showing what environment variables are required, without actual values. Safe to commit to version control.

- **Q: How do you validate required environment variables?**
  A: Check `process.env.VAR_NAME` at startup. If undefined, log an error and call `process.exit(1)` to fail fast.

- **Q: How does production typically handle environment variables?**
  A: Hosting platforms (Heroku, AWS, Vercel, Railway) provide UI/CLI to set env vars securely — no `.env` file on the server.
