# Async / Await

`async/await` is a cleaner, more readable way to write **asynchronous code** — it makes async code look and behave like normal synchronous code.

- `async` — marks a function as asynchronous. It always returns a Promise.
- `await` — pauses execution inside an async function until the Promise resolves.

---

## Code Example

### Basic Async/Await
```js
async function getUser() {
  const response = await fetch("https://jsonplaceholder.typicode.com/users/1");
  const user = await response.json();
  console.log(user.name); // "Leanne Graham"
}

getUser();
```

### With Error Handling (try/catch)
```js
async function loadData() {
  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/posts/1");

    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }

    const post = await response.json();
    console.log("Post title:", post.title);
  } catch (error) {
    console.error("Failed to load:", error.message);
  } finally {
    console.log("Done loading");
  }
}

loadData();
```

## Async/Await vs Promises

Same logic, different style:

```js
// With Promises
function getPost() {
  return fetch("https://jsonplaceholder.typicode.com/posts/1")
    .then((res) => res.json())
    .then((data) => console.log(data.title))
    .catch((err) => console.error(err));
}

// With Async/Await — cleaner!
async function getPost() {
  try {
    const res = await fetch("https://jsonplaceholder.typicode.com/posts/1");
    const data = await res.json();
    console.log(data.title);
  } catch (err) {
    console.error(err);
  }
}
```

## Sequential vs Parallel Async Calls

### Sequential (one after another — slower)
```js
async function loadSequential() {
  const user = await fetchUser();      // Wait for user
  const posts = await fetchPosts();    // Then wait for posts
  // Total time = time(user) + time(posts)
}
```

### Parallel (at the same time — faster)
```js
async function loadParallel() {
  const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
  // Total time = max(time(user), time(posts))
  console.log(user, posts);
}
```

## Key Rules of Async/Await
1. `await` can only be used **inside** an `async` function.
2. An `async` function always returns a **Promise** — even if you return a plain value.
3. `await` only works with **Promises** (or Promise-like objects).

```js
async function example() {
  return 42; // implicitly returns Promise.resolve(42)
}

example().then(console.log); // 42
```

## Interview Questions

- **Q: What is `async/await`?**
  A: Syntactic sugar over Promises. `async` marks a function as returning a Promise; `await` pauses execution until a Promise settles.

- **Q: What does `async` return?**
  A: An `async` function always returns a Promise. If you return a value, it's wrapped in `Promise.resolve(value)`.

- **Q: Can you use `await` outside of an `async` function?**
  A: Not in older JS. In modern environments (Node.js v14+ and ESM modules), top-level `await` is supported.

- **Q: How do you handle errors with `async/await`?**
  A: Wrap the code in a `try/catch` block. The `catch` block handles rejected Promises.

- **Q: What is the difference between sequential and parallel async calls?**
  A: Sequential uses `await` one by one (slower). Parallel uses `Promise.all()` to run multiple async operations simultaneously (faster).

- **Q: Is `async/await` better than Promises?**
  A: Not "better" — just more readable. They both use Promises under the hood. Use `async/await` for linear flows; Promises directly for more control.

- **Q: What happens if you forget `await`?**
  A: You get a Promise object instead of the resolved value. The code continues without waiting, which can cause bugs.
