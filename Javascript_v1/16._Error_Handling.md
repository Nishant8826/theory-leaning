# Error Handling (try / catch)

Errors happen — a user enters wrong input, a network request fails, a value is undefined. **Error handling** lets you **catch those errors gracefully** instead of crashing the whole program.

`try/catch` is like a safety net:
- **`try`** — try to run this code.
- **`catch`** — if something goes wrong, handle it here.
- **`finally`** — always run this, no matter what.

---

## Code Example

### Basic try/catch
```js
try {
  const result = JSON.parse("invalid json {{{}"); // will throw an error
  console.log(result);
} catch (error) {
  console.error("JSON parsing failed:", error.message);
} finally {
  console.log("This always runs");
}
// Output:
// JSON parsing failed: Unexpected token 'i', "invalid js..." is not valid JSON
// This always runs
```

### try/catch with API call
```js
async function loadUser(id) {
  try {
    const response = await fetch(`https://api.example.com/users/${id}`);

    if (!response.ok) {
      throw new Error(`Server error: ${response.status}`);
    }

    const user = await response.json();
    console.log("User:", user.name);
  } catch (error) {
    console.error("Could not load user:", error.message);
  } finally {
    hideLoadingSpinner(); // always hide spinner, success or not
  }
}
```

## The Error Object
```js
try {
  null.property; // accessing property on null — TypeError
} catch (error) {
  console.log(error.name);    // "TypeError"
  console.log(error.message); // "Cannot read properties of null"
  console.log(error.stack);   // Full stack trace (useful for debugging)
}
```

## Throwing Custom Errors
You can throw your **own errors** with custom messages.

```js
function divide(a, b) {
  if (b === 0) {
    throw new Error("Cannot divide by zero!"); // custom error
  }
  return a / b;
}

try {
  console.log(divide(10, 0));
} catch (error) {
  console.error("Math error:", error.message); // "Math error: Cannot divide by zero!"
}
```

## Custom Error Classes
```js
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
  }
}

function validateAge(age) {
  if (typeof age !== "number") {
    throw new ValidationError("Age must be a number", "age");
  }
  if (age < 0 || age > 120) {
    throw new ValidationError("Age must be between 0 and 120", "age");
  }
  return true;
}

try {
  validateAge("twenty"); // throws ValidationError
} catch (error) {
  if (error instanceof ValidationError) {
    console.error(`Validation failed on field '${error.field}': ${error.message}`);
  } else {
    console.error("Unexpected error:", error.message);
  }
}
```

## Error Types in JavaScript

| Error Type | When it happens |
|-----------|----------------|
| `SyntaxError` | Invalid JS syntax (typos, missing brackets) |
| `ReferenceError` | Using a variable that doesn't exist |
| `TypeError` | Wrong type (calling a non-function, null property access) |
| `RangeError` | Value out of range (e.g., invalid array length) |
| `Error` | Generic base error class |

## Error Handling with Promises
```js
// Using .catch()
fetch("https://api.example.com/data")
  .then((res) => res.json())
  .catch((error) => console.error("Fetch failed:", error));

// Using async/await + try/catch
async function getData() {
  try {
    const res = await fetch("https://api.example.com/data");
    return await res.json();
  } catch (error) {
    console.error("Failed:", error.message);
  }
}
```

## Interview Questions

- **Q: What is `try/catch` in JavaScript?**
  A: A way to handle runtime errors. Code in `try` runs normally; if it throws, execution jumps to `catch` with the error object.

- **Q: What does `finally` do?**
  A: Runs after `try` and `catch`, regardless of whether an error occurred. Used for cleanup (e.g., closing DB connections, hiding spinners).

- **Q: What is the `Error` object?**
  A: It has `name` (error type), `message` (description), and `stack` (trace). You can also extend it to create custom error classes.

- **Q: What is the difference between `throw` and `console.error`?**
  A: `throw` stops execution and creates an actual error that can be caught. `console.error` just logs to the console — execution continues.

- **Q: How do you handle errors in Promises?**
  A: Use `.catch()` on a Promise chain, or `try/catch` around `await` in an async function.

- **Q: What is a custom error class?**
  A: A class that extends `Error` to create specific error types (e.g., `ValidationError`, `NetworkError`) with extra properties.

- **Q: What is the difference between `SyntaxError` and `ReferenceError`?**
  A: `SyntaxError` = invalid code that JS can't parse. `ReferenceError` = accessing a variable that doesn't exist at runtime.
