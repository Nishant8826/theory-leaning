# React.memo, useMemo, and useCallback (In-Depth Guide for Junior Developers)

React provides **React.memo**, **useMemo**, and **useCallback** to help optimize performance by avoiding unnecessary re-renders and computations.  
Before using them, it’s critical to understand **how React renders components**.

---

## How React Re-renders Components

A React component re-renders when:
1. Its **state changes**
2. Its **props change**
3. Its **parent component re-renders**

Even if the UI looks the same, **the component function runs again**.

---

## 1. React.memo — Prevent Unnecessary Re-renders

### What is React.memo?

`React.memo` is a **higher-order component** that memoizes a component.

> It re-renders the component **only if its props change**.

---

### Problem Without React.memo

```js
function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(count + 1)}>+</button>
      <Child name="Nishant" />
    </>
  );
}

function Child({ name }) {
  console.log("Child rendered");
  return <div>{name}</div>;
}
```

`Child` re-renders even though `name` never changes.

---

### Solution With React.memo

```js
const Child = React.memo(function Child({ name }) {
  console.log("Child rendered");
  return <div>{name}</div>;
});
```

Now `Child` only re-renders if `name` changes.

---

### How React.memo Works

- Uses **shallow comparison**
- Compares old props vs new props
- Works well with **primitive values**

Shallow comparison example:

```js
{} === {} // false
```

---

### When to Use React.memo

Use when:
- Component is large
- Props rarely change
- Parent re-renders often

Avoid when:
- Component is small
- Props change frequently
- Used everywhere without need

---

## 2. useMemo — Memoize a Computed Value

### What Problem Does useMemo Solve?

Expensive calculations run on **every render**:

```js
const sortedList = bigArray.sort(); // runs every time
```

---

### Using useMemo

```js
const sortedList = useMemo(() => {
  return bigArray.sort();
}, [bigArray]);
```

Calculation only runs when `bigArray` changes.

---

### Mental Model

> “Recalculate **only when dependencies change**.”

---

### When to Use useMemo

Use for:
- Sorting large arrays
- Filtering big lists
- Heavy computations

Avoid for:
- Simple calculations
- Primitive values
- Over-optimization

---

## 3. useCallback — Memoize a Function

### Why Functions Cause Re-renders

Functions are recreated on every render:

```js
const handleClick = () => console.log("Clicked");
```

If passed to a memoized child, it causes re-render.

---

### Using useCallback

```js
const handleClick = useCallback(() => {
  console.log("Clicked");
}, []);
```

Same function reference across renders.

---

### Important Concept

```js
useCallback(fn, deps) === useMemo(() => fn, deps)
```

---

## 4. Using React.memo + useCallback Together

### Child Component

```js
const Child = React.memo(({ onClick }) => {
  console.log("Child rendered");
  return <button onClick={onClick}>Click</button>;
});
```

---

### Wrong Parent Implementation

```js
function Parent() {
  const handleClick = () => console.log("Hi");
  return <Child onClick={handleClick} />;
}
```

Child re-renders every time.

---

### Correct Parent Implementation

```js
function Parent() {
  const handleClick = useCallback(() => {
    console.log("Hi");
  }, []);

  return <Child onClick={handleClick} />;
}
```

Child does NOT re-render unnecessarily.

---

## Common Mistakes 

- Using memo everywhere
- Using useCallback without memoized children
- Missing dependencies
- Optimizing too early
- Making code harder to read

---

##  Quick Decision Table

| Situation | Use |
|--------|------|
| Prevent child re-renders | React.memo |
| Heavy calculation | useMemo |
| Passing function to memoized child | useCallback |
| Small/simple component | Nothing |
| No performance issue | Nothing |

---

## Final Mental Model

- **React.memo** → controls *component re-rendering*
- **useMemo** → controls *value recalculation*
- **useCallback** → controls *function identity*

> Optimize only when necessary. Readability first, performance second.
