# Pods

## Why This Exists
In Docker, the smallest unit of deployment is a single Container. In Kubernetes, the smallest unit is a **Pod**. 

A Pod is a wrapper around one or more containers. Kubernetes doesn't run containers directly; it runs Pods. Why? Because sometimes containers are so tightly coupled that they need to share resources (like network and storage) and live on the exact same server. The Pod provides this shared environment.

## Real World Analogy
Think of a Pod as a **Peapod** or a **Shared Apartment**.
- The peas (containers) live together in the same pod.
- Roommates (containers) in a shared apartment share the same address (IP), the same plumbing (Network), and the same storage space (Volumes).

## Core Concepts
- **Pod**: The smallest deployable unit in Kubernetes.
- **Single-Container Pod**: The most common use case (1 pod = 1 container).
- **Multi-Container Pod**: Used when containers must work together (e.g., an app container and a helper log-shipper container).
- **Shared IP**: All containers in a Pod share the same IP address and port space (they can talk to each other via `localhost`).

## Practical Commands

```bash
# Run a simple nginx pod imperatively
kubectl run my-nginx --image=nginx:alpine

# Get a list of all running pods
kubectl get pods

# Get detailed information about a specific pod
kubectl describe pod my-nginx

# Delete a pod
kubectl delete pod my-nginx
```

## Hands-On Exercise
Let's create a Pod using a YAML definition file (declarative way).

1. Create a file named `pod.yaml`:
   ```yaml
   apiVersion: v1
   kind: Pod
   metadata:
     name: my-web-app
     labels:
       app: web
   spec:
     containers:
     - name: nginx-container
       image: nginx:alpine
       ports:
       - containerPort: 80
   ```
2. Apply it to your cluster:
   ```bash
   kubectl apply -f pod.yaml
   ```
3. Verify it is running:
   ```bash
   kubectl get pods
   ```

## Mini Project
**Task**: Create a Multi-Container Pod with a "Sidecar" pattern.

1. Create a file named `multi-pod.yaml`:
   ```yaml
   apiVersion: v1
   kind: Pod
   metadata:
     name: sidecar-pod
   spec:
     containers:
     - name: main-container
       image: alpine
       command: ["sh", "-c", "while true; do echo 'Generating logs' >> /var/log/app.log; sleep 5; done"]
       volumeMounts:
       - name: log-vol
         mountPath: /var/log

     - name: sidecar-container
       image: alpine
       command: ["sh", "-c", "tail -f /var/log/app.log"]
       volumeMounts:
       - name: log-vol
         mountPath: /var/log

     volumes:
     - name: log-vol
       emptyDir: {}
   ```
2. Apply it. Here, the sidecar container reads and displays logs generated by the main container. This is a common pattern for log shipping.

## Real Production Usage
- **Don't create Pods directly**: In production, you almost never create naked Pods using `kubectl run` or Pod YAMLs. If a naked Pod dies, it is not recreated. Instead, you use **Deployments** or **StatefulSets** which manage Pods for you and handle scaling and self-healing.

## Common Mistakes
- **Port conflicts in a Pod**: Since all containers in a Pod share the same network space, they cannot bind to the same port.
- **Tightly coupling unrelated apps**: Putting your Node.js app and MySQL database in the same Pod. They should be in separate Pods so they can scale independently.

## Debugging Guide
- **Container is failing**: Use `kubectl logs <pod-name>` to see the logs. If it's a multi-container pod, specify the container: `kubectl logs <pod-name> -c <container-name>`.
- **Pod stuck in Pending**: Run `kubectl describe pod <pod-name>`. Usually, this means the cluster doesn't have enough resources (CPU/RAM) or can't pull the image.

## Best Practices
- **One process per container**: Keep containers focused on one job.
- **Use Sidecars wisely**: Only put containers in the same Pod if they strictly need to share resources and lifecycle.

## Interview Questions
1. **What is a Pod in Kubernetes?**
   *Answer*: A Pod is the smallest deployable unit in Kubernetes, representing a single instance of a running process. It can contain one or more tightly coupled containers.
2. **Do containers in the same Pod share the same IP address?**
   *Answer*: Yes, they share the same network namespace and IP address, and can communicate via `localhost`.

## Summary
Pods are the atomic unit of Kubernetes. While you rarely create them directly in production, understanding how they share networks and storage is essential for mastering Kubernetes.
