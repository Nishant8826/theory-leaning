# Project 6: Deploying Next.js (SSR) on AWS

## What Is This Project?
This project focuses exclusively on deploying a Next.js application leveraging Server-Side Rendering (SSR). Unlike a standard React SPA (which is just static HTML/JS/CSS files hosted on S3), Next.js SSR requires an active Node.js server to render pages on the fly. 

## Why Do This Project?
Next.js is the dominant React framework, but many developers mistakenly try to deploy it to an S3 bucket and find that their API routes and `getServerSideProps` functions are completely broken. You must understand how to host Next.js properly using AWS Amplify or ECS to retain its powerful SSR and ISR features.

## How to Build It (Step-by-Step)

### Step 1: Prepare Next.js for Production
By default, Next.js builds a massive `.next` folder. To deploy to Docker efficiently, we need a "standalone" output.
1. Open your `next.config.js` and add the standalone directive:
   ```javascript
   module.exports = {
     output: 'standalone',
   }
   ```
2. Run `npm run build`. This generates a `.next/standalone` directory that includes *only* the necessary files and `node_modules` to run the server, drastically shrinking the deployment size.

### Step 2: Option A - Fully Managed (AWS Amplify)
*Best for solo developers and small teams who want a "Vercel-like" experience on AWS.*
1. Go to the **AWS Amplify** Dashboard.
2. Click **New App** > **Host web app**.
3. Connect your GitHub repository and select your Next.js branch.
4. Amplify will automatically detect it is a Next.js App.
5. Review the build settings. If you use environment variables (e.g., `NEXT_PUBLIC_API_URL`), add them in the Advanced Settings tab.
6. Click **Save and deploy**.
   *(Amplify automatically provisions the Node.js server, CloudFront CDN, and handles the CI/CD pipeline internally).*

### Step 3: Option B - Dockerized (ECS Fargate)
*Best for enterprise teams who need the Next.js app to live inside a custom VPC next to their backend APIs and databases.*
1. Create a `Dockerfile` in the root of your Next.js project:
   ```dockerfile
   FROM node:22-alpine AS runner
   WORKDIR /app
   
   ENV NODE_ENV production
   
   # Copy the standalone output generated by next build
   COPY --chown=node:node .next/standalone ./
   COPY --chown=node:node .next/static ./.next/static
   COPY --chown=node:node public ./public
   
   USER node
   EXPOSE 3000
   ENV PORT 3000
   
   CMD ["node", "server.js"]
   ```
2. Build and push the Docker image to Amazon ECR (following the steps from Project 2).
3. Deploy the container to ECS Fargate behind an Application Load Balancer.

### Step 4: Asset Optimization (CloudFront)
If you choose Option B (ECS), your Next.js Docker container will be serving the heavy image files (`/public/hero.jpg`) and static CSS. This wastes expensive compute resources.
1. Create an Amazon CloudFront distribution.
2. Set the Origin to your Application Load Balancer.
3. CloudFront will cache the static assets globally, meaning your Next.js ECS containers only have to process dynamic SSR requests, significantly reducing your ECS auto-scaling costs.

## Production Impact
- **SEO & Performance**: Proper SSR deployment ensures that web crawlers (Googlebot) see fully rendered HTML pages instantly, drastically improving your search engine rankings compared to an empty React SPA.
- **Flexibility**: Deploying Next.js via Docker on ECS ensures you are not locked into proprietary hosting platforms (like Vercel). You own the entire infrastructure.

## Knowledge Transfer (KT)
- **Static Export vs SSR**: If you run `next build && next export` (or use `output: 'export'`), Next.js removes all Node.js features and gives you pure HTML/JS files. *That* specific output CAN be deployed to S3 like a normal React app, but you lose API routes, SSR, and Image Optimization.
- **Cold Starts**: If you deploy Next.js using AWS Lambda (via Serverless Framework), users may experience a 1-2 second delay when visiting a new page due to cold starts. ECS Fargate keeps the Node server warm 24/7, eliminating cold starts entirely.
