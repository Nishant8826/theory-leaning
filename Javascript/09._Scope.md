# Scope & Lexical Environment

## Simple Explanation
**Scope** = Where in your code a variable **can be accessed**.

Not all variables are available everywhere. Some are limited to a specific function or block of code.

**Lexical Environment** = When a function is created, JS remembers **which scope it was created in** — and that becomes its "lexical environment". This is the foundation of closures.

---

## Real-World Example
Think of it like a house with rooms:
- **Global Scope** = The hallway — everyone can access it.
- **Function Scope** = Your bedroom — only you can access it.
- **Block Scope** = A locked drawer in your room — only accessible inside that block.

---

## Types of Scope

### 1. Global Scope
Variables declared outside any function or block — accessible **everywhere**.

```js
const appName = "MyApp"; // global scope

function show() {
  console.log(appName); // ✅ accessible here
}

show(); // "MyApp"
console.log(appName); // ✅ accessible here too
```

### 2. Function Scope
Variables declared inside a function — only accessible **within that function**.

```js
function login() {
  const token = "abc123"; // function scope
  console.log(token); // ✅ accessible here
}

login();
// console.log(token); // ❌ ReferenceError — token doesn't exist here
```

### 3. Block Scope (`let` and `const` only)
Variables declared inside `{}` (if, for, while, etc.) — only accessible inside that block.

```js
if (true) {
  let message = "Hello from block!";
  const greeting = "Hi!";
  var legacy = "I am var!"; // var ignores block scope!
  console.log(message); // ✅
}

// console.log(message);  // ❌ ReferenceError
// console.log(greeting); // ❌ ReferenceError
console.log(legacy); // ✅ var leaks out of block!
```

## Scope Chain
When JS can't find a variable, it looks **outward** — up through each enclosing scope — until it reaches global scope. This is called the **scope chain**.

```js
const global = "I'm global";

function outer() {
  const outerVar = "I'm outer";

  function inner() {
    const innerVar = "I'm inner";
    // inner can access all of these:
    console.log(innerVar);  // ✅ own scope
    console.log(outerVar);  // ✅ outer scope
    console.log(global);    // ✅ global scope
  }

  inner();
  // console.log(innerVar); // ❌ can't go inward
}

outer();
```

## Lexical Environment
A **Lexical Environment** is created every time a function or block is executed. It consists of:
1. **Environment Record** — stores all local variables.
2. **Outer reference** — pointer to the parent lexical environment.

This forms the **scope chain** at the time of creation (not at call time).

```js
function makeCounter() {
  let count = 0; // stored in makeCounter's Lexical Environment

  return function () {
    count++;             // refers to parent lexical environment
    return count;
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
// Even though makeCounter() is done, count is kept alive by closure
```

## `var` vs `let` vs `const` — Scope Summary

| | `var` | `let` | `const` |
|---|-------|-------|---------|
| Scope | Function | Block | Block |
| Hoisted | ✅ (as `undefined`) | ✅ (but TDZ) | ✅ (but TDZ) |
| Re-assignable | ✅ | ✅ | ❌ |
| Re-declarable | ✅ | ❌ | ❌ |

## Interview Questions

- **Q: What is scope in JavaScript?**
  A: Scope defines where a variable is accessible. JavaScript has global, function, and block scope.

- **Q: What is the difference between `var`, `let`, and `const` in terms of scope?**
  A: `var` is function-scoped; `let` and `const` are block-scoped. `var` is also hoisted with `undefined`, while `let`/`const` are in TDZ.

- **Q: What is the scope chain?**
  A: When a variable isn't found in the current scope, JS looks outward to enclosing scopes, forming a "chain" up to the global scope.

- **Q: What is a Lexical Environment?**
  A: A data structure that stores variable bindings and a reference to the outer environment. It's created when a function/block is executed and defines the scope chain.

- **Q: What is lexical scoping?**
  A: Scope is determined by where the function is **written** in the code, not where it's **called**. This is how closures work.

- **Q: Why does `var` leak out of blocks?**
  A: Because `var` is function-scoped, not block-scoped. Only function boundaries stop `var` from leaking.
