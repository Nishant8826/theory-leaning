# Closures

## Simple Explanation
A **closure** is a function that **remembers and accesses** variables from its outer scope even after the outer function has finished executing.

Think of it like this: Imagine you write a note with important info, and even after you leave a room, that note (the inner function) still has the info from that room (the outer function's variables).

---

## Real-World Example
A **counter** — you want to keep track of how many times a user clicked a button, and you don't want that count to be accessible or changeable from outside.

---

## Code Example

### Basic Closure
```js
function outer() {
  let count = 0; // count lives here

  function inner() {
    count++;
    console.log("Count:", count);
  }

  return inner; // return the inner function
}

const counter = outer(); // outer() is done, but count is remembered!
counter(); // Count: 1
counter(); // Count: 2
counter(); // Count: 3
```

### Classic Closure (Password Manager Example)
```js
function createPasswordManager() {
  let password = "secret123"; // private, no one can touch it directly

  return {
    getPassword: () => "••••••••",    // hide real password
    changePassword: (newPass) => {
      password = newPass;
      console.log("Password updated!");
    },
  };
}

const pm = createPasswordManager();
console.log(pm.getPassword()); // "••••••••"
pm.changePassword("newPass456");
```

### Original Example (from MDN)
```js
function init() {
  var name = "Mozilla"; // name is a local variable created by init
  function displayName() {
    // displayName() is the inner function, forming a closure
    console.log(name); // uses variable from the parent function
  }
  displayName();
}
init(); // "Mozilla"
```

## Why Are Closures Useful?
| Use Case | Example |
|----------|---------|
| **Data privacy / encapsulation** | Counter that can't be reset from outside |
| **Factory functions** | Create multiple customized functions |
| **Memoization** | Cache results of expensive functions |
| **Event handlers** | Remember variables when event fires later |

### Factory Function Example
```js
function greetingMaker(greeting) {
  return function (name) {
    return `${greeting}, ${name}!`;
  };
}

const sayHello = greetingMaker("Hello");
const sayHi = greetingMaker("Hi");

console.log(sayHello("Nishant")); // "Hello, Nishant!"
console.log(sayHi("Raj"));       // "Hi, Raj!"
```

## Common Closure Mistake (with loops)
```js
// ❌ Wrong — all setTimeout callbacks share the same `i`
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 1000);
}
// Output: 3, 3, 3

// ✅ Fixed — using let creates a new scope for each iteration
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 1000);
}
// Output: 0, 1, 2
```

## Interview Questions

- **Q: What is a closure?**
  A: A closure is when an inner function retains access to the variables of its outer function, even after the outer function has finished executing.

- **Q: Why are closures useful?**
  A: They allow data encapsulation (private variables), factory functions, and are core to patterns like memoization and module design.

- **Q: What is the difference between a closure and a regular function?**
  A: A regular function doesn't "remember" outer variables after execution. A closure does — it captures a reference to the outer scope.

- **Q: Can closures cause memory leaks?**
  A: Yes. If a closure holds reference to large objects and is never released, those objects stay in memory. Avoid unnecessary long-lived closures.

- **Q: What happens with closures in loops with `var`?**
  A: All iterations share the same `var` variable because `var` is function-scoped. Using `let` (block-scoped) fixes this by creating a new binding per iteration.

- **Q: What is a practical use of closure in React?**
  A: Every React event handler is a closure — it captures `state` or `props` from the enclosing render.
