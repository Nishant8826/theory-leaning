# How JavaScript Works?

## Simple Explanation
When you run a JavaScript program, JS doesn't just jump into executing code blindly. It goes through a **two-phase process** inside something called the **Global Execution Context (GEC)**:

1. **Memory Phase** – JS reads all your variables and functions and gives them a temporary value (`undefined`).
2. **Execution Phase** – JS runs your code line by line, assigning real values.

Think of it like reading a recipe first (memory phase) and then actually cooking (execution phase).

---

## Real-World Example
Imagine you're organizing a birthday party:
- **Memory Phase** = You write a checklist of everything you need (cake, balloons, etc.) but haven't bought anything yet.
- **Execution Phase** = You go out and actually buy everything on the list.

---

## Code Example

```js
console.log(name); // undefined (memory phase gave it undefined)
var name = "Nishant";
console.log(name); // "Nishant" (execution phase assigned the real value)

greet(); // works! function is fully stored in memory phase
function greet() {
  console.log("Hello!");
}
```

---

## Global Execution Context (GEC)
- Every JavaScript program starts with a GEC.
- It has **Memory Phase** and **Execution Phase**.
- Once all code is done, GEC is removed from the **Call Stack**.

### Function Execution Context (FEC)
When a function is called:
- JS creates a **new execution context** just for that function.
- It has its own memory phase and execution phase.
- When the function finishes, its context is removed (popped) from the Call Stack.

---

## Hoisting & TDZ

**Hoisting** = JS moves all variable and function *declarations* to the top of their scope during the memory phase.

```
Hoisting is the default behavior of JavaScript where all variables and
function declarations are moved to the top of their scope during the
memory allocation phase.
```

**Temporal Dead Zone (TDZ)** = The period where `let` and `const` variables exist in memory but cannot be accessed yet (until the line where they're declared is reached).

```
TDZ refers to the period during which let & const variables exist in
the scope but cannot be accessed until they're declared.
```

```js
console.log(x); // ❌ ReferenceError — x is in TDZ
let x = 10;
```

---

## Call Stack

The **Call Stack** is like a stack of trays in a cafeteria. The last tray placed on top is the first one taken off (**LIFO — Last In, First Out**).

- JS is **single-threaded** — it does one thing at a time.
- Every function call gets pushed onto the stack.
- When it finishes, it gets popped off.
- JS **never waits** — it moves on immediately.

```js
function first() {
  second();
  console.log("First done");
}
function second() {
  console.log("Second done");
}
first();
// Output:
// "Second done"
// "First done"
```

```
How it works:
-> JS starts → GEC is pushed to Call Stack
-> first() is called → FEC of first() is pushed
-> second() is called inside first() → FEC of second() is pushed
-> second() finishes → popped off
-> first() finishes → popped off
-> GEC finishes → popped off
```

---

## Callback Queue (Task Queue / Macrotask Queue)
- A waiting room for **asynchronous callbacks** (like `setTimeout`, DOM events).
- Follows **FIFO (First In, First Out)**.
- Callbacks here are only processed **after** the Call Stack and Microtask Queue are empty.

---

## Microtask Queue
- A higher-priority waiting room for **Promises** and **`queueMicrotask()`**.
- Also follows **FIFO**.
- **Has MORE priority** than the Callback Queue.
- The Event Loop empties the Microtask Queue **completely** before touching the Callback Queue.

---

## Event Loop

The **Event Loop** is JS's traffic controller for async operations. Even though JS is single-threaded, the Event Loop lets it handle async tasks without freezing.

```
Event Loop checks:
1. Is the Call Stack empty?
2. If yes → check Microtask Queue → run all tasks there
3. Then → check Callback Queue → run one task
4. Repeat forever
```

```js
console.log("Start");

setTimeout(() => console.log("setTimeout"), 0);

Promise.resolve().then(() => console.log("Promise"));

console.log("End");

// Output:
// Start
// End
// Promise       ← Microtask Queue (higher priority)
// setTimeout    ← Callback Queue
```

---

## Starving
When too many tasks pile up in the **Microtask Queue**, the **Callback Queue** tasks keep getting delayed. This is called **starvation**.

```
Solutions:
1. Use Web Workers for CPU-heavy tasks.
2. Break long-running tasks into smaller chunks.
```

## Interview Questions

- **Q: What is the Global Execution Context?**
  A: It's the default environment where JS runs. It has a Memory Phase (stores variables as `undefined`) and an Execution Phase (runs code line by line).

- **Q: What is Hoisting?**
  A: JS moves variable and function declarations to the top of their scope during the memory phase. `var` is hoisted with `undefined`; `let`/`const` are hoisted but stay in TDZ.

- **Q: What is the TDZ (Temporal Dead Zone)?**
  A: The period between the start of the scope and the line where a `let`/`const` variable is declared. Accessing it in this window causes a `ReferenceError`.

- **Q: What is the Call Stack?**
  A: A data structure (LIFO) that manages the execution of function calls. Each function gets its own context pushed/popped from the stack.

- **Q: What is the Event Loop?**
  A: A mechanism that continuously checks if the Call Stack is empty and then moves tasks from the Microtask Queue (first) and Callback Queue (second) to the Call Stack.

- **Q: What is the difference between Microtask Queue and Callback Queue?**
  A: Microtask Queue (Promises) has higher priority. The Event Loop always empties the Microtask Queue before processing any Callback Queue task.

- **Q: Is JavaScript single-threaded?**
  A: Yes. JS executes one task at a time on a single thread, but uses the Event Loop, Web APIs, and queues to handle async code without blocking.
