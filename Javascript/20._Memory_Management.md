# Memory Management (Basics)

## Simple Explanation
Every variable, object, or function you create in JavaScript takes up **memory**. When you're done using something, that memory should be freed.

JavaScript does this **automatically** using a **Garbage Collector (GC)** — you don't have to manually free memory like in C/C++. But you can still accidentally cause **memory leaks** where memory is never freed.

Think of memory like a whiteboard:
- You write things on it as you work.
- When you're done, you erase them to make room.
- A **memory leak** = forgetting to erase something you no longer need.

---

## How JavaScript Manages Memory

### 3 Steps:
1. **Allocate** — Memory is reserved when you create a variable/object.
2. **Use** — You read/write to that memory.
3. **Release** — JS's GC detects it's no longer needed and frees it.

```js
// Step 1: Allocation
let user = { name: "Nishant", age: 22 }; // memory allocated

// Step 2: Use
console.log(user.name);

// Step 3: Release (happens automatically)
user = null; // now the object has no reference → GC can free it
```

---

## Garbage Collection

JavaScript uses the **Mark and Sweep** algorithm:
1. Start from "roots" (global variables, call stack).
2. Mark everything reachable.
3. Sweep (remove) everything NOT marked.

```js
function createUser() {
  const user = { name: "Nishant" }; // allocated
  return user; // returned — still reachable
}

const myUser = createUser(); // still in memory (referenced by myUser)
// When myUser goes out of scope or is set to null → GC frees it
```

## Common Memory Leaks

### 1. Global Variables (Accidental)
```js
function leak() {
  userName = "Nishant"; // ❌ No var/let/const — becomes a global! Never GC'd
}
```

### 2. Forgotten Event Listeners
```js
// ❌ Adding listener in a loop or component without removing it
document.getElementById("btn").addEventListener("click", handler);

// ✅ Remove when no longer needed
document.getElementById("btn").removeEventListener("click", handler);
```

### 3. Closures Holding References
```js
function createBigClosure() {
  const largeData = new Array(1000000).fill("data"); // 1 million items

  return function () {
    console.log(largeData[0]); // closure keeps largeData alive forever!
  };
}

const fn = createBigClosure(); // largeData stays in memory as long as fn exists
```

### 4. Detached DOM Elements
```js
// DOM element removed from page but still referenced in JS
let detachedNode = document.getElementById("old-banner");
document.body.removeChild(detachedNode);
// detachedNode still holds a reference → GC can't free the DOM element
detachedNode = null; // ✅ Fix: remove the JS reference too
```

### 5. Uncleared Timers
```js
// ❌ setInterval runs forever, keeps callback and its closure alive
const intervalId = setInterval(() => {
  // heavy operation
}, 1000);

// ✅ Always clear when done
clearInterval(intervalId);
```

## Tips for Good Memory Management

| Tip | Why |
|-----|-----|
| Use `let`/`const` not `var` globally | Avoids accidental global variables |
| Remove event listeners when done | Prevents memory leaks in SPAs |
| Clear `setInterval`/`setTimeout` | Prevents long-running background tasks |
| Set large objects to `null` when done | Helps GC collect them faster |
| Use `WeakMap`/`WeakSet` for caches | Keys are held weakly — auto-collected |

---

## WeakMap & WeakSet (Advanced)

These hold **weak references** — if the key/item has no other references, it can be garbage collected automatically.

```js
const cache = new WeakMap();

function processUser(user) {
  if (cache.has(user)) return cache.get(user); // reuse cached result

  const result = expensiveComputation(user);
  cache.set(user, result); // auto-cleaned when user is GC'd
  return result;
}
```

## Interview Questions

- **Q: What is garbage collection in JavaScript?**
  A: An automatic memory management process where JS identifies and frees memory occupied by objects that are no longer reachable.

- **Q: What is a memory leak?**
  A: When memory that's no longer needed is never released, causing the application to consume more and more memory over time.

- **Q: What are common causes of memory leaks in JS?**
  A: Accidental global variables, forgotten event listeners, closures holding large data, detached DOM nodes, and uncleaned timers.

- **Q: What is the Mark and Sweep algorithm?**
  A: GC starts from roots, marks all reachable objects, then sweeps (removes) all unmarked ones.

- **Q: What is the difference between `WeakMap` and `Map`?**
  A: `WeakMap` keys are held weakly — if the key object has no other references, it's garbage collected. `Map` prevents GC by holding strong references.

- **Q: How do you prevent memory leaks in a React/SPA app?**
  A: Remove event listeners in cleanup functions, clear intervals/timeouts, and avoid storing large data in closures unnecessarily.
