# Hoisting

## Simple Explanation
**Hoisting** is JavaScript's behavior of moving **declarations** to the top of their scope before code runs.

Think of it like a teacher who reads the class register first (memory phase), knowing all students' names before class begins — even if a student arrives late.

> 📌 Only the **declaration** is hoisted — NOT the assignment/value.

---

## Real-World Example
Imagine you're writing a script and you call a helper function at the top before defining it below. With declarations, JS already "knows" about it.

---

## Code Example

### `var` — Hoisted with `undefined`
```js
console.log(name); // undefined (not an error — hoisted!)
var name = "Nishant";
console.log(name); // "Nishant"

// JS sees it as:
// var name; // hoisted
// console.log(name); // undefined
// name = "Nishant";
// console.log(name); // "Nishant"
```

### `let` and `const` — Hoisted but in TDZ (Temporal Dead Zone)
```js
console.log(age); // ❌ ReferenceError — in TDZ
let age = 22;

console.log(city); // ❌ ReferenceError — in TDZ
const city = "Delhi";
```

Even though `let`/`const` are hoisted, they cannot be accessed until the declaration line is reached.

### Function Declarations — Fully Hoisted
```js
greet(); // ✅ Works! "Hello!"

function greet() {
  console.log("Hello!");
}
```

The entire function body is hoisted, so you can call it before the definition.

### Function Expressions & Arrow Functions — NOT Hoisted
```js
sayHi(); // ❌ TypeError: sayHi is not a function
var sayHi = function () {
  console.log("Hi!");
};

sayBye(); // ❌ ReferenceError — let is in TDZ
let sayBye = () => console.log("Bye!");
```

## Hoisting Summary Table

| Type | Hoisted? | Initial Value | Usable Before Declaration? |
|------|---------|--------------|--------------------------|
| `var` | ✅ Yes | `undefined` | ✅ (but value is `undefined`) |
| `let` | ✅ Yes | TDZ | ❌ ReferenceError |
| `const` | ✅ Yes | TDZ | ❌ ReferenceError |
| `function` declaration | ✅ Yes | Full function | ✅ Yes |
| `function` expression (`var`) | ✅ Var part | `undefined` | ❌ TypeError |
| Arrow function (`let`) | ✅ Let part | TDZ | ❌ ReferenceError |

---

## Why Avoid Relying on Hoisting?
```js
// ❌ Confusing — relying on var hoisting
console.log(score); // undefined — not obvious!
var score = 100;

// ✅ Clear — declare before use
let score = 100;
console.log(score); // 100
```

Always declare variables **before** using them — don't rely on hoisting for clarity.

## Interview Questions

- **Q: What is hoisting in JavaScript?**
  A: Hoisting is JS's behavior of moving variable and function declarations to the top of their scope during the memory phase, before code executes.

- **Q: What is the difference between `var` and `let` hoisting?**
  A: `var` is hoisted with an initial value of `undefined`, so it can be accessed (but gives `undefined`). `let`/`const` are hoisted but remain in the TDZ — accessing them throws a `ReferenceError`.

- **Q: What is the Temporal Dead Zone (TDZ)?**
  A: The period between the start of the scope and the line where a `let`/`const` variable is declared. During TDZ, accessing the variable throws a `ReferenceError`.

- **Q: Are function declarations hoisted?**
  A: Yes — the entire function body is hoisted, so you can call a function declaration before it's defined.

- **Q: Are function expressions hoisted?**
  A: Only the variable declaration is hoisted (not the function value). If declared with `var`, you get `undefined`; with `let`/`const`, you get a `ReferenceError`.

- **Q: Why is relying on hoisting a bad practice?**
  A: It leads to confusing code. A variable accessed before assignment gives `undefined`, which is hard to debug. Best practice is to always declare before using.
