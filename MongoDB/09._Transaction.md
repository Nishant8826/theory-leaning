# Transactions in MongoDB

## Simple Explanation
A **transaction** is a group of database operations that either **all succeed** or **all fail together** — never a partial result.

Think of a bank transfer:
- Deduct ₹5000 from Account A
- Add ₹5000 to Account B

If step 1 succeeds but step 2 fails (power cut, crash), you've lost money. A **transaction** ensures both happen or neither happens — this is called **atomicity**.

---

## Real-World Example
An e-commerce checkout:
1. Deduct item from inventory
2. Create an order record
3. Charge the user's account

If step 3 fails, steps 1 and 2 should be **rolled back** — e-commerce can't lose stock and not create an order.

---

## ACID Properties (What Transactions Guarantee)

| Property | Meaning |
|---------|---------|
| **Atomicity** | All operations succeed or all are rolled back |
| **Consistency** | Data stays valid before and after the transaction |
| **Isolation** | Transactions don't interfere with each other |
| **Durability** | Once committed, data is permanently saved |

---

## When to Use Transactions

✅ Use when:
- Multiple documents must be updated together
- Money transfers, order placement, inventory updates
- Any "all or nothing" scenario

❌ Don't overuse — transactions are slower. Use them only when truly needed.

---

## MongoDB Transactions — Requirements

- Transactions work on **replica sets** (not standalone servers)
- For simple single-document operations, MongoDB is already atomic by default — no transaction needed
- Multi-document transactions supported since MongoDB 4.0
- Cross-collection transactions supported since MongoDB 4.2

## Code Example

### Shell — Using a Session
```js
// Start a session
const session = db.getMongo().startSession();

try {
  // Start the transaction
  session.startTransaction();

  // All operations use the same session
  db.accounts.updateOne(
    { userId: "user1" },
    { $inc: { balance: -5000 } },
    { session }
  );

  db.accounts.updateOne(
    { userId: "user2" },
    { $inc: { balance: 5000 } },
    { session }
  );

  // Commit — save all changes
  session.commitTransaction();
  console.log("Transaction committed!");

} catch (err) {
  // Rollback — undo all changes
  session.abortTransaction();
  console.error("Transaction aborted:", err.message);

} finally {
  session.endSession();
}
```

---

### Node.js with Mongoose
```js
const mongoose = require("mongoose");

async function transferMoney(fromUserId, toUserId, amount) {
  const session = await mongoose.startSession();

  try {
    session.startTransaction();

    // Deduct from sender
    const sender = await Account.findByIdAndUpdate(
      fromUserId,
      { $inc: { balance: -amount } },
      { session, new: true }
    );

    if (sender.balance < 0) {
      throw new Error("Insufficient balance");
    }

    // Add to receiver
    await Account.findByIdAndUpdate(
      toUserId,
      { $inc: { balance: amount } },
      { session }
    );

    // Create a transaction record
    await Transaction.create([{
      from: fromUserId,
      to: toUserId,
      amount,
      status: "completed",
      createdAt: new Date()
    }], { session });

    await session.commitTransaction();
    console.log("Transfer successful!");

  } catch (err) {
    await session.abortTransaction();
    console.error("Transfer failed, rolled back:", err.message);
    throw err;

  } finally {
    session.endSession();
  }
}
```

## Single Document vs Multi-Document Operations

| | Single Document | Multi-Document |
|--|----------------|---------------|
| Atomic by default | ✅ Yes | ❌ No — need a transaction |
| Transaction needed | ❌ No | ✅ Yes (for all-or-nothing) |
| Performance | Faster | Slightly slower |
| Example | Update user's name | Transfer money between accounts |

---

## Transaction Flow

```
1. Start session
      ↓
2. Start transaction within session
      ↓
3. Run all operations (passing session to each)
      ↓
4a. All OK? → commitTransaction() → changes saved
4b. Error? → abortTransaction() → all changes rolled back
      ↓
5. End session
```

## Interview Questions

- **Q: What is a transaction in MongoDB?**
  A: A group of operations that execute as a single unit — all succeed or all are rolled back. Useful for multi-document operations that must be atomic.

- **Q: What does ACID stand for?**
  A: Atomicity (all or nothing), Consistency (data stays valid), Isolation (transactions don't interfere), Durability (changes are permanent after commit).

- **Q: When did MongoDB add multi-document transaction support?**
  A: MongoDB 4.0 added replica set transactions. Version 4.2 added cross-shard transactions.

- **Q: Is a single MongoDB document update atomic?**
  A: Yes — single document operations are inherently atomic in MongoDB. Transactions are only needed for multi-document operations.

- **Q: What is `session.abortTransaction()`?**
  A: Rolls back all changes made within the current transaction — undoes everything back to the state before the transaction started.

- **Q: What is the difference between `commitTransaction()` and `abortTransaction()`?**
  A: `commit` permanently saves all changes. `abort` rolls back all changes as if the operations never happened.

- **Q: Why should transactions be used sparingly?**
  A: Transactions lock resources and add overhead — they're slower than non-transactional operations. Only use them when true atomicity is required.
