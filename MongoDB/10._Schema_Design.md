# Schema Design & Data Modeling

## Simple Explanation
**Schema design** = deciding how to structure your data in MongoDB — what fields to include, how to organize them, and how documents relate to each other.

Even though MongoDB has a **flexible schema** (no rigid structure required), a well-thought-out schema is critical for performance, scalability, and maintainability.

**Rule of thumb in MongoDB:**
> "Model your data for how your application queries it — not how it's stored."

---

## Real-World Example
You're building a blog app. You need to decide:
- Should comments be inside each post document (embedded)?
- Or stored in a separate `comments` collection (referenced)?
- What fields should a post have?

Great schema design answers these questions based on your access patterns.

---

## Key Schema Design Principles

### 1. Model for Your Queries (Access Patterns)
Before designing, answer: *"How will I query this data?"*

```
"Do I always need comments when I load a post?" 
→ Embed comments

"Do comments need independent pagination/search?"
→ Reference comments separately
```

### 2. Embedding vs Referencing (Quick Guide)

| Decision Factor | Embed | Reference |
|----------------|-------|-----------|
| Always accessed together | ✅ | |
| Data is small & bounded | ✅ | |
| Data grows unboundedly | | ✅ |
| Data updated independently | | ✅ |
| Many-to-many relationship | | ✅ |
| Need to avoid 16MB limit | | ✅ |

## Code Example — Schema Design Patterns

### Pattern 1: Embedded (Simple Product with Reviews)
```js
// ✅ Good: reviews are small, always shown with product
{
  _id: ObjectId("..."),
  name: "Laptop",
  price: 55000,
  category: "electronics",
  specs: {
    ram: "16GB",
    storage: "512GB SSD",
    brand: "Dell"
  },
  reviews: [
    { user: "Nishant", rating: 5, comment: "Amazing!" },
    { user: "Raj", rating: 4, comment: "Good value" }
  ]
}
// ❌ Problem: if a product has 10,000 reviews → hits 16MB limit
```

### Pattern 2: Referenced (Users → Orders)
```js
// users collection
{
  _id: ObjectId("userId1"),
  name: "Nishant",
  email: "n@example.com"
}

// orders collection — references user
{
  _id: ObjectId("orderId1"),
  userId: ObjectId("userId1"),  // reference
  products: [
    { productId: ObjectId("pId1"), qty: 2 },
    { productId: ObjectId("pId2"), qty: 1 }
  ],
  total: 56500,
  status: "shipped",
  createdAt: new Date()
}
```

### Pattern 3: Hybrid (Partial Embed + Reference)
```js
// Product with top 3 reviews embedded, rest referenced
{
  _id: ObjectId("..."),
  name: "Laptop",
  price: 55000,
  topReviews: [
    { user: "Nishant", rating: 5 },  // small preview
  ],
  reviewCount: 2345,
  avgRating: 4.6
  // Full reviews stored in a separate `reviews` collection
}
```

## Common Schema Design Patterns

### 1. Bucket Pattern
Group time-series data into "buckets" instead of one document per event.
```js
// ❌ Bad: one document per temperature reading (millions of docs)
{ deviceId: "sensor1", temp: 22.5, at: ISODate("2026-03-24T08:00:00Z") }

// ✅ Good: bucket hourly readings into one document
{
  deviceId: "sensor1",
  date: "2026-03-24",
  hour: 8,
  readings: [22.5, 22.6, 22.8, 23.0, ...],  // 60 readings per doc
  count: 60,
  avg: 22.7
}
```

### 2. Outlier Pattern
Store the "typical case" embedded, handle edge cases with a reference.
```js
{
  name: "Popular Post",
  content: "...",
  likes: ["user1", "user2", "user3", ...],  // normally small
  hasExtraLikes: false  // if likes > 1000, set true and store the rest referenced
}
```

### 3. Computed Pattern
Pre-compute heavy aggregations and store the result.
```js
// Instead of querying SUM(sales) every time
{
  productId: ObjectId("..."),
  name: "Laptop",
  price: 55000,
  totalSold: 450,          // pre-computed
  totalRevenue: 24750000,  // pre-computed
  lastUpdated: new Date()
}
```

## Field Naming Best Practices

```
✅ Use camelCase: firstName, createdAt
✅ Be descriptive: isActive, not a
✅ Use consistent patterns: createdAt, updatedAt (not created, date_modified)
❌ Avoid short or cryptic names: a, b, x
❌ Don't use spaces or dots in field names
```

## Interview Questions

- **Q: Does MongoDB require a schema?**
  A: No — MongoDB has a flexible schema by default. But Mongoose (ODM) allows you to enforce schemas in your application layer. MongoDB 3.6+ also supports JSON Schema validation.

- **Q: What is the most important decision in MongoDB schema design?**
  A: Embedding vs referencing. The choice depends on access patterns, data size, update frequency, and relationship cardinality.

- **Q: What is the 16MB document limit in MongoDB?**
  A: Each MongoDB document can be at most 16 MB. Large embedded arrays can hit this limit, making referencing necessary.

- **Q: What is the "access pattern first" principle?**
  A: In MongoDB, you design your schema based on how your application reads data — not just how it's logically related. Read-heavy apps benefit from embedding.

- **Q: What is the computed pattern in MongoDB schema design?**
  A: Pre-computing expensive aggregations (totals, averages) and storing them alongside documents to avoid repeated heavy queries.

- **Q: What is data modeling in MongoDB?**
  A: The process of deciding how to structure documents, what to embed, what to reference, and how to optimize for the application's query patterns.
