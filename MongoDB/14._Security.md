# Security Basics in MongoDB

## Simple Explanation
**Security** in MongoDB means protecting your database from unauthorized access, data breaches, and injection attacks.

Think of your MongoDB database like a house:
- **Authentication** = the lock on the front door (who can enter?)
- **Authorization (roles)** = which rooms each person can access
- **Encryption** = the safe inside the house (protects what's stored)
- **Input sanitization** = checking what visitors bring in (prevent injections)

---

## Real-World Example
A hacker finds your MongoDB server exposed on the internet without a password. In seconds, they can:
- Read all your user data
- Delete your entire database
- Ransom your data

Proper security prevents this.

## 1. Authentication — Who Can Connect?

By default, MongoDB requires authentication when running in production.

```bash
# Start MongoDB with authentication enabled (mongod.conf)
security:
  authorization: enabled
```

### Create an Admin User
```js
// Connect to mongo without auth first (initial setup)
use admin

db.createUser({
  user: "adminUser",
  pwd: "strongPassword123!",
  roles: [{ role: "root", db: "admin" }]
})
```

### Create an App-Specific User (Least Privilege)
```js
use myapp

db.createUser({
  user: "appUser",
  pwd: "appSecurePass!",
  roles: [
    { role: "readWrite", db: "myapp" }  // only access to myapp DB
  ]
})
```

### Connect with Authentication
```bash
# Shell connection with auth
mongosh "mongodb://appUser:appSecurePass!@localhost:27017/myapp"
```

## 2. Built-in MongoDB Roles

| Role | Access |
|------|--------|
| `read` | Read data only |
| `readWrite` | Read + write data |
| `dbAdmin` | Manage DB (indexes, stats, etc.) |
| `dbOwner` | Full control of a database |
| `clusterAdmin` | Manage the cluster |
| `root` | Superuser — all privileges |
| `userAdmin` | Create and manage users |

✅ **Principle of Least Privilege** — give each user/service only the minimum access they need.

## 3. Network Security

```yaml
# mongod.conf — Bind to localhost only (not exposed to internet)
net:
  port: 27017
  bindIp: 127.0.0.1   # only local connections
  # Do NOT use: bindIp: 0.0.0.0 (exposes to all IPs)
```

### MongoDB Atlas Network Access
```
Atlas Dashboard → Network Access → Add IP Address
- Allow only your server's IP
- Allow 0.0.0.0/0 only for testing (never in production)
```

## 4. Encryption

### Transport Encryption (TLS/SSL)
```bash
# Always use TLS in production (wss:// equivalent for MongoDB)
mongosh "mongodb+srv://user:pass@cluster.mongodb.net/mydb?tls=true"
```

### Encryption at Rest
MongoDB Atlas provides encryption at rest by default. For self-hosted, use:
- WiredTiger storage encryption (Enterprise feature)
- OS-level disk encryption

## 5. NoSQL Injection Prevention

Unlike SQL injection (with SQL strings), MongoDB is vulnerable to **operator injection** where malicious `$` operators are sent in query inputs.

```js
// ❌ Vulnerable — user controls the query
const { email, password } = req.body;
const user = await User.findOne({ email, password });

// Attacker sends: { "email": { "$gt": "" }, "password": { "$gt": "" } }
// This matches ALL users → login bypassed!
```

### Prevention with `express-mongo-sanitize`
```bash
npm install express-mongo-sanitize
```

```js
const mongoSanitize = require("express-mongo-sanitize");

// Removes $ and . from user input
app.use(mongoSanitize());

// Now the attack payload: { "$gt": "" } → becomes: { "gt": "" } → no match
```

## 6. Mongoose Schema Validation (App-Level Defense)
```js
const userSchema = new mongoose.Schema({
  email: {
    type: String,
    required: true,
    match: /^\S+@\S+\.\S+$/,   // enforces proper email format
    lowercase: true
  },
  password: {
    type: String,
    minlength: 8,
    select: false               // never return password in queries
  }
});
```

## 7. Environment Variables — Never Hardcode Credentials
```js
// ❌ NEVER do this
mongoose.connect("mongodb://admin:password123@localhost:27017/myapp");

// ✅ Always use environment variables
require("dotenv").config();
mongoose.connect(process.env.MONGODB_URI);
```

## Security Checklist

```
✅ Enable authentication (mongod.conf: authorization: enabled)
✅ Create separate users with minimal permissions (least privilege)
✅ Bind MongoDB to localhost only (not 0.0.0.0)
✅ Use TLS/SSL for all connections (Atlas does this by default)
✅ Store connection strings in environment variables (.env)
✅ Sanitize user inputs to prevent NoSQL injection (express-mongo-sanitize)
✅ Hash passwords with bcrypt before storing
✅ Use select: false on password fields in Mongoose
✅ Whitelist IPs in MongoDB Atlas Network Access
✅ Keep MongoDB version updated (security patches)
❌ Never expose MongoDB port (27017) publicly
❌ Never use root user for app connections
```

## Interview Questions

- **Q: What is authentication in MongoDB?**
  A: The process of verifying who is connecting to MongoDB. Enabled via `authorization: enabled` in mongod.conf, requiring a valid username and password.

- **Q: What is the principle of least privilege?**
  A: Give each user/service only the minimum access they need — no more. App users should have `readWrite` only, not `root`.

- **Q: What is NoSQL injection in MongoDB?**
  A: An attack where malicious MongoDB operators (like `$gt`, `$where`) are injected via user input to bypass authentication or retrieve unauthorized data.

- **Q: How do you prevent NoSQL injection?**
  A: Use `express-mongo-sanitize` to strip `$` and `.` from user inputs. Also use Mongoose schema validation to enforce correct data types.

- **Q: What is `select: false` in Mongoose?**
  A: Hides a schema field (like password) from query results by default — prevents accidentally exposing sensitive data.

- **Q: Why should you never use the root user for app connections?**
  A: The root user has full server access. If the app is compromised, the attacker gets full control. Use a limited `readWrite` user instead.

- **Q: What is TLS/SSL in MongoDB?**
  A: Transport Layer Security — encrypts data in transit between the app and MongoDB server. MongoDB Atlas enforces TLS by default.
