# Performance Optimization & Pagination

## Simple Explanation
Performance in MongoDB means making your queries **fast** and your server **efficient** — even with millions of documents.

**Pagination** = Breaking large result sets into pages (10 items per page, etc.) instead of sending thousands of records at once.

Think of it like a library:
- **Performance** = organizing books on shelves properly (indexes) so you don't search the whole library
- **Pagination** = checking out 10 books at a time, not all 10,000 at once

---

## Real-World Example
A product listing page with 50,000 products:
- ❌ `find()` all 50,000 → slow, huge response, bad UX
- ✅ `find().sort().limit(12).skip(24)` → page 3, 12 items → fast

## Performance Optimization Techniques

### 1. Always Use Indexes on Query Fields
```js
// Without index: full collection scan (slow)
db.users.find({ email: "n@n.com" })

// Create index first:
db.users.createIndex({ email: 1 })

// Now query uses index (fast!)
db.users.find({ email: "n@n.com" })

// Verify with explain()
db.users.find({ email: "n@n.com" }).explain("executionStats")
// Look for "stage": "IXSCAN" (index scan) not "COLLSCAN"
```

---

### 2. Use Projection — Only Fetch Needed Fields
```js
// ❌ Bad: returns entire document (slow if docs are large)
db.products.find({ category: "electronics" })

// ✅ Good: only return what you need
db.products.find(
  { category: "electronics" },
  { name: 1, price: 1, _id: 0 }
)
```

---

### 3. Limit Results
```js
// ❌ Bad: fetch all matching docs
db.products.find({ category: "electronics" })

// ✅ Good: limit to what you actually show
db.products.find({ category: "electronics" }).limit(20)
```

---

### 4. Use `countDocuments()` Carefully
```js
// ❌ Slow: counts by scanning
db.products.find({ category: "electronics" }).count()  // deprecated

// ✅ Better: countDocuments (uses index if available)
db.products.countDocuments({ category: "electronics" })

// ✅ Best for large collections with rough estimate
db.products.estimatedDocumentCount()  // very fast approximate count
```

---

### 5. Avoid Large `$in` Arrays
```js
// ❌ Bad: $in with thousands of values
db.products.find({ _id: { $in: [id1, id2, ...10000ids] } })

// ✅ Better: redesign the schema or use aggregation
```

## Pagination Techniques

### Technique 1: Offset Pagination (skip/limit)
Classic page-based pagination. Easy but gets slow at high page numbers.

```js
const page = 3;
const limit = 10;
const skip = (page - 1) * limit; // = 20

db.products.find()
  .sort({ createdAt: -1 })
  .skip(skip)
  .limit(limit)

// Response with total count:
const total = await Product.countDocuments(filter);
const products = await Product.find(filter).sort().skip(skip).limit(limit);

res.json({
  data: products,
  page,
  limit,
  totalPages: Math.ceil(total / limit),
  totalItems: total
});
```

**⚠️ Problem with large skip values:**
- `skip(100000)` is slow — MongoDB still scans through 100,000 docs to skip them.

---

### Technique 2: Cursor-based Pagination (Recommended for large data)
Uses `_id` or a cursor field to fetch the next page — no skip needed.

```js
// First page
const page1 = await Product.find()
  .sort({ _id: 1 })
  .limit(10);

// Save the last _id: page1[page1.length - 1]._id

// Next page — fetch documents AFTER the last _id
const page2 = await Product.find({ _id: { $gt: lastId } })
  .sort({ _id: 1 })
  .limit(10);
```

```js
// Express route example
app.get("/products", async (req, res) => {
  const { cursor, limit = 10 } = req.query;

  const filter = cursor ? { _id: { $gt: cursor } } : {};

  const products = await Product.find(filter)
    .sort({ _id: 1 })
    .limit(Number(limit));

  const nextCursor = products.length > 0
    ? products[products.length - 1]._id
    : null;

  res.json({ data: products, nextCursor });
});
```

---

### Comparison: Offset vs Cursor Pagination

| Feature | Offset (skip/limit) | Cursor-based |
|---------|--------------------|--------------
| Easy to implement | ✅ Yes | 🟡 Medium |
| Supports "go to page 5" | ✅ Yes | ❌ No |
| Performance on large datasets | ❌ Degrades with high skip | ✅ Consistent |
| Works with sorted data | ✅ Yes | ✅ Yes |
| Best for | Admin panels, small datasets | Infinite scroll, APIs |

## Reading Performance with `explain()`
```js
const result = db.products.find({ category: "electronics" })
  .sort({ price: -1 })
  .explain("executionStats");

// Key metrics to check:
console.log(result.executionStats.executionTimeMillis); // query time in ms
console.log(result.executionStats.totalDocsExamined);   // docs scanned
console.log(result.executionStats.totalKeysExamined);   // index keys checked
// Good: totalDocsExamined ≈ nReturned (not scanning more than needed)
```

## Performance Checklist

```
✅ Add indexes to all frequently queried/sorted fields
✅ Use projection — never fetch fields you don't need
✅ Always limit results — never return thousands of docs at once
✅ Use cursor pagination for large, real-time data
✅ Run explain() to verify queries use indexes
✅ Use $match early in aggregation pipelines (reduces docs processed)
✅ Use estimatedDocumentCount() for total counts in listings
❌ Avoid large .skip() values
❌ Avoid $where (JavaScript evaluation — very slow)
❌ Don't index boolean fields (low cardinality)
```

## Interview Questions

- **Q: What is pagination in MongoDB?**
  A: Splitting large result sets into pages using `skip()` and `limit()` (offset pagination) or cursor-based pagination using `_id` comparisons.

- **Q: What is the problem with large `skip()` values?**
  A: MongoDB must still process and discard all skipped documents — `skip(100000)` scans 100,000 docs before returning results, which is slow.

- **Q: What is cursor-based pagination?**
  A: Instead of using `skip()`, use the last document's `_id` as a starting point for the next page: `find({ _id: { $gt: lastId } })`. Very fast — uses index directly.

- **Q: How do you check if a query is using an index?**
  A: Use `.explain("executionStats")` — look for `"stage": "IXSCAN"` (good) vs `"COLLSCAN"` (bad — no index).

- **Q: What is `estimatedDocumentCount()` and when should you use it?**
  A: A very fast approximate count using collection metadata — no query required. Use for total count in listings when exact count isn't critical.

- **Q: Why should `$match` come early in an aggregation pipeline?**
  A: Placing `$match` first reduces the number of documents passed to subsequent stages — fewer docs to process = faster pipeline.
