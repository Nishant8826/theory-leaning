# Relationships in MongoDB

## Simple Explanation
In real apps, data is connected — a user has orders, an order has products, a post has comments.

MongoDB handles these connections in two ways:
1. **Embedded Documents (Denormalization)** — store related data inside the same document
2. **Reference Documents (Normalization)** — store only an ID pointing to another collection

Think of it like two ways to store a recipe:
- **Embedded** = Write the full ingredient list inside the recipe book
- **Referenced** = Write "see ingredient list #42" and keep ingredients in a separate book

---

## Real-World Example

**User and Address:**
- One user has one address → **Embedded** (address is small, always retrieved together)

**User and Orders:**
- One user can have thousands of orders → **Referenced** (orders grow unboundedly)

**Students and Courses:**
- Many students enroll in many courses → **Referenced** with a join collection

## Types of Relationships

### 1. One-to-One
One document related to exactly one other document.

**Example**: User → Profile (each user has one profile)

```js
// Embedded approach ✅ (profile always accessed with user)
db.users.insertOne({
  name: "Nishant",
  email: "n@example.com",
  profile: {
    bio: "Full-stack developer",
    avatar: "profile.jpg",
    website: "https://nishant.dev"
  }
})

// Referenced approach (only if profile is rarely needed)
db.profiles.insertOne({
  userId: ObjectId("64abc..."),
  bio: "Full-stack developer",
  avatar: "profile.jpg"
})
```

---

### 2. One-to-Many
One document related to many others.

**Example**: User → Orders (one user, many orders)

```js
// Embedded (use ONLY if "many" is small and bounded)
db.users.insertOne({
  name: "Nishant",
  addresses: [
    { type: "home", city: "Mumbai", zip: "400001" },
    { type: "work", city: "Pune", zip: "411001" }
  ]
})

// Referenced approach ✅ (orders can grow large)
// orders collection
db.orders.insertOne({
  userId: ObjectId("64abc..."),  // reference to users collection
  product: "Laptop",
  amount: 55000,
  status: "shipped"
})

// Query orders for a user
db.orders.find({ userId: ObjectId("64abc...") })
```

---

### 3. Many-to-Many
Many documents related to many others.

**Example**: Students ↔ Courses

```js
// Option 1: Store array of references in one document
db.students.insertOne({
  name: "Nishant",
  enrolledCourses: [
    ObjectId("courseId1"),
    ObjectId("courseId2")
  ]
})

// Option 2: Junction collection (like a SQL join table)
db.enrollments.insertOne({
  studentId: ObjectId("studentId"),
  courseId: ObjectId("courseId"),
  enrolledAt: new Date(),
  grade: null
})
```

## Embedded Documents (Denormalization)

Store related data **inside** the parent document.

```js
db.users.insertOne({
  name: "Nishant",
  address: {
    street: "123 MG Road",
    city: "Mumbai",
    zip: "400001"
  },
  recentOrders: [
    { product: "Laptop", amount: 55000 },
    { product: "Mouse", amount: 500 }
  ]
})
```

### ✅ Advantages
- Single query to get all data (fast reads)
- No joins needed
- Good for data always accessed together

### ❌ Disadvantages
- Document limit is **16 MB**
- Updating nested data requires rewriting the parent
- Bad if embedded data grows unboundedly (like thousands of orders)

---

## Reference Documents (Normalization)

Store only the `_id` of related documents — like a foreign key in SQL.

```js
// users collection
db.users.insertOne({ name: "Nishant", email: "n@example.com" })

// orders collection — references the user
db.orders.insertOne({
  userId: ObjectId("64abc..."),    // reference
  product: "Laptop",
  amount: 55000
})
```

### Fetching with `$lookup` (aggregation join)
```js
db.orders.aggregate([
  {
    $lookup: {
      from: "users",              // join with this collection
      localField: "userId",       // field in orders
      foreignField: "_id",        // field in users
      as: "userDetails"           // name for resulting array
    }
  },
  {
    $unwind: "$userDetails"       // flatten the array to object
  }
])
```

### Fetching with Mongoose `populate()` (Node.js)
```js
const order = await Order.findById(orderId).populate("userId", "name email");
// order.userId is now { name: "Nishant", email: "n@example.com" }
```

### ✅ Advantages
- Good for large, unbounded relationships
- Data updated in one place (no duplication)
- Avoids 16MB document limit

### ❌ Disadvantages
- Requires joins (multiple queries or `$lookup`)

## Embedding vs Referencing — Decision Guide

| Scenario | Use |
|---------|-----|
| Data is always accessed together | Embedded |
| Data is small and bounded | Embedded |
| Data is large or grows over time | Referenced |
| Data is frequently updated independently | Referenced |
| Many-to-many relationships | Referenced |
| Avoiding 16MB document limit | Referenced |

## Interview Questions

- **Q: What are the two ways to handle relationships in MongoDB?**
  A: Embedding (store related data inside the document) and Referencing (store only the `_id` of related documents, like a foreign key).

- **Q: When should you embed vs reference?**
  A: Embed when data is small, bounded, and always accessed together. Reference when data is large, grows over time, is frequently updated independently, or has many-to-many relationships.

- **Q: What is `$lookup` in MongoDB?**
  A: An aggregation stage that performs a left outer join with another collection — similar to SQL JOIN.

- **Q: What is `populate()` in Mongoose?**
  A: A method that automatically replaces ObjectId references with the actual document data from the referenced collection.

- **Q: What is the document size limit in MongoDB?**
  A: 16 MB per document. Large embedded arrays can hit this limit — a key reason to use referencing.

- **Q: What is denormalization?**
  A: Storing related data together in one document (embedding) to avoid joins — trades write efficiency for read performance.

- **Q: What is normalization in MongoDB?**
  A: Splitting data into separate collections and using references — reduces duplication and avoids document size issues, at the cost of needing joins.
