# Indexes

## Simple Explanation
An **index** is a data structure that helps MongoDB **find documents faster** — without scanning every document in the collection.

Think of it like a **book index** (at the back of a textbook):
- Without an index → you read every page to find "promises" (slow!)
- With an index → you go to page 45 directly (fast!)

By default, MongoDB creates an index on `_id`. You can create custom indexes on other fields.

---

## Real-World Example
You have a `users` collection with 1 million documents. A query `db.users.find({ email: "n@n.com" })` without an index scans all 1 million documents (**Collection Scan**). With an index on `email`, MongoDB jumps directly to the matching document (**Index Scan**) — instantly fast.

---

## How Indexes Work

Indexes are stored in a **B-Tree (Balanced Tree)** structure:
- Values are kept **sorted**
- MongoDB can quickly binary-search for a value
- Finding a document takes `O(log n)` time vs `O(n)` for a full scan

```
Without index:       With email index:
[scan 1000 docs]     [B-Tree: binary search]
→ slow               → fast
```

## Code Examples

### Creating Indexes
```js
// Single-field index (ascending)
db.users.createIndex({ email: 1 })

// Single-field index (descending)
db.users.createIndex({ createdAt: -1 })

// Compound index — index on multiple fields
db.users.createIndex({ city: 1, age: 1 })
// Good for: find({ city: "Mumbai", age: 22 })

// Unique index — prevents duplicate values
db.users.createIndex({ email: 1 }, { unique: true })
// Inserting a duplicate email → throws error

// Multikey index — indexes each element of an array
db.products.createIndex({ tags: 1 })
// Good for: find({ tags: "gaming" })

// Text index — enables full-text search
db.articles.createIndex({ title: "text", content: "text" })
// Use: db.articles.find({ $text: { $search: "mongodb tutorial" } })

// TTL index — auto-delete documents after time expires
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })
// Documents auto-deleted 1 hour after createdAt
```

---

### Managing Indexes
```js
// View all indexes on a collection
db.users.getIndexes()

// Drop a specific index
db.users.dropIndex({ email: 1 })

// Drop an index by name
db.users.dropIndex("email_1")

// Drop ALL indexes (except _id)
db.users.dropIndexes()
```

---

### Analyzing Query Performance with `explain()`
```js
// "executionStats" shows how query was executed
db.users.find({ email: "n@n.com" }).explain("executionStats")

// Look for these in the output:
// "stage": "IXSCAN"     → Good! Using an index
// "stage": "COLLSCAN"   → Bad! Scanning all documents
// "totalDocsExamined": 1 → Perfect (only 1 doc checked)
// "executionTimeMillis": 0 → Fast
```

## Types of Indexes Summary

| Index Type | Use Case |
|-----------|----------|
| **Single-field** | Queries on one field (`{ email: 1 }`) |
| **Compound** | Queries on multiple fields (`{ city: 1, age: 1 }`) |
| **Unique** | Prevent duplicate values (`{ email: 1 }` + unique) |
| **Multikey** | Queries on array fields (`{ tags: 1 }`) |
| **Text** | Full-text search on string fields |
| **TTL** | Auto-expire documents (sessions, logs) |
| **Geospatial** | Location-based queries (`2dsphere`) |

---

## Index Trade-offs

### ✅ Advantages
- **Massively speeds up read operations** (find, sort, range queries)
- `findById()` is fast because `_id` is always indexed
- Required for `unique` constraints

### ❌ Disadvantages
- **Slows down write operations** — every insert/update/delete must also update the index
- **Uses disk space** — each index takes extra storage
- Too many indexes = performance nightmare on write-heavy collections

---

## Index Best Practices

```
✅ Index fields used in WHERE/filter conditions
✅ Index fields used in sort operations
✅ Use compound index for multi-field queries
✅ Use TTL index for session/cache collections
✅ Run explain() to verify indexes are being used
❌ Don't create indexes on low-cardinality fields (like boolean: true/false)
❌ Don't over-index write-heavy collections
❌ Remove unused indexes (they waste space and slow writes)
```

## Interview Questions

- **Q: What is an index in MongoDB?**
  A: A data structure (B-Tree) that stores field values in sorted order, allowing MongoDB to quickly find documents without scanning the entire collection.

- **Q: What is the default index in MongoDB?**
  A: Every collection has a default index on the `_id` field — that's why `findById()` is fast.

- **Q: What is a compound index?**
  A: An index on multiple fields. Useful for queries that filter or sort on more than one field. Order of fields matters.

- **Q: What is a TTL index?**
  A: A Time-To-Live index that automatically deletes documents after a specified number of seconds. Useful for sessions, temporary tokens, and logs.

- **Q: What is the disadvantage of indexing?**
  A: Indexes slow down write operations (insert/update/delete) because the index data structure must be updated. They also use additional disk space.

- **Q: What does `explain("executionStats")` tell you?**
  A: Details about how a query was executed — whether it used an index (IXSCAN) or a full scan (COLLSCAN), and how many documents were examined.

- **Q: When should you NOT create an index?**
  A: On low-cardinality fields (like a boolean with only true/false), on small collections (COLLSCAN is fine), or on write-heavy collections where write performance matters more.
