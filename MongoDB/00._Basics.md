# What is a Database? — MongoDB & NoSQL Basics

## Simple Explanation
A **database** is a place where you store data in an **organized** way so it can be easily found, updated, and deleted.

Think of a database like a **filing cabinet**:
- The cabinet = the database
- Each drawer = a collection (a category of data)
- Each folder inside = a document (one record)

**MongoDB** is a **NoSQL** database that stores data as **JSON-like documents** — instead of rows and columns like Excel/SQL.

---

## Real-World Example
A regular SQL table stores users like this:
```
| id | name    | email           |
|----|---------|-----------------|
| 1  | Nishant | n@example.com   |
```

MongoDB stores the same user as a flexible **document**:
```json
{
  "_id": ObjectId("64abc..."),
  "name": "Nishant",
  "email": "n@example.com",
  "age": 22,
  "address": {
    "city": "Mumbai",
    "zip": "400001"
  },
  "hobbies": ["coding", "music"]
}
```
Notice: MongoDB can store **nested objects** and **arrays** directly — no extra tables needed!

## MongoDB Architecture

```
MongoDB Server (mongod)
    └── Database (e.g., myapp)
            └── Collection (e.g., users)
                    └── Document (e.g., { name: "Nishant" })
```

| MongoDB Term | SQL Equivalent | Description |
|-------------|----------------|-------------|
| Database | Database | Container for collections |
| Collection | Table | Group of related documents |
| Document | Row | A single record (JSON-like object) |
| Field | Column | A key-value pair inside a document |
| `_id` | Primary Key | Unique identifier — auto-generated |

## SQL vs NoSQL — Full Comparison

| Feature | SQL (Relational) | NoSQL (Non-relational) |
|---------|-----------------|----------------------|
| **Data Model** | Tables with rows & columns | Flexible documents, key-value, graphs |
| **Schema** | Fixed — define before inserting | Dynamic — add fields anytime |
| **Scalability** | Vertical (more power to one server) | Horizontal (add more servers) |
| **Transactions** | ACID compliant (strict consistency) | BASE (speed-first, some support ACID now) |
| **Querying** | SQL — powerful JOINs, aggregations | JSON-style queries, varies per DB |
| **Best For** | Banking, ERP, e-commerce (structured data) | Social apps, analytics, IoT (flexible/fast) |
| **Examples** | PostgreSQL, MySQL | MongoDB, Redis, Cassandra |

## When to Choose MongoDB

✅ Choose MongoDB when:
- Your data structure can change over time (flexible schema)
- You need to scale across many servers
- You're storing JSON-like data (user profiles, products, logs)
- You're building real-time apps, catalogs, content systems

❌ Avoid MongoDB when:
- You need complex JOINs between many related tables
- Strict ACID transactions are required (e.g., banking)
- Your data is highly relational and structured

---

## MongoDB Key Features
- **Flexible schema** — no rigid table structure
- **Document-oriented** — data stored as BSON (Binary JSON)
- **Horizontal scaling** (sharding)
- **Replication** — automatic data backup across multiple servers
- **Rich query language** with operators and aggregation pipeline
- **Full-text search**, geospatial queries, and more

## Interview Questions

- **Q: What is MongoDB?**
  A: A NoSQL document database that stores data as flexible JSON-like BSON documents. Designed for horizontal scaling, flexible schemas, and fast development.

- **Q: What is the difference between SQL and NoSQL?**
  A: SQL stores data in fixed-schema tables with rows/columns. NoSQL stores data in flexible formats (documents, key-value, etc.). SQL is better for structured data with complex relationships; NoSQL is better for scalable, fast, flexible data.

- **Q: What is a collection in MongoDB?**
  A: A group of related MongoDB documents — similar to a table in SQL. A collection doesn't enforce a fixed schema by default.

- **Q: What is a document in MongoDB?**
  A: A single data record stored as a BSON object (like JSON). Each document has a unique `_id` field.

- **Q: What is BSON?**
  A: Binary JSON — a binary-encoded serialization of JSON-like documents. Supports additional data types like `ObjectId`, `Date`, `Binary`, and more.

- **Q: What is horizontal vs vertical scaling?**
  A: Vertical = add more resources to one server (CPU, RAM). Horizontal = add more servers. MongoDB supports horizontal scaling via sharding.

- **Q: What is `_id` in MongoDB?**
  A: The primary key of every document. Automatically generated as an `ObjectId` (12-byte unique identifier) unless specified manually.
