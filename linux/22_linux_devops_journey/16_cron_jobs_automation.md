# 1. Scenario: Automating Daily Database Backups

## 2. Real-world Context
Your database administrator successfully created a bash script `/opt/scripts/backup_db.sh` that takes a full snapshot of the production MySQL database. However, currently they have to log in manually at 3:00 AM every single morning to run it. If they forget, the company is at massive risk. You need to automate this script to run relentlessly, every single night, while everyone is sleeping.

## 3. Objective
Schedule an automated recurrent task using the Linux Cron daemon.

## 4. Step-by-step Solution

### Prerequisite: Setting up the Practice Environment
*To practice automating scripts with cron, the target directory and the mock backup script must exist first:*

```bash
sudo mkdir -p /opt/scripts
sudo bash -c 'cat <<EOF > /opt/scripts/backup_db.sh
#!/bin/bash
echo "Backup starting at \$(date)"
# Database dump logic goes here...
echo "Backup complete!"
EOF'
```
* **What:** Creates the `/opt/scripts` system directory and generates a simple skeleton shell script named `backup_db.sh`.
* **Why:** The scenario requires modifying the permissions on this script and scheduling it. If the file doesn't actually exist, the `chmod` command and the automated job will instantly fail.
* **How:** Elevated permissions (`sudo mkdir`) create the restricted folder, and an automated heredoc securely injects the basic script text directly into the file.
* **Impact:** Provides the exact prerequisite target script needed for your cron scheduler to successfully execute.

**Step 1: Ensure the script is executable**
```bash
chmod +x /opt/scripts/backup_db.sh
```
* **What:** Adds the 'execute' permission to the backup script.
* **Why:** The scheduling daemon (cron) relies on executing the file just like a human would. If it lacks execution rights, the scheduled job will silently fail.
* **How:** `chmod +x` is standard practice for bash scripts.
* **Impact:** Prevents the #1 reason automation scripts fail on Linux.

**Step 2: Open the crontab editor for the current user**
```bash
crontab -e
```
* **What:** Opens the specialized file containing the schedule of automated tasks (cron jobs).
* **Why:** You must declare *when* and *what* should run.
* **How:** `crontab -e`. (It might ask you to choose an editor like nano or vim the first time).
* **Impact:** Grants access to the most fundamental mechanism of automation in standard Linux.

**Step 3: Define the cron schedule**
```text
0 3 * * * /opt/scripts/backup_db.sh >> /var/log/db_backup.log 2>&1
```
* **What:** Tells cron: "At minute 0, hour 3 (3:00 AM), every day, every month, every day-of-week: run the backup script, and append all output or errors into a log file."
* **Why:** Standardizing backups during low-traffic hours (3 AM) minimizes performance impact on customers.
* **How:** Cron syntax is 5 asterisks: `[Minute] [Hour] [Day_of_Month] [Month] [Day_of_Week]`. `>>` appends regular output. `2>&1` ensures error messages are *also* dumped heavily into the log file.
* **Impact:** Instantly solves human error. The server is now self-maintaining.

**Step 4: View your active scheduled tasks**
```bash
crontab -l
```
* **What:** Lists the contents of your crontab.
* **Why:** Always verify that your edits saved correctly upon exiting the editor.
* **How:** `crontab -l`.
* **Impact:** Peace of mind that the 3 AM job actually exists in the scheduler system.

## 6. Expected Output
```text
$ chmod +x /opt/scripts/backup_db.sh

$ crontab -e
crontab: installing new crontab

$ crontab -l
# DO NOT EDIT THIS FILE - edit the master and reinstall.
# m h  dom mon dow   command
0 3 * * * /opt/scripts/backup_db.sh >> /var/log/db_backup.log 2>&1
```

## 7. Tips / Best Practices
* **Use Absolute Paths:** A cron environment is barebones. It does NOT load your normal `PATH` variables. Always use the full absolute path `/usr/bin/python3 /opt/script.py` instead of just `python3 script.py`.
* **Crontab Guru:** Always use the website `crontab.guru` to translate your 5 asterisks into English (e.g. `*/5 * * * *` means "every 5 minutes"). It prevents scheduling mistakes that run scripts too frequently.

## 8. Interview Questions
1. **Q:** What do the 5 stars `* * * * *` represent in a crontab file?
   **A:** Minute (0-59), Hour (0-23), Day of Month (1-31), Month (1-12), and Day of Week (0-6, Sunday=0).
2. **Q:** Why do we append `>> /var/log/job.log 2>&1` to the end of a cron job?
   **A:** Cron runs blindly in the background. Without redirection, any `echo` or error throw will vanish, making it impossible to debug if the background script fails.
3. **Q:** How do you edit the crontab for a *different* user, such as 'admin'?
   **A:** Using `sudo crontab -u admin -e`.

## 9. DevOps Insight
Cron is the grandfather of automation. While invaluable for basic servers, modern DevOps heavily replaces Cron with tools like GitLab Scheduled Pipelines, AWS EventBridge, or Kubernetes CronJobs. These modern tools provide dashboards, automatic retry logic, and Slack alerts if a job fails, whereas Linux Cron silently fails unless you explicitly write log-monitoring scripts around it.
