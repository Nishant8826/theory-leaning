# 1. Scenario: Troubleshooting Website "502 Bad Gateway"

## 2. Real-world Context
Your team just deployed a new version of a Node.js backend. A few minutes later, customers report that visiting the website displays a stark, white screen screaming "502 Bad Gateway" generated by Nginx. The frontend is dead. You need to untangle the layers, figure out why Nginx is angry, verify the backend health, and restore the communication bridge.

## 3. Objective
Understand the architecture of reverse proxies, diagnose a 502 error, check local service availability, and analyze application-to-proxy logs.

## 4. Step-by-step Solution

**Step 1: Understand what a 502 actually means**
* **Concept:** A 502 Bad Gateway means Nginx (the frontend proxy) successfully received the customer's request over the internet, turned around to hand it to the Node.js backend Application running locally on port 3000... but the Node app entirely failed to respond. Nginx gives up and throws the 502 to the customer.

**Step 2: Check the Nginx error logs**
```bash
sudo tail -n 20 /var/log/nginx/error.log
```
* **What:** Pulls the last 20 lines of the web server crash reports.
* **Why:** You need explicit proof of what Nginx is attempting to talk to and why it's failing.
* **How:** `tail -n 20` limits the output so as not to overwhelm your terminal.
* **Impact:** The log reveals: `connect() failed (111: Connection refused) while connecting to upstream, client: x.x.x.x, server: domain.com, request: "GET /", upstream: "http://127.0.0.1:3000/"`. This proves Nginx is healthy, and the bug is strictly with port 3000.

**Step 3: Check if the backend application is actually running on port 3000**
```bash
sudo netstat -tulpn | grep 3000
```
* **What:** Checks the network stack to see if any application has registered port 3000.
* **Why:** "Connection refused" means literally nobody is home at that port.
* **How:** `netstat` dumps active ports, `grep` isolates 3000.
* **Impact:** The command returns completely empty. This proves the Node.js app is totally dead/offline.

**Step 4: Check systemd status for the backend**
```bash
sudo systemctl status backend-node
```
* **What:** Checks systemd to see why it thinks the app is dead.
* **Why:** You found the app is offline; now you need to know why. Did the developer's new code crash on startup?
* **How:** `systemctl status`.
* **Impact:** Output clearly states: `SyntaxError: Missing initializer in const declaration`. The developer pushed broken code!

**Step 5: Rollback code and restart the backend**
```bash
cd /var/www/backend && git checkout v1.0
sudo systemctl restart backend-node
```
* **What:** Reverts the directory back to a known working codebase (v1.0) and restarts the service.
* **Why:** The primary goal of an outage is "Restore Service First, Debug Later".
* **Impact:** Port 3000 comes alive. Nginx reconnects. The 502 disappears for users.

## 7. Tips / Best Practices
* **500 vs 502 vs 504:**
  * **500 Internal Server Error:** The backend app is running, received traffic from Nginx, but threw a fatal code exception (e.g. bad database query) and sent the error back.
  * **502 Bad Gateway:** The backend app is dead, rebooting, or completely refuses the connection from Nginx.
  * **504 Gateway Timeout:** The backend app is alive, accepted the traffic from Nginx, but is hanging endlessly (endless loop, slow DB query) and took over 60 seconds to reply. Nginx got bored and hung up.

## 8. Interview Questions
1. **Q:** Customers report a 502 Bad Gateway. Do you look at the Nginx logs or the Application code first?
   **A:** Look at the Nginx `error.log` first to confirm exactly *which* upstream backend it was trying to connect to. Then verify via `netstat` if the backend process is actually running on that port.
2. **Q:** Your app is running locally on port 5000. Why do you need Nginx?
   **A:** Node/Python servers are not designed to natively handle raw unfiltered web traffic, DDos attacks, thousands of idle keep-alive connections, or terminate SSL/HTTPS certificates rapidly. Nginx protects them as a hardened Reverse Proxy.

## 9. DevOps Insight
To prevent these scenarios, DevOps utilizes complex CI/CD "Zero Downtime Deployments" (like Blue/Green deployment). The new code would spin up on a temporary port (3001). Test scripts check if port 3001 answers 200 OK. Only if the tests pass does the pipeline dynamically rewrite the Nginx config to point at 3001 instead of 3000 securely. This completely eliminates end-user 502s during rollouts.
