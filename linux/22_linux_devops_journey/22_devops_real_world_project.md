# 1. Scenario: FINAL PROJECT - Website Down After Deployment

## 2. Real-world Context
You are the On-Call DevOps Engineer. At 2:00 AM on a Friday, the lead developer pushes a "minor hotfix" to the production branch. The CI/CD pipeline runs successfully. 5 minutes later, PagerDuty begins heavily alerting: **"CRITICAL: Company homepage is entirely unreachable."** Customer traffic is slamming the server, generating thousands of errors. The developer is panicking. You must step in, take command of the Linux infrastructure, systematically diagnose the stack, find the root cause, and restore the service immediately.

## 3. Objective
Deploy a full-stack debugging methodology: Inspect routing, assess disk capacity, analyze service states, comb through unstructured proxy/application logs, and enact a resolution.

## 4. Step-by-step Solution

### Prerequisite: Setting up the Practice Environment
*This is the final project. To simulate this major catastrophe authentically without needing a massive full backend codebase, we will mock the proxy, the systemd daemon, and explicitly build the permission crisis:*

```bash
sudo apt update && sudo apt install -y nginx
sudo bash -c 'cat <<EOF > /etc/nginx/sites-available/default
server { listen 80 default_server; location / { proxy_pass http://127.0.0.1:4000; } }
EOF'
sudo systemctl restart nginx
sudo groupadd appgroup || true
sudo useradd -M -g appgroup appuser || true
sudo mkdir -p /etc/node
sudo touch /etc/node/production.json
sudo chown root:root /etc/node/production.json
sudo chmod 600 /etc/node/production.json
sudo bash -c 'cat <<EOF > /etc/systemd/system/node-api.service
[Service]
User=appuser
ExecStart=/bin/bash -c "if ! cat /etc/node/production.json >/dev/null 2>&1; then echo \"ERROR: Cannot read config file: /etc/node/production.json - Permission denied\"; exit 1; else python3 -m http.server 4000; fi"
[Install]
WantedBy=multi-user.target
EOF'
sudo systemctl daemon-reload
sudo systemctl restart node-api || true
```
* **What:** Constructs a fake "node-api" web service running as a newly restricted unprivileged user, and sets it to aggressively read a config file unfortunately owned solely by `root`. It deliberately crashes because it naturally generates a "Permission denied" error.
* **Why:** It actively engineers the exact "murder mystery" root cause—proving Nginx works flawlessly, that the Node app crashes instantly on boot, and leaving undeniable forensic evidence strictly in `journalctl`.
* **How:** Standard Linux permissions (`chown`, `chmod`) securely isolate the file, and an intelligent `systemd` bash script is ordered to execute against it (answering on port 4000 if successful).
* **Impact:** Deploys a brilliant, highly sophisticated simulation of a major PagerDuty production incident. You can now follow the step-by-step diagnostic chain comprehensively and organically.

**Step 1: Network Check (Is the server actually online?)**
```bash
curl -I http://localhost
```
* **What:** Initiates a local HTTP request against the server's own networking stack.
* **Why:** You must first determine if the problem is external (AWS Load Balancer/DNS) or internal (the machine itself).
* **Impact:** The command returns `502 Bad Gateway`. This explicitly confirms the server is online, Nginx is active, but the backend application is dead.

**Step 2: Service Status Check (Did the backend crash?)**
```bash
sudo systemctl status node-api --no-pager
```
* **What:** Checks the explicit systemd health state of the application.
* **Why:** To see if the deployment script failed to restart the service correctly, or if the code crashed entirely upon launch.
* **Impact:** Output shows: `Active: failed (Result: exit-code) / Main process exited`. The service is definitively dead.

**Step 3: Disk Space Check (Did the deployment fill the drive?)**
```bash
timeout 5s df -h /
```
* **What:** Validates remaining storage capacity on the root partition.
* **Why:** Often, deployments create massive backup folders or download gigantic `.tar` artifacts. If the disk hits 100%, applications crash immediately because they cannot write logs.
* **Impact:** Output shows `Use% 42%`. The disk is perfectly fine. The crash is not a storage bottleneck.

**Step 4: Log Analysis (Why did the code crash?)**
```bash
sudo journalctl -u node-api -n 20 --no-pager
```
* **What:** Extracts the last 20 lines of logs explicitly generated by the failing application right before it died.
* **Why:** You ruled out Network and Disk. Therefore, it is overwhelmingly likely a code or configuration bug.
* **Impact:** The log prominently states: `ERROR: Cannot read config file: /etc/node/production.json - Permission denied`. The bug is revealed!

**Step 5: Permissions Fix (Resolve the Root Cause)**
```bash
ls -l /etc/node/production.json
sudo chown appuser:appgroup /etc/node/production.json
sudo chmod 640 /etc/node/production.json
```
* **What:** Corrects the ownership and attributes of the required configuration file.
* **Why:** The automated deployment script accidentally wrote the new config file as the `root` user. When systemd tried to start the app using `appuser`, Linux's strict file security brutally blocked it.
* **Impact:** Restores accessibility. The application now legally has the Linux permissions to read its own configuration.

**Step 6: Service Restart & Verification (Restoring the site)**
```bash
sudo systemctl restart node-api
curl -I http://localhost
```
* **What:** Reboot the service and run the identical network ping from Step 1.
* **Why:** You must definitively prove the incident is fully resolved before closing the PagerDuty alert.
* **Impact:** Output shows `200 OK`. The server is functionally accepting users again. Disaster averted.

## 6. Expected Output
```text
$ curl -I http://localhost
HTTP/1.1 502 Bad Gateway

$ sudo systemctl status node-api
Active: failed (Result: exit-code)

$ sudo journalctl -u node-api -n 5
node-api.service: Main process exited, code=exited, status=1/FAILURE
ERROR: Cannot read config file: /etc/node/production.json - Permission denied

$ sudo chown appuser:appgroup /etc/node/production.json
$ sudo systemctl restart node-api

$ curl -I http://localhost
HTTP/1.1 200 OK
```

## 7. Preventive Automation Idea
If the CI/CD pipeline (e.g., Jenkins) ran a smoke test script containing `curl -I http://localhost` at the very end of the deployment sequence, the pipeline would have instantly caught the 502 error, and could have automatically run `git checkout [previous_commit]` and `systemctl restart`, triggering an instant, automated rollback. The users would have experienced 2 seconds of downtime, and you wouldn't have been paged at 2 AM.

# 🔥 EXTRA SECTIONS 🔥

## 1. Top 50 DevOps Interview Questions (Linux Focus)

1. **What is Linux?** An open-source unix-like kernel handling core OS functions.
2. **What is Kernel limit tuning?** Altering OS variables (via `sysctl`) like open file limits and TCP buffers.
3. **What is an inode?** A data structure holding metadata (permissions, disk blocks) about a file. If you run out of inodes, you cannot create new files even if disk space is free.
4. **Explain the Boot Process.** BIOS -> MBR -> GRUB -> Kernel -> Init (Systemd) -> Runlevel services.
5. **What does `chmod 777` do?** Grants full read/write/execute permissions to everyone. Considered extreme security risk.
6. **Difference between Soft and Hard Links?** Soft link points to the filename (breaks if original is moved). Hard link points to the inode (survives if original is renamed).
7. **What is the `umask`?** Sets default permissions for newly created files/folders.
8. **What is a Daemon?** A background process serving system functions (e.g., `sshd`, `httpd`).
9. **Explain `grep`.** Global Regular Expression Print. Filters text streams based on patterns.
10. **Explain `awk`.** A programming language used for advanced column-based text parsing.
11. **Explain `sed`.** Stream editor, primarily used for regex search-and-replace dynamically.
12. **Difference between `df` and `du`?** `df` checks mounted filesystem capacity. `du` calculates explicit folder weights.
13. **Difference between `TCP` and `UDP`?** TCP guarantees delivery and order (via handshakes). UDP fires blindly without verification (used for video/voice).
14. **What is Port 80 and 443?** HTTP and HTTPS, respectively.
15. **What is Port 22?** SSH (Secure Shell).
16. **How to check open ports?** `netstat -tulpn` or `ss -tulpn`.
17. **What is the `hosts` file?** A local DNS override file located at `/etc/hosts` that resolves IPs manually before checking public DNS servers.
18. **Difference between `curl` and `ping`?** `curl` tests application layer (L7) HTTP protocols. `ping` tests network layer (L3) ICMP packet delivery.
19. **What is `systemd`?** The central init system managing service statuses, dependencies, and logging in modern Linux.
20. **How do you kill a hanging process?** Find PID using `ps aux | grep [name]`, then execute `kill -9 [PID]`.
21. **What are Zombie processes?** A process that has completed but its parent hasn't read its exit status.
22. **What is `top` / `htop`?** Interactive task managers showing live RAM, CPU, and process loads.
23. **What causes High Load Average but Low CPU usage?** Usually wait I/O: the CPU is stuck doing nothing, waiting for a slow hard drive or network response.
24. **Difference between Physical Mem and Swap?** Physical is actual RAM. Swap is slow hard disk space used as emergency overflow when RAM is 100% full.
25. **What is `OOM Killer`?** Out of Memory Killer. The Linux kernel forcefully annihilating running apps to save itself from crashing.
26. **What is Bash?** Bourne Again Shell. The default command-line interface and scripting language.
27. **What is `#!/bin/bash`?** The shebang. Placed at line 1, it instructs the OS to parse the script using the bash interpreter.
28. **How to automate tasks?** Use `crontab` for time-based scheduling.
29. **What does `2>&1` do?** Redirects standard error stream (2) to standard output stream (1), merging them.
30. **What is `sudo`?** SuperUser DO. Temporarily grants root privileges to a standard user.
31. **What is an SSH key pair?** A cryptographic mechanism (Public Key + Private Key) replacing passwords for secure server remote access.
32. **Where are SSH keys stored?** Usually inside the `~/.ssh/` hidden directory within the user's home folder.
33. **What is SCP?** Secure Copy Protocol, transferring files over the SSH tunnel.
34. **What is RSYNC?** A more robust file transfer tool that only syncs mutated differential file data (deltas).
35. **What is IPtables / UFW?** The primary firewall subsystem for controlling incoming/outgoing packet acceptance.
36. **Difference between `apt` and `yum`?** `apt` is for Debian/Ubuntu (dpkg). `yum/dnf` is for RedHat/CentOS (rpm).
37. **What does `chown` do?** Changes file Ownership (User and Group).
38. **Explain a 502 error.** Bad Gateway. Web proxy received request but backend service failed to respond.
39. **Explain a 500 error.** Internal Server Error. Backend application crashed internally processing a request.
40. **Explain a 403 error.** Forbidden. Client has valid connection but server intentionally blocked access (bad permissions).
41. **Explain a 200 code.** HTTP OK. Standard success code.
42. **What is Log Rotation?** Using `logrotate` to archive or delete old logs daily, preventing disks from reaching 100% full.
43. **Where are general logs found?** Inside the `/var/log/` directory.
44. **What does `tail -f` do?** Recursively follows the end of a file, live-updating to the screen.
45. **What is `stdout` vs `stderr`?** File descriptor 1 vs 2. Allows segregating successful programmatic output from error messages.
46. **What is a reverse proxy?** A server (like Nginx) sitting in front of web applications, forwarding client requests dynamically and handling load balancing.
47. **What happens if you type `rm -rf /`?** It attempts to recursively force delete the entire operating system, destroying the server instantly.
48. **How do you safely view a large file without triggering OOM?** Use `less [filename]`, which loads data into memory page by page instead of caching the whole string.
49. **How do you securely delete sensitive data?** Use `shred`, which overwrites the file bytes with randomized data multiple times before deletion.
50. **Why is Linux preferred in DevOps over Windows?** It is entirely open source, natively integrates seamlessly into CI/CD CLI pipelines, possesses minimal resource overhead (no GUI required), and operates exactly identically in a local Docker container as it does on a massive Kubernetes cluster.

## 2. Common Mistakes Engineers Make in Production

* **Ignoring the Backup:** Deploying complex architectural changes straight to Production without backing up the current config mapping.
* **Typing before Looking:** Relying blindly on historical knowledge ("This is always how the DB boots!") instead of running `systemctl status` or `df -h` to verify assumptions first.
* **Sudo Blindness:** Masking application errors by running everything as `sudo` rather than aggressively fixing the underlying User permissions matrix.
* **Editing Production Code:** Live editing backend code via VIM directly on servers instead of utilizing the formal CI/CD Git process, creating "Drift" and causing identical outages on the next automated deployment.
* **Panic Restarting:** Rebooting a server the instant it lags, inevitably destroying the temporary logs/memory core dumps severely needed to isolate the root cause.

## 3. How to Become a DevOps Engineer (Roadmap)

1. **Master Linux:** You cannot manage infrastructure without mastering the operating system underneath it. Gain proficiency entirely via CLI without graphic interfaces.
2. **Understand Networking Concepts:** Gain absolute clarity on DNS resolution, TCP handshakes, Subnets, Load Balancers, and Firewalls.
3. **Learn Source Control:** Master `git`. Not just `commit`/`push`, but resolving merge conflicts, branching strategies, and reverting commits.
4. **Learn a Server-Side Language:** Grasp fundamental Python or Bash to write critical automation scripts connecting distinct infrastructure APIs.
5. **Study Containerization (Docker):** Convert raw applications into isolated, portable Docker containers. Stop depending directly on `apt install`.
6. **Learn CI/CD Implementation:** Master GitHub Actions or Jenkins to automate the test/build/deploy lifecycle anytime new code is merged.
7. **Infrastructure as Code (IaC):** Dive into Terraform or Ansible. Stop clicking buttons in the AWS Console. Write `.tf` files to provision networks implicitly.
8. **Orchestration (Kubernetes):** Learn how to organize and deploy hundreds of Docker containers across multiple hardware nodes synchronously.
9. **Observability:** Integrate Prometheus/Grafana or Datadog to visualize CPU, Memory, and Error metrics comprehensively across scattered infrastructures.

## 4. Real-world Impact of Linux in DevOps
Linux is arguably the largest and most critical linchpin holding modern enterprise architecture fully together. When an engineer configures a Kubernetes cluster, they are simply programming heavily automated Linux control groups (`cgroups`) and namespaces. 

Every API that functions, every database that commits, and every container that launches relies on the precise orchestration of Linux file descriptors, systemd commands, and kernel boundaries. Therefore, the most elite "DevOps Practitioners" are, at their absolute core, highly systematic Linux System Administrators capable of writing code. By completing this journey, you are equipped to not just build pipelines, but fundamentally understand and salvage the systems they run on.
